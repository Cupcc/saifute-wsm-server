# 价格层出库与成本追溯

**version**: v1.0 | **domain**: 库存 / 销售出库 | **status**: 探索中

## 简介

这篇文档用于解决“同一物料存在多个库存成本价时，前台如何简化出库录单，同时后台仍能准确核算和追溯成本”的问题。核心思路是让用户按价格层选择库存，系统在相同价格层内自动完成 FIFO 分配，从而兼顾前台操作简单、库存校验明确以及出库成本链路可审计。

## 目标

在同一物料存在多个库存价格时，提供一套简单、清晰、可审计的出库方式：

- 用户按 `物料 + 价格 + 数量` 录单
- 系统保证该价格库存足够
- 系统在同价格内自动 FIFO 到具体入库来源
- 成本可核算、可追溯，但不把批次复杂度暴露给用户

## 已确认的业务口径

- 出库单上的“价格”定义为 `库存成本价 / 价格层`
- 价格只能从当前有库存的价格层中选择
- 不允许手输一个库存里不存在的价格
- 同一单据内，`同一物料 + 同一价格` 只能有一条
- 数量不能超过该价格层可用库存
- 用户不需要选择具体入库批次
- 后台必须在该价格层内部自动 FIFO 到具体来源层
- 已出库历史成本保留快照，不因后续入库改单价而回算

## 最简单的交互

一行一行录入：

1. 新建一行
2. 选择物料
3. 选择价格
4. 输入数量

交互要求：

- 物料支持模糊搜索、缩写、拼音
- 价格下拉只显示该物料当前有库存的价格层
- 价格项应显示可用量，例如：
  - `104元（可用100）`
  - `117元（可用200）`
- 数量支持输入和滚轮调整
- 若超出该价格层库存，立即提示并禁止提交
- 若同一物料同一价格重复出现，立即提示并禁止提交

示例：

- 成品A，104元，50个
- 成品A，117元，100个

## 系统设计原则

### 1. 不先改总库存主表

第一版不建议直接把总库存余额改成“按价格分库存”。

继续保留：

- 总库存余额：按 `物料 + 仓别`

另外增加一个价格层口径：

- 价格层可用库存：按 `物料 + 单价` 聚合现有来源层得到

这样最简单，改动最小。

### 2. 用户选价格层，系统追具体来源

用户前台选择的是：

- `物料`
- `价格层`
- `数量`

系统后台真正落账时：

1. 找出这个物料、这个价格层下的所有可用来源
2. 在这些来源中按 FIFO 分配
3. 写入来源占用记录

因此：

- 用户视角追到价格层
- 系统视角追到入库单行或库存流水

### 3. 来源追溯不要只到单据号

“来源批次”在用户视角可以理解为某张入库单对应的价格来源。

但系统内部不要只记入库单据号，至少要追到：

- `入库单行`
或
- `inventory_log.sourceLogId`

否则同一张单里有多行同物料时会混。

## 成本规则

一条出库行的成本口径：

- 用户选定价格层
- 系统在该价格层内自动 FIFO
- `costAmount` 由实际来源分配汇总
- `costUnitPrice` 作为历史成本快照保存

如果同一价格层内部单价一致，则：

- `costUnitPrice = 选定价格`
- `costAmount = 数量 × 选定价格`

## 对数据库的影响

### 第一版建议

不强制新增“价格库存余额主表”。

优先做：

- 基于现有来源层聚合出 `物料 + 价格` 的可用库存查询
- 出库按价格层选择
- 出库成功后仍写具体来源占用

### 可能需要确认的技术点

如果当前销售出库单行里的 `unitPrice` 表示销售价，而不是成本价，那么：

- 不能直接复用 `unitPrice` 作为价格层字段
- 需要新增一个专门字段，例如 `selectedUnitCost`

如果当前 `unitPrice` 本来就是库存成本价，则可以复用现有字段口径。

## 最小验收标准

1. 某物料存在 104 元库存 100 个、117 元库存 200 个时，界面能展示这两个价格层
2. 选择 `104元 + 50` 可提交
3. 选择 `104元 + 101` 不可提交
4. 同一单据重复录入 `物料A + 104元` 第二条时，不可提交
5. `104元` 这条只能消耗 104 元价格层库存，不能自动借用 117 元库存
6. 后台可追溯到每条出库行最终用了哪些具体来源层
7. 出库完成后保留历史成本快照

## 当前结论

这套方案的核心不是“按批次录单”，而是：

- 前台按价格层录单
- 后台按同价内 FIFO 追溯

这样既不增加用户复杂度，又能把成本核算和追溯做对。
