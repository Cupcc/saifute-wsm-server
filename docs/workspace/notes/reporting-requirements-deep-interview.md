# 报表需求 Deep Interview 草稿

**version**: v1.0 | **domain**: reporting / dashboard / export | **status**: 探索中

## 文档说明

这份文档不是正式需求草案，也不是最终 PRD。

在“报表需求”尚未澄清前，只记录每一轮 deep interview 的对话内容、当前假设、已知边界和待确认问题，用作后续正式文档的输入材料。

约束：

- 当前阶段不下正式结论
- 当前阶段不提前冻结字段、交互或实现方案
- 每轮只问一个最高杠杆问题
- 每个问题必须带选项，并说明各选项的含义与影响

## 当前仓内事实

基于现有代码与架构文档，可先确认以下事实：

- `reporting` 在架构上被定义为只读聚合层，不拥有事务写模型
- 当前已有接口包括：
  - `GET /api/reporting/home`
  - `GET /api/reporting/inventory-summary`
  - `GET /api/reporting/material-category-summary`
  - `GET /api/reporting/trends`
  - `GET /api/reporting/export`
- 架构文档已明确“首页 / 库存报表 / 月报 / Excel 导出 / AI 查询”都属于 `reporting` 的输出形态
- 架构文档还提到三类语义应区分：
  - 正式月报
  - 人工重算结果
  - 日期范围报表

这些事实说明：当前系统不是“没有报表”，而是“缺少一份清楚描述报表目标、边界和语义的需求文档”。

## 当前假设

当前最需要澄清的不是字段明细，而是：

- 这份“报表描述”到底是为了支持哪类决策
- 它优先补的是哪一类报表语义
- 它应明确纳入哪些内容，明确排除哪些内容

如果这个问题不先定清，后续“月报、库存汇总、项目报表、经营统计、导出模板”会混成一份模糊文档，后面很容易返工。

## 对话记录

### Round 1

**提问目标**：先确认“这份报表描述最主要服务什么目的”，用来决定后续整份文档的主结构。

**问题**：

这次要补的“报表需求描述”，你希望它首先解决哪一种核心问题？

**选项**：

1. `A. 经营总览`
   说明：重点是给老板或管理层看整体经营状态，例如入库、出库、库存、金额、趋势、异常预警。文档会优先定义首页看板、汇总指标和趋势口径。
2. `B. 业务对账`
   说明：重点是让业务人员核对“数量/金额/单据”是否一致。文档会优先定义明细报表、筛选条件、导出字段、对账口径和差异解释。
3. `C. 成本追溯`
   说明：重点是让人追到“这笔出库/领料/项目消耗的成本从哪里来”。文档会优先定义来源链、批次/价格层/FIFO 追溯和历史快照口径。
4. `D. 月报沉淀`
   说明：重点是形成正式月报体系。文档会优先定义正式月报、重算结果、日期范围报表三者的关系，以及生成、冻结、重跑和审计规则。
5. `E. 项目/RD 专题报表`
   说明：重点是围绕销售项目或研发项目看投入、消耗、库存和结果。文档会优先定义项目维度口径，而不是全局库存总览。
6. `F. 先做报表地图，不先押某一类`
   说明：你的目标是先把“系统里应该有哪些报表”分类盘清，再决定先写哪一类细化需求。这样更稳，但第一阶段会偏框架梳理，落地速度略慢。

**用户回答**：

- 选择 `B. 业务对账`

**当前理解**：

- 这份报表需求描述的第一优先目标，不是经营驾驶舱，也不是先做正式月报，而是先支持业务人员核对“数量 / 金额 / 单据”是否一致
- 因此后续访谈会优先澄清：
  - 先对哪一类账
  - 对账时以什么口径为准
  - 差异出现时，报表要给出什么解释能力

**当前状态**：已确认主目标，进入下一轮。

### Round 2

**提问目标**：在“业务对账”方向已经明确后，继续确认第一优先要对的是哪一类账。

**问题**：

如果第一版报表描述以“业务对账”为核心，你最想先解决的是哪一种对账场景？

**选项**：

1. `A. 库存实存对账`
   说明：重点核对“系统库存数是否可信”，优先关心现存量、可用量、预警、按仓/物料/分类汇总，以及为什么某个库存余额会变成现在这样。
2. `B. 单据流水对账`
   说明：重点核对“入库、出库、领料、退料这些单据汇总后是否对得上”，优先关心业务单据、流水汇总、时间区间、来源去向和异常单据。
3. `C. 金额成本对账`
   说明：重点核对“数量对了以后，金额和成本是否也对”，优先关心单价、成本额、来源价格层、历史快照和金额差异解释。
4. `D. 月末结存对账`
   说明：重点核对“某个月的期初、发生、期末是否闭环”，优先关心月结口径、期间汇总、正式结果与重算结果的差异。
5. `E. 先做综合对账地图`
   说明：暂时不先压某一种，而是先定义系统里有哪些账、它们之间怎么关联、各自由哪些报表承担。这样更全面，但第一版不会很快落到某一张具体报表。

**用户回答**：

- 选择 `B + C + D`

**当前理解**：

- 用户理解中的“业务对账”不是单点查询，而是一条连续闭环：
  - 先核对 `单据流水`
  - 再核对 `金额 / 成本`
  - 最后落到 `月末结存`
- 这说明第一版报表描述不能只停留在库存余额或单据列表层面，而要交代不同报表之间的衔接关系
- 但当前仍未明确：
  - 第一版文档应以哪一类对账作为主线
  - 哪些内容只是配套章节，而不是第一版的核心交付

**当前状态**：已确认对账链路范围，进入下一轮。

### Round 3

**提问目标**：确认第一版报表描述的主线，避免 `单据流水 / 金额成本 / 月末结存` 三块同时展开导致文档失焦。

**问题**：

在你选的 `B + C + D` 里，第一版文档最应该以哪一种主线展开？

**选项**：

1. `A. 以单据流水为主线`
   说明：先定义“所有对账都从业务单据和库存流水开始”。金额成本与月末结存作为下游结果来描述。这样最适合先把事实链和来源去向讲清。
2. `B. 以金额成本为主线`
   说明：先定义“对账的核心是金额要说得清”。单据流水只作为金额的来源证据，月末结存作为金额汇总结果。这样最适合强调成本、差异和核算解释。
3. `C. 以月末结存为主线`
   说明：先定义“所有对账最终都服务于月结闭环”。单据流水和金额成本都作为月报形成过程中的支撑层。这样最适合面向月结和正式报表沉淀。
4. `D. 以三段闭环为主线`
   说明：明确把第一版写成一条链路：`单据流水 -> 金额成本 -> 月末结存`。这样结构最完整，但文档复杂度更高，第一版更容易写大。
5. `E. 先只写边界，不定主线`
   说明：先把这三类报表各自负责什么、不负责什么写清楚，暂时不决定主从关系。这样更稳，但第一版仍然不够收敛。

**用户补充**：

- “主要是仓库需要和财务进行对账”
- 财务角色目前脱离本系统，财务使用自己的系统
- 财务最关心的是 `金额`，不是物料数量
- 数量与单据主要用于：
  - 帮助核对金额是否正确
  - 当金额不一致时做追溯排查

**基于当前上下文的工作建议**：

- 将原建议进一步收敛为：
  - 以 `月末金额闭环` 为主线
  - 数量与单据作为金额核对的证据层

**推荐理由**：

1. 现在已明确财务不在系统里，所以系统内报表更像“给仓库拿去和外部财务账核对”的证据面，而不是给财务直接操作的事务系统
2. 财务关注的是 `金额` 是否对得上，因此第一版文档的顶层对象不应是库存数量本身，而应是“月末金额结果及其形成过程”
3. 数量和单据仍然重要，但它们在当前场景下是金额核对的解释层、追溯层，而不是顶层目标
4. 当前仓内已有 `docs/requirements/domain/monthly-reporting.md`，并且架构文档已经冻结了“正式月报 / 人工重算 / 日期范围报表”的三类语义，因此最稳的切法不是做一张单纯金额表，而是做 `月末金额闭环`

**但必须同时保留的下钻链路**：

- 即便以 `月末金额闭环` 为主线，正文里也不能只有汇总结果
- 应明确下钻关系：
  - `月末金额结果`
  - `-> 金额 / 成本构成`
  - `-> 数量变化依据`
  - `-> 单据流水与库存流水证据`
- 也就是说，推荐的不是“只做金额、不管数量”，而是：
  - `金额` 作为对账目标
  - `数量 + 单据` 作为核对与追溯证据

**当前状态**：主线判断继续收敛，进入下一轮。

### Round 4

**提问目标**：在“金额是核心、数量与单据是证据层”已经明确后，继续确认第一版最需要和财务对齐的是哪一种金额对象。

**问题**：

如果第一版是给仓库拿去和外部财务系统对账，你最优先想对齐的是哪一种金额？

**选项**：

1. `A. 期末库存金额`
   说明：重点核对“这个月结账时仓库账上的库存金额是多少”，适合先解决期末余额是否可信，但对期间发生解释较弱。
2. `B. 本期出入库发生金额`
   说明：重点核对“这个月发生了多少入库金额、出库金额、退料回补、报废损耗等”，适合先抓期间变动，但单看它不足以闭合期末余额。
3. `C. 收发存金额闭环`
   说明：按 `期初金额 + 本期入 - 本期出 +/- 调整 = 期末金额` 这条链核对。最完整，也最符合月末仓库对财务的对账逻辑，但口径要求最高。
4. `D. 差异调整金额`
   说明：重点只看那些导致仓库账与财务账不一致的差异金额，例如补录、调价、盘点调整、跨期修正。适合快速查问题，但不能替代完整月报。
5. `E. 分两层：先闭合期末库存金额，再追本期发生金额`
   说明：先保证期末余额能对上，再逐步解释期间发生。比 `C` 更易推进，但需要明确哪一层是第一版必交付、哪一层是后续增强。

**用户回答**：

- 选择 `B. 本期出入库发生金额`
- 补充理由：
  - 按月核算
  - 上月数据可能不准
  - 历史数据可能因系统崩溃等原因不可靠
  - 因此基于“本月变动”来核对更靠谱
  - 希望做到“一月核对一月”

**对前述建议的修正**：

- 先前“以月末金额闭环为主线”的建议，不再作为第一版优先方案
- 当前更准确的第一版主线应修正为：
  - `按月核对本期变动金额`
  - 历史余额不作为第一版可信前提
  - 数量与单据继续作为金额核对和差异追溯的证据层

**当前理解**：

- 第一版报表的第一目标不是证明“历史累计余额都正确”
- 而是先把“某一个月内发生了哪些金额变动”讲清楚、核清楚
- 这意味着第一版应优先支持：
  - 按自然月固定区间统计
  - 月内各类业务变动金额汇总
  - 金额异常时可追到数量依据与单据证据
- 这也意味着第一版可以暂时把：
  - 历史结转余额可信性
  - 全链收发存余额闭环
  - 跨多年历史修复
  放在后续阶段处理

**当前状态**：已确认第一版金额对象，进入下一轮。

### Round 5

**提问目标**：在“按月核对本期变动金额”已经明确后，继续确认第一版报表里，本期金额应该按什么口径拆分，才能真的和财务对上。

**问题**：

第一版月度对账报表里，你最希望“本期变动金额”按什么方式拆开给仓库核对财务？

**选项**：

1. `A. 按仓库业务动作拆分`
   说明：按系统里的真实业务类型分开统计，例如入库、销售出库、车间领料、退料、报废、盘点调整、调价等。最贴近仓库操作和单据追溯，但需要仓库自己再理解它和财务科目的对应关系。
2. `B. 按财务对账口径拆分`
   说明：直接按财务拿来核对的金额栏目或分类输出，例如采购入账、生产耗用、销售发出、损耗调整等。最贴近财务核对，但系统内必须补清“仓库业务 -> 财务口径”的映射关系。
3. `C. 两层都要`
   说明：第一层按财务对账口径汇总，第二层可下钻到仓库业务动作、数量和单据。最适合跨系统对账，但第一版复杂度最高。
4. `D. 先只看本月总发生金额`
   说明：先只给出本月总入金额、总出金额、总调整金额三大数。推进最快，但一旦对不上，排查帮助有限。
5. `E. 先只盯异常类金额`
   说明：第一版优先看报废、盘点调整、调价、补录修正这类最容易导致金额差异的项目。适合先抓问题源，但不能覆盖完整月度对账。

**用户回答**：

- 选择 `C. 两层都要`

**当前理解**：

- 第一版报表不应只给出仓库业务动作金额，也不应只给出财务侧汇总数
- 更合适的结构是：
  - 第一层：`财务对账口径汇总`
  - 第二层：`仓库业务动作 -> 数量依据 -> 单据 / 流水证据`
- 这意味着系统内报表的职责不是替代财务系统，而是：
  - 先产出能让仓库直接拿去对财务的金额汇总
  - 再在系统内提供追溯路径，帮助解释为什么这个金额成立或为什么对不上

**当前风险提醒**：

- 这种双层结构最符合实际对账，但如果第一版不先限制金额类别范围，很容易扩成“大而全月报”
- 所以下一轮必须优先确定：
  - 第一版先覆盖哪些金额类别
  - 哪些类别明确后置

**当前状态**：已确认双层结构，进入下一轮。

### Round 6

**提问目标**：在“双层结构”已经明确后，继续确认第一版先覆盖哪些本月变动金额类别，避免范围失控。

**问题**：

第一版里，财务对账口径汇总最优先应该覆盖哪一组“本月变动金额”？

**选项**：

1. `A. 采购 / 入库类金额`
   说明：先对采购到货、验收入库、生产入库这类“进”的金额。适合先解决入账金额核对，但不能覆盖耗用和损耗。
2. `B. 领用 / 出库类金额`
   说明：先对销售出库、车间领料、项目领用这类“出”的金额。适合先解决成本结转或耗用核对，但不能覆盖入库端。
3. `C. 调整 / 异常类金额`
   说明：先对报废、盘点调整、调价、补录修正这类最容易造成差异的金额。适合先抓问题源，但不能代表完整月度变动。
4. `D. 入 + 出 + 调整都要`
   说明：第一版直接覆盖本月所有主要变动金额类别，形成一份完整“本期发生金额”对账表。最完整，但第一版范围最大。
5. `E. 先做一张总表，再逐步细分类别`
   说明：第一版先给总入、总出、总调整三大类金额，后面再慢慢拆到具体业务类型。更容易推进，但第一轮排查能力有限。

**用户回答**：

- 选择 `D. 入 + 出 + 调整都要`
- 补充约束：
  - 后续若非核心问题，直接采纳建议即可

**当前理解**：

- 第一版不是局部试点，而是要形成一份完整的“本期发生金额”对账表
- 覆盖范围应同时包括：
  - `入库类金额`
  - `出库 / 领用类金额`
  - `调整 / 异常类金额`
- 这样做的意义是：
  - 仓库拿到一张完整月度发生金额报表即可去对财务
  - 若金额不一致，再在系统内按业务动作、数量、单据继续排查

**访谈策略更新**：

- 从这一轮开始：
  - 非核心问题默认采用建议方案，不再逐一反复确认
  - 只继续追问会改变产品边界、数据模型或验收口径的核心问题

**当前默认建议（作为暂定口径，后续如无冲突直接沿用）**：

1. `时间口径`
   说明：默认按自然月统计，不做跨月滚动窗口。
2. `第一层栏目命名`
   说明：默认优先采用财务更容易理解的对账口径名称，同时保留仓库业务动作映射关系。
3. `第二层排查粒度`
   说明：默认至少能下钻到单据头与业务类型；是否细到单据行，留待后续核心问题再判断。
4. `结果形态`
   说明：默认系统内可查看，并支持导出给仓库与财务离线核对。

**当前状态**：已确认第一版金额类别范围，进入下一轮。

### Round 7

**提问目标**：确认系统内是否需要直接承接“财务侧对比数”，因为这会直接决定第一版只是“仓库侧金额报表”，还是“系统内差异对账工具”。

**问题**：

第一版里，你希望系统只生成“仓库侧本月发生金额报表”给人拿去和财务系统比，还是希望系统内能直接录入/导入财务侧数字并自动算差异？

**选项**：

## 文档调查补充

### 对 `docs/requirements/domain/monthly-reporting.md` 的调查结论

结合当前仓内真源文档、现有 `reporting` 实现和本轮 deep interview，可得出下面结论。

#### 1. 这份文档的定位没有问题，但范围明显大于当前眼前需求

`monthly-reporting.md` 目前定位为：

- 报表域长期真源
- 面向管理核对、经营分析、补录复盘
- 覆盖公司整体、车间、销售域、销售项目、`RD` 小仓协同五类月报视角

这个定位本身是成立的，且与架构文档一致。

但当前 deep interview 明确出的即时诉求是：

- 主要是 `仓库需要和财务进行对账`

这比 `monthly-reporting.md` 当前覆盖的范围窄得多，也更偏“月末核对闭环”而不是“整套报表域都定义完”。

#### 2. 这份文档已经给出了正确骨架，但还没有把“仓库-财务对账”作为第一优先主线写透

文档里已经有不少正确方向：

- 强调月报是 `可核对、可追溯、可导出`
- 强调结果来自 `业务单据 + 库存流水 + 成本分配`
- 强调 `正式月报 / 人工重算 / 日期范围报表` 三类语义分离
- 强调查询既要有汇总，也要能回到明细

但从“仓库与财务对账”视角看，仍有几个缺口没有被明确写成第一层对象：

- 财务侧用户与责任边界没有被点名写入“使用者”
- 没有明确声明“第一优先对账对象”是 `期初 + 本期发生 + 期末结存`
- 没有明确写出“仓库账”和“财务账”如何对齐，差异要按什么类型解释
- 没有明确第一阶段先收敛哪些月报视角，避免五类视角一起启动

#### 3. 当前代码实现离这份长期文档还有明显距离

现有 `reporting` 模块已实现的主要是：

- `home`
- `inventory-summary`
- `material-category-summary`
- `trends`
- `export`

这说明当前系统已有基础查询和导出表面，但还没有落到：

- 正式月报结果沉淀
- 人工重算结果管理
- 日期范围报表与正式月报语义分离
- 面向仓库-财务对账的月末闭环结构

所以，如果现在直接把 `monthly-reporting.md` 当作“下一步马上全部实现”的需求，会过大。

## 当前建议

### 建议一：不要新起一份与 `monthly-reporting.md` 平行竞争的正式文档

建议把 `monthly-reporting.md` 继续保留为：

- 报表域长期真源
- 月报总纲

而把当前 deep interview 的成果收敛成：

- “月报主题下的第一优先切片”
- 或者说，“仓库-财务对账口径收敛稿”

也就是说：

- 长期真源仍是 `monthly-reporting.md`
- 当前草稿用于把第一阶段真正要落地的范围钉小、钉实

### 建议二：第一版正式需求应以“仓库-财务月末对账”作为 `monthly-reporting` 的首个落地切片

建议把第一版主线收敛为：

- `月末结存对账`

建议的结构是：

1. `期初`
2. `本期入`
3. `本期出`
4. `本期调整`
5. `期末结存`
6. `期末金额 / 成本`
7. `差异解释`
8. `下钻到金额构成`
9. `下钻到单据流水 / 库存流水`

换句话说，第一版不是先铺开五类月报，而是先把“仓库和财务怎么核这一个月”讲清楚。

### 建议三：第一版明确缩小范围，避免月报文档继续失焦

建议第一版正式需求只先覆盖：

- 公司整体月报
- 主仓 / `RD` 小仓按仓别隔离的月末对账
- 与月末结存直接相关的数量、金额、成本字段
- 汇总到明细的追溯链
- 正式月报与人工重算结果分离

建议第一版先不展开为主角的内容：

- 销售项目月报专题
- `RD` 协同全景专题
- 面向经营分析的趋势 / 看板扩展
- 所有业务域同时并行的多视角页面设计

这些内容可以继续保留在 `monthly-reporting.md` 的长期范围内，但不应成为当前第一阶段的实现压力。

### 建议四：后续正式需求里必须补上 4 个现在还不够尖锐的口径

从仓库-财务对账视角，建议在未来正式需求中补充以下 4 类关键口径：

1. `对账对象`
   说明：到底对的是库存数量、库存成本金额、还是两者同时；各自以什么结果为准。
2. `对账时点`
   说明：正式月报何时冻结；月末后补录单据如何进入重算，不得模糊。
3. `差异分类`
   说明：差异至少要能区分为单据遗漏、时间跨期、成本纠偏、库存调整、口径差异等类型。
4. `责任与操作`
   说明：仓库负责核对什么，财务负责核对什么，发现差异后由谁发起重算或复核。

## 暂定结论

当前最合理的方向不是推翻 `monthly-reporting.md`，而是：
