## Topic 边界判断

### 建议纳入第一版的能力

- `用户 / 角色 / 部门 / 菜单 / 字典 / 参数 / 通知`
- `在线用户`
- `登录日志`
- `操作日志`

### 建议暂不纳入第一版的能力

- `调度任务`
- `AI 支持`

### 关键边界

- `rbac` 继续作为权限字符串、路由树、按钮可见性、角色菜单关系与数据权限的统一真源。
- `session` 继续作为 `JWT + Redis session` 的运行态真源；在线用户属于管理面，不反向决定授权。
- `audit-log` 继续作为登录日志、操作日志的记录真源；审计不反向改写权限结果。
- `consoleMode` 只是视角裁切，不替代真实授权。
- 组织树不能直接替代 `stockScope / workshopScope` 等业务库存范围。

## 已收敛结论

- `F2` 最终结论：真实组织按 `研发部 / 采购部 / 仓库` 三部门建模；主角色按 `系统管理员 / 仓库管理员 / 研发小仓管理员 / 采购人员` 设计；`老板 / 财务` 只保留预留查看位；岗位当前不单独维护；账号维护职责以 `系统管理员` 为主，业务角色不维护角色权限。
- `F3` 最终结论：`在线用户 / 登录日志 / 操作日志` 纳入 `system-management` 第一版长期主题，作为平台治理、排障与审计能力统一规划；`调度 / AI 支持` 仅保持邻接关系，不纳入当前长期主题边界。
- 上述结论已在 2026-03-31 同步收口到 topic、项目级需求、架构文档、需求/任务中心与 workspace dashboard。

## 后续追记

- 真实 Chrome 回归曾暴露出 3 个实现缺口：

  - 前端 `userStore` 已承接 `department`，`operator / procurement` 首页“所属部门”会显示真实部门。
  - `admin / operator / rd-operator / procurement` 四类账号再次验证通过，`system/*` 对非管理员仍保持 404 收口。
