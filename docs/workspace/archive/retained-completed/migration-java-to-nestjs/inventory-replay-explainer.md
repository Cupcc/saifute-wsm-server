# 库存重放说明

## 一句话结论

迁移分两步：第一步"数据搬家"，第二步"库存副作用重放"。两步都已完成。

## 什么是库存重放

在新系统里，一张业务单据（出库单、项目单、退货单等）被"生效"时，会自动产生一组库存副作用：

| 副作用 | 对应表 | 含义 |
|--------|--------|------|
| 库存余额变动 | `inventory_balance` | 某个物料在某个仓库的当前库存数 |
| 库存流水 | `inventory_log` | 一条"某时某单导致某物料增/减了多少"的记录 |
| 来源追踪 | `inventory_source_usage` | 某次出库消耗的是哪次入库的库存 |

迁移脚本先把单据数据写进新系统的表里，但没有触发上面这些库存副作用。

"库存重放"就是：拿迁入的单据，按时间顺序模拟一次"生效"，让新系统补上库存余额和流水记录。

## 为什么不能在数据搬家时一起做

- 库存余额是所有域共享的：出库、入库、项目、退货等单据都会影响同一批物料的库存数。
- 如果只搬了一半域就开始重放，中间状态的库存余额是错的。
- 所以必须等所有域的数据都搬完，再统一重放一次。

- **已知限制**: 230 个 `(materialId, workshopId)` 余额为负值。原因是迁移数据只包含单据的增减变动，不包含开业时的初始库存余额。如果你的旧系统有初始库存数据可以导入，可以补录；如果没有，上线后系统会从当前余额开始正常记账。

## 如果现在要停旧启新

迁移脚本是幂等的（重跑不会重复插入），所以如果现在要从旧系统切到新系统，操作流程很简单：

1. 在旧系统停止录入新业务数据
2. 重跑一次迁移脚本，把最后一批数据搬过来
3. 开始使用新系统

这是一份独立的运维切换说明，不再作为"迁移是否完成"的 blocker。
