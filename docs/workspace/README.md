# Workspace — 决策与 Draft 工作区

Workspace 是面向人类决策者的工作区，也是用户与 AI 头脑风暴的持久化草稿区，帮助人类在最短时间内理解当前状况、逐步挖掘真实意图、做出最优决策。

## 定位

| 文档层                      | 回答的问题                               | 服务对象                 |
| ------------------------ | ----------------------------------- | -------------------- |
| `docs/requirements/*.md` | 用户要什么 + 执行状态                        | 意图确认                 |
| `**docs/workspace/**`    | **当前到哪了、要决定什么、有哪些想法仍在草拟、AI 下一步做什么** | **人类决策者 + AI 恢复上下文** |
| `docs/tasks/*.md`        | 怎么执行、谁负责、交接什么                       | agent 执行链            |

Workspace 是**工作交代区 + 探索草稿区**：既是人类了解全局的入口，也是 AI 恢复上下文后判断“下一步做什么”的起点。它不替代 requirement 或 task doc，而是把分散在各处的待办、待确认、待实施，以及尚未定稿但值得持久化的想法，聚合到一个可扫、可续写的工作区中。

## 核心职责

1. **待确认看板**——用户需要确认或决策的事项，从各工作流中聚合到 Dashboard 顶部
2. **AI 待办看板**——AI 可以推进的任务及其优先级、当前状态、阻塞原因
3. **进展叙事**——用人话讲当前到哪了，不是 agent 能读的 metadata
4. **选项 + Trade-off**——每个待决策项列出选项、代价、风险、推荐
5. **决策日志**——已做出的决策及理由，后续可追溯
6. **Draft 持久化**——把用户与 AI 的头脑风暴、意图假设、澄清过程沉淀下来，而不是只留在聊天里
7. **意图挖掘**——帮助 AI 从模糊表达中提炼“用户真正要什么”，再升级为 requirement 的待确认项
8. **干预判断**——当前是否需要人类介入，为什么

## 目录结构

```text
docs/workspace/
├── README.md                           # 本文件：机制说明
├── DASHBOARD.md                        # 全局决策/推进面板
├── <workflow-name>/                    # 活跃工作流子文件夹
│   ├── README.md                       # 工作流入口与当前状况
│   ├── draft.md                        # 可选：头脑风暴/意图挖掘草稿
│   └── decisions.md                    # 可选：待决策与已决策
├── archive/
│   ├── retained-completed/
│   │   └── <workflow-name>/            # 已完成但保留溯源/说明的工作流
│   └── cleanup-candidate/
│       └── <workflow-name>/            # 待用户确认后才可删除的工作流
```

## 命名规则

- 子文件夹使用纯语义名（如 `migration-java-to-nestjs`），不加时间戳
- 每个子文件夹必有 `README.md` 作为入口
- 简单工作流可以只有一个 `README.md`，不强制丰富结构
- `draft.md` 用于探索期草稿，记录用户意图整理稿与简洁对话留痕；不需要时可以不创建
- 决策日志统一命名 `decisions.md`，在工作流复杂度需要时创建

## Dashboard 规则

`DASHBOARD.md` 是人类打开 workspace 的第一个入口，也是 AI 恢复后判断下一步的第一个入口：

- 「当前状态」永远最顶：一句话说清当前全局是什么状态——等人确认 / AI 正在执行 xxx / 空闲无待办。用加粗关键词开头，后接简要说明
- 「需要你确认的」紧随其后：从各活跃工作流中提取已经整理好的待用户回答问题，逐条列出；无待确认项时写明"当前无待确认项"
- 「AI 待办」紧随其后：列出 AI 可以推进的任务、优先级、当前状态和阻塞说明；若某任务依赖用户确认，标注"阻塞于上方确认项"
- 健康度只用三个状态：`●` 就绪 / `⚠` 有阻塞 / `○` 等待输入
- 一行一个工作流，不展开细节
- Dashboard 只展示从 `draft.md` 中提炼后的确认项与推进状态，不直接贴原始脑暴或长对话
- 已归档工作流折叠在底部
- 某工作流一旦归档，`DASHBOARD.md` 链接也必须改为 `archive/**` 下的新路径；不要只在看板写"已归档"而把文件夹继续留在根目录
- 每次 agent 结束 turn 前，若有工作流状态变更，应同步更新 Dashboard 的待确认项和 AI 待办

## 子文件夹 README.md 结构

```markdown
# [工作流名称]

关联需求: `docs/requirements/req-xxx.md`
关联任务: `docs/tasks/task-xxx.md`（若有）
阶段: 执行中 / 需求确认 / 已完成 / 已归档
创建: YYYY-MM-DD
最后更新: YYYY-MM-DD

## 当前状况

[2-3句话描述当前进展，面向人类阅读]

## 待决策项

[列出或链接到 decisions.md#pending]

## 草稿入口

[若存在 draft.md，用 1 句话说明它记录什么以及何时查看]

## 背景与上下文

[帮助理解的关键背景，可引用图片、表格等资产]

## 关键里程碑

| 时间 | 事件 |
|------|------|

## 本文件夹资产索引

| 文件 | 用途 |
|------|------|
```

工作流已完成时，`待决策项` 可以明确写 `当前无待决策项。`；不要为了保持模板完整而再造一个待决项。
若存在 `draft.md`，应在 `本文件夹资产索引` 中列出；若当前脑暴已收敛完毕，也可以保留 `draft.md` 作为溯源，不必强行删除。

## decisions.md 结构

decisions.md 保持简洁：每条决策只写结论性信息。需要详细解释的背景、概念或 trade-off 分析放到独立的 `*-explainer.md` 文档，decisions.md 中用链接引用。

```markdown
# 决策日志

## 待决策 {#pending}

### DEC-NNN: 标题
- 提出时间: YYYY-MM-DD
- 影响范围: ...
- 紧急度: ...

**背景**: [1-2句话概述，详见 xxx-explainer.md]

| 选项 | 描述 | 代价 | 风险 |
|------|------|------|------|

**支撑数据**: [链接到本文件夹中的数据资产或 explainer]

---

## 已决策

### DEC-NNN: 标题
- 决策时间: YYYY-MM-DD
- 结论: ...
- 理由: ...
- 执行: [在哪个 task/代码中落地]
```

## *-explainer.md 约定

- 命名：`<主题>-explainer.md`，如 `inventory-replay-explainer.md`
- 用途：为 decisions.md 中的待决策项或复杂概念提供面向人类的详细解释
- 内容：用非技术语言写清"这是什么 → 为什么需要 → 当前状态 → 你需要做什么"
- decisions.md 中通过链接引用，不内联大段解释

## draft.md 结构

`draft.md` 用来记录尚未定稿、但值得保留的探索内容。它服务于继续讨论，不承担 requirement 或 decision 的确认职责。

最小结构：

```markdown
# 草稿

关联需求: `docs/requirements/req-xxx.md`（若已有）
最后更新: YYYY-MM-DD
状态: 探索中 / 已提炼 / 已收敛
文档说明: 记录当前讨论主题、关键判断与后续接手点。

## 用户意图

[列出当前已明确的用户意图；下面的小节按需要自由展开]

## 对话留痕

| 时间 | 来源 | 关键信息 |
|------|------|----------|
```

写作规则：

- 先写已明确的用户意图，再按主题自由补充讨论内容
- 小节不固定；有需要就分节，没需要就用短段落或短列表
- 只保留对后续理解和接手有价值的信息，不复制完整聊天记录
- `对话留痕` 的 `来源` 至少区分 `用户`、`AI 建议`、`代码现状`、`文档说明`
- `关联需求` 不是必填；前置脑暴阶段可以先不写

流转规则：

- 仍在探索、尚未确认的内容放入 `draft.md`
- 可发给用户确认的问题提炼到 `docs/requirements/*.md` 的 `待确认`
- 已明确的方案选择或结论写入 `decisions.md`

## 维护职责

- **写入权：Parent orchestrator 独占**
- Subagent 通过 handoff 协议中的 `decision_candidates` 字段供料
- Parent 收到后判断：需要人类决策或值得持久化的探索内容 → 写入 workspace；只是执行细节 → 留在 task doc
- 工作流归档时将子文件夹移入 `archive/` 并更新 Dashboard；归档路径、Dashboard 行、README 阶段三者必须一致

## 与其他层的关系

- Workspace 引用 requirement doc 和 task doc，但不重复它们的内容
- Requirement doc 保留 `用户需求`、`当前进展`、`待确认` 等固有结构，不因 workspace 存在而瘦身
- Task doc 保留完整的执行 brief 和 handoff 结构
- Workspace 的独特价值不仅是 trade-off 分析、选项比较、决策理由，还包括探索期草稿、意图假设与对话收敛过程的持久化
- `draft.md` 不是 requirement 的替代品：未经提炼与确认的内容，不应直接当作需求真源
- 但 `draft.md` 可以先于 requirement 独立存在；是否关联需求，取决于当前是否已经进入 requirement 层
- `draft.md` 也不是 task 的替代品：执行计划、验证、review 结论仍然属于 `docs/tasks/*.md`
