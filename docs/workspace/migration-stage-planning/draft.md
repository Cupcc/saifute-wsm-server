说明：

- 这里讨论的不是“Java 数据是不是已经搬完”，而是“如果把整个项目当作从 `0` 开始建设，它应该如何分阶段推进，才能真正指导开发方向”。
- 范围不是纯后端替换，而是以 `Java -> NestJS` 为主线，同时覆盖 `需求确认`、`底座承接`、`数据迁移`、`核心业务域开发`、`必要前端适配`、`正确性完善`、以及 `切换前准备`。
- 这份草稿首先服务于用户自己判断“如果从项目 `0` 到最终形态拉一条完整主线，当前处在哪一段、下一步该投哪里”，不是先为了汇报包装阶段名称。

## 用户意图

- 用户希望通过多轮对话逐步澄清真实诉求，而不是一开始就把所有阶段和边界说死。
- 项目需要一条总主线来判断整体推进方向，但具体到模块和业务域时，也要允许每块走自己的小闭环。
- 用户真正想看的更像“每个阶段交付了哪些成果”，而不只是抽象的“成功标准”。
- 前端不宜再机械拆成两个孤立的大阶段，更适合跟着每个业务域的小闭环一起推进。
- 你真正需要的不是一张静态路线图，而是一张能指导“现在该做什么、为什么先做这个、做到什么算这一段成立”的推进图。
- 整个项目应该有一条总主线：`需求确认 -> 架构确认 -> 底座承接 -> 迁移规则确认 -> 迁移能力构建与全量迁移 -> 后端模块开发完整 -> 前后端联调适配 -> 切换准备`。
- 在这条主线上，人工审查不是某一个特殊阶段，而是每一个阶段收口进入下一阶段前都要发生的动作。
- 前端在这条主线上主要承担“适配和联调”的角色，而不是在中段插入一次大规模页面重构。
- 这张阶段图最终应服务于开发决策，而不是先服务于汇报。

### 项目开发主线

## AI建议的阶段模型

当前更适合采用“项目总主线 + 模块小闭环”的混合结构：

- 总主线用于判断整个项目从 `0` 到最终形态正在走到哪一段、下一步资源该往哪里投。
- 小闭环用于约束每个核心业务域都走完 `确认 -> 实现 -> 验收`，避免一直停留在“功能开发中”的模糊状态。

阶段候选稿：

1. `阶段1：需求确认`
  - `工作内容`：明确项目要解决的业务问题、目标范围、核心业务域、优先级与边界。
  - `交付结果`：形成后续判断“做什么、不做什么”的共同真源。
2. `阶段2：架构确认`
  - `工作内容`：确认系统分层、业务域拆分、模块边界、数据模型主线、迁移主线与整体技术方案。
  - `交付结果`：形成后续建设的结构真源。
3. `阶段3：底座承接`
  - `工作内容`：搭好 NestJS 可持续开发的底座，明确共享基础能力、认证会话权限、主数据、库存核心等后续承接能力。
  - `交付结果`：形成边界清晰、可以持续承接后续开发的系统底座。
4. `阶段4：迁移规则确认`
  - `工作内容`：梳理旧系统到新系统的映射规则、迁移范围、迁移顺序、核对口径与差异处理方式。
  - `交付结果`：形成可供下一阶段实现和执行的迁移方案。
5. `阶段5：迁移能力构建与全量数据迁移`
  - `工作内容`：构建迁移功能并执行旧系统到新系统的全量数据迁移，完成整体数据落库、关联关系承接、迁移结果核对、问题修正与复跑。
  - `交付结果`：形成完整数据底账，以及经执行验证有效的迁移功能和迁移规则。
6. `阶段6：后端基础功能模块和核心业务功能模块开发完整`
  - `工作内容`：完成后端基础功能模块和核心业务功能模块开发。
  - `交付结果`：形成完整可用的后端业务能力，以及可供前端接入和联调的稳定接口。
7. `阶段7：前后端联合调试适配`
  - `工作内容`：进行前后端联调、旧页面接口适配、权限菜单串接和流程验证，前端以适配现有页面为主。
  - `交付结果`：形成前后端已经打通、主要流程可操作的系统，不以大改页面风格和整体布局为目标。
8. `阶段8：正确性与运营完善`
  - `工作内容`：完善库存语义、来源追溯、改单作废、报表、权限、仓别隔离等正确性与运营能力。
  - `交付结果`：形成正确、可信、可长期运行的系统。
9. `阶段9：切换准备与试运行`
  - `工作内容`：完成试运行、补录修正、数据核对、异常回退与角色侧真实使用准备。
  - `交付结果`：形成可进入正式切换的运行准备状态。
10. `阶段10：非核心与持续优化`
  - `工作内容`：按价值逐步补齐非核心模块、体验、性能、效率工具和辅助能力。
  - `交付结果`：让系统从“能用”继续走向“好用、稳用、低成本维护”。

## 待确认点

- `阶段1` 的需求确认在你的主观判断里是否已经基本完成，还是仍有核心业务范围尚未定死。
- `阶段2` 的架构确认，你更想强调业务域拆分、数据库模型主线、还是迁移技术路线先定死。
- `每个阶段` 的人工审查，你是否希望进一步写死：谁来审、审什么、达到什么标准才允许进入下一阶段。
- `阶段6` 的后端开发完整，你更想先强调基础支撑模块完整，还是核心业务模块完整，还是两者必须同时作为完成标志。
- `阶段7` 的前后端联调适配，你希望适配范围覆盖全部旧页面，还是先覆盖主流程页面。
- `阶段9` 的“试运行”你更想强调真实业务试跑，还是更想强调切换前核账和问题收口。
- 后续这份阶段图是否还要顺手兼顾团队协作表达，目前仍未完全确认。
