# 草稿

关联需求: `docs/requirements/topics/rd-subwarehouse.md`
最后更新: 2026-03-30
状态: 探索中
文档说明: 记录用户对「RD 小仓前端应为双一级界面」的明确意图、两套一级界面的业务层功能抽象，以及与主题真源的对齐；不含路由、权限码、壳层实现细节。

## 用户意图

- 研发小仓在**展示与信息架构**上应是**不同的一级界面**：大仓管理员一个一级操作面，小仓管理员另一个一级操作面（并列、边界清晰，而非单一后台内仅多一个菜单组的主观感受）。
- **系统仓库管理员**应能**同时看到**上述两个一级界面（可进入、可核对），且业务上应理解为「双侧可见与管控」，**不是第三套独立业务流程**。
- 本轮只需要**抽象层 / 业务层**：每个一级界面解决什么问题、包含哪些业务能力；不需要技术实现方案。
- 已确认一级菜单命名：大仓侧叫 `研发协同`，小仓侧叫 `研发小仓`。
- 大仓管理员在 `研发协同` 中不应只看到“大仓保管阶段”的研发物料，还应能查看**研发小仓全部物料的状态与库存状态**，同时方便完成大仓视角下的验收、发放 / 交接、退回确认等操作。

## 已确认信息（2026-03-30）

- 一级菜单命名确认：
  - 大仓管理员一级菜单：`研发协同`
  - 小仓管理员一级菜单：`研发小仓`
- `研发协同` 的业务定位确认：它不是只看“还在大仓里的研发物料”，而是站在大仓管理员视角，查看**研发物料全链路状态**，并承接大仓侧必要操作。
- 研发退回流程确认：由小仓管理员选择退回物料并生成退料单，携货到大仓后，再由大仓管理员在系统中确认通过，完成真正退回；目标是用尽量少的操作完成交接闭环。

## 与主题真源对齐摘要

- `topics/rd-subwarehouse.md` 已约定：非镜像双视角、小仓独立工作台与本仓数据为主、系统管理员双侧全量可见（C4）；大仓以主仓运营面 + 必要 RD 协同为主且不进入 RD 日常台，小仓不看主仓运营面，系统管理员非第三套业务流（角色与展示口径）。
- 本草稿把上述口径**显式落为「两个一级业务操作面」**的命名与功能清单，便于前端信息架构与验收对齐；若与现网菜单层级不一致，以本工作流收敛后的用户确认为准，再反写主题或新开切片。

## 一级界面一：大仓管理员（一级菜单：研发协同）

**定位**：货物进入公司后在主仓的验收、保管、对外协同发放与公司级退回/退货；同时站在大仓责任视角，查看研发物料从采购、验收、交接、小仓在库、领用、报废到退回的全链路状态，并处理其中由大仓承担的动作。

**建议聚焦的业务能力（抽象）**


| 域          | 能力要点 |
| ----------- | -------- |
| 验收与入库      | 外部到货验收、入库事实；与 RD 采购信息的选取与核对（主仓视角） |
| 研发物料全景状态 | 查看研发物料全链路状态，不只限于主仓保管阶段；应能看到待采购、采购中、已验收、待交接、已交接、小仓在库、已领用、已报废、已退回等业务状态 |
| 小仓库存可见性   | 能查看研发小仓当前库存状态、数量变化、去向和异常，用于核对、追溯和后续大仓动作判断 |
| 发放 / 交接    | 向 RD 发放 / 交接、查看协同单据与交接结果；**不进入** RD 日常领用 / 盘点 / 报废执行台 |
| 退回处理      | 查看退料单、确认小仓实物已退回到大仓，完成真正退回 |
| 公司级流转      | 公司级退回、对外退货等仍走主仓链路的处理与查询 |
| 跨域核对       | 按职责查看与 RD 相关的协同记录、小仓状态与退回结果，用于对账与异常追溯 |

**页面设计方向（业务层）**

- `研发协同` 里至少应同时覆盖两类信息：
  - 一类是**看状态**：研发物料当前走到哪、哪些已经进小仓、哪些仍在小仓库存里、哪些已经领用 / 报废 / 退回。
  - 一类是**做动作**：大仓管理员能顺手完成验收、向小仓发放 / 交接、退回确认等属于自己职责的动作。
- 因此它不宜只是一张“主仓在库表”，而更像一个面向大仓管理员的**研发协同总览 + 操作面**。

**明确不承担**：RD 小仓日常项目领用执行、本仓盘点、本仓报废、本仓库存调整等「小仓内部日清日结」作业。

## 一级界面二：小仓管理员（一级菜单：研发小仓）

**定位**：固定本研发小仓范围内的「今天要管什么、缺什么、用了什么、剩多少、异常怎么处理」闭环。

**建议聚焦的业务能力（抽象）**


| 域      | 能力要点                            |
| ------ | ------------------------------- |
| 工作台    | 研发工作台：待办、关键状态、本仓视角入口聚合          |
| 本仓账实   | 本仓库存、本仓流水、分类/分布类本仓报表            |
| 主仓协同结果 | 主仓到本仓交接/自动入库结果查看（无需二次收货确认的业务语义） |
| 采购与跟踪  | 研发采购需求发起与在链路中的跟踪（与主仓验收联动）       |
| 出库与归集  | 项目领用/出库（强绑定项目或项目式归集项）           |
| 本仓内部动作 | 本仓盘点、本仓库存调整、本仓报废                |
| 物料业务状态 | 按主题已定义状态链理解与展示（与单据审核状态区分）       |


**明确不承担**：外部采购直入小仓、对外退货、退回主仓等公司级二次流转的执行面（仍由主仓侧或系统管理员按职责处理）。

## 系统仓库管理员（双侧可见）

- **不是**第三套并列的一级业务流程；**是**同时保留 **大仓一级界面** 与 **小仓一级界面** 的进入与全量可见能力，用于统一配置、排障、跨域核对与审计对齐。
- 验收口径建议：在权限允许前提下，能分别从两套一级界面完成各自职责下的查看与必要操作，且不因「视角切换」丢失另一侧入口或关键页面（与主题 C4「不受专属壳层裁切影响」一致，具体 UI 形态待确认）。

## 当前展示形式（现状）

- 当前前端**已经不是完全混在一起**，而是存在「普通后台主视角 + RD 专属入口 / 工作台」的双视角雏形。
- 对大仓管理员来说，当前更像是**以主仓后台为主界面**，在主仓业务组之外带有一些与 RD 协同有关的入口；其核心心智仍然是“我在一个统一后台里处理主仓业务，并顺带查看或处理部分 RD 协同事项”。
- 对小仓管理员来说，当前已经有相对独立的 `研发工作台` 和本仓相关页面，能围绕本仓库存、流水、自动入库结果、项目领用、本仓报废等内容开展日常工作。
- 对系统仓库管理员来说，当前口径是**主仓侧与 RD 侧都可见**，用于统一查看、配置、排障和跨域核对。
- 但从**信息架构和用户感受**来看，当前整体仍更接近「一个主后台 + 一个 RD 专属入口 / 菜单组 / 工作台」，而不是「大仓管理员一级界面」与「小仓管理员一级界面」两个**并列、同层级、心智上对等**的一级操作面。
- 也就是说，当前现状更偏“RD 已经被独立出来了一部分”，但还没有完全达到你要求的“两个一级界面一开始就并列成立”的展示效果。

## 已确认流程：研发退回

- 第一步：小仓管理员在 `研发小仓` 中选择要退回的物料与数量，生成退料单。
- 第二步：小仓管理员携带实物到大仓。
- 第三步：大仓管理员在 `研发协同` 中对该退料单进行确认通过，完成真正退回。
- 流程目标：小仓侧发起一次，大仓侧确认一次，不额外增加重复录单或多余审批动作。

## 与「当前前端感受」的差异假设（待用户确认）

- 若当前产品仍像「一个后台 + RD 子菜单」，与用户期望的「两个一级界面」在**心智模型与导航层级**上不一致，需在确认本草稿后，再定义一级导航命名、默认落地页与切换方式（仍属业务/信息架构，实现另议）。

## 待澄清清单（可升级为 requirement `待确认`）

- `研发协同` 这个一级界面内部，是否应进一步拆成“状态总览 / 验收入库 / 发放交接 / 退回处理 / 协同查询”等二级页，还是先做成一个总览工作台再分入口。
- 大仓管理员查看“小仓全部物料状态”时，默认看到的粒度应以**状态汇总**为主，还是直接支持下钻到**物料明细 / 项目归集 / 当前在库数量**。
- 系统仓库管理员默认落地：偏主仓、偏列表选择、或记忆上次界面。
