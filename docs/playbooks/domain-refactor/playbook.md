# Domain Refactor Playbook

本页记录跨 schema、跨后端、跨前端的领域重构经验，重点覆盖“字段退场、语义拆分、合同同步、分批闭环”这类高风险改造。

---

## DR-001 动 schema 后先 generate 再继续写代码

> 只要改了 `prisma/schema.prisma`，下一步就应该先刷新 generated client。否则后面看到的大量 repository / include / input type 报错，很多只是生成物过期。

**场景**: 在做“字段删除、relation 调整、nullable 变化、include 扩展”这类 schema 改造时，如果先继续改 service / repository / controller，而不是先 `prisma generate`，运行时和类型层会同时漂移，错误会被放大。
**结论**: `schema` 变更和运行时代码改造不能混在同一批里推进。generated client 是这类重构的第一层 contract，必须先更新。
**做法**: 一旦改了 `prisma/schema.prisma`，立即执行 `prisma generate`；先过一轮类型检查，确认 generated 层错误清零，再继续写 repository / service / controller。
**成熟度**: 初步观察

## DR-002 先列字段退场矩阵，再做全局替换

> 删除一个旧字段，不等于完成一次重构。必须先明确“谁下线、谁接管、哪里禁留兼容”。

**场景**: 当一个字段同时承载了主数据含义、权限范围、默认值查找、前端文案和业务分支判断时，直接全局替换很容易留下半新半旧状态，例如旧 helper 还在从旧字段反推新语义。
**结论**: 跨领域字段重构不能只靠搜索替换，必须先做字段退场矩阵。否则运行时、测试夹具和前端页面会分层残留旧语义。
**做法**: 开工前至少列 3 张表：
- 字段删改表：旧字段、替代字段、是否保留兼容、最终禁留点
- 入口清单：DTO、controller、service、repository、frontend API、页面、spec、stub、fixture
- 语义归属表：每个语义的唯一真源是谁，禁止谁继续代持
**成熟度**: 初步观察

## DR-003 合同变更必须按批次闭环，不能把测试留到最后

> DTO、实体返回结构、include shape 一旦变化，spec / stub / mock 就已经过期了。它们不是收尾项，而是 contract 的一部分。

**场景**: 运行时代码改完后再统一修测试，会让 `tsc` 和 review 同时被旧夹具噪音淹没，主代理很难判断当前剩下的是实现问题、类型问题还是夹具问题。
**结论**: 任何 contract 变更都必须同批更新测试夹具。否则最后看到的不是“剩余问题”，而是“全链路漂移”。
**做法**: 按下面的批次推进，每一批都必须闭环：
1. `schema`
2. `generate`
3. repository / service / controller
4. frontend API / 页面
5. spec / stub / mock / fixture
6. compile / verify
禁止把 `spec`、`stub`、`mock` 留到最后集中修。
**成熟度**: 初步观察

## DR-004 领域语义拆分后，前端必须做标签审计

> 数据流改对了，不代表用户界面改对了。尤其是“库存范围”和“业务车间”这类原本混用的概念，最容易在 label、placeholder 和默认选择上继续混淆。

**场景**: 当后端已经把“库存池”和“业务归属”拆成两个对象后，前端如果仍沿用旧命名、旧文案和旧默认搜索逻辑，用户会继续被旧模型误导。
**结论**: 领域拆分不仅是接口改造，也是页面语义改造。必须单独检查展示层到底在显示什么概念。
**做法**: 每次拆语义后，单独做一次前端标签审计：
- 哪些地方显示库存范围
- 哪些地方显示业务车间
- 哪些地方原来是假仓库，现应改成真实部门
- 哪些地方的默认值还在依赖旧字段或旧文案搜索
**成熟度**: 初步观察

## DR-005 不要靠最后一次 tsc 才发现问题

> 最后一次总编译只应该做收口，不应该承担“第一次发现系统性漂移”的职责。

**场景**: 当改造面同时跨 schema、后端、前端和测试时，如果长时间不编译，最终爆出来的问题会混成一团，排查成本远高于逐批修正。
**结论**: 编译应当是节拍器，不是垃圾回收器。
**做法**: 每做完一个可闭环批次就执行一次最小验证，至少包含一次类型检查；如果当前批次涉及 schema，验证顺序必须是 `schema -> generate -> typecheck`。
**成熟度**: 初步观察
