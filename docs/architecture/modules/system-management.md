# `system-management` 模块设计

## 模块目标与职责

负责平台治理主题下的系统管理能力，回答"谁能登录、归属哪个组织、拥有哪些平台权限，以及哪些平台配置由系统统一维护"。第一版范围收口 `用户 / 角色 / 部门 / 菜单 / 岗位 / 字典 / 参数 / 通知` 八类核心系统页，并把 `在线用户 / 登录日志 / 操作日志` 作为同主题下的治理与审计能力统一规划。

## 当前实现与目标范围

**当前实现**：

- `src/modules/rbac` 已通过 `SystemManagementService` 与 `system/`* 控制器承接八类核心系统页，前端系统管理菜单也已落到 `/system/`* 路由。
- `在线用户` 当前由 `session` 暴露管理接口；`登录日志 / 操作日志` 当前由 `audit-log` 承接，`sys_logininfor` 与 `sys_oper_log` 已有规范化 Prisma model 与独立 repository，是审计数据的落库真源。
- 当前样例数据已收敛到 `研发部 / 采购部 / 仓库` 三个一级部门，并提供 `admin / operator / rd-operator / procurement` 四类代表性账号来承接当前登录、权限、RD 视角与采购协同测试口径；其中 `operator` 仍保留兼容用户名，但其业务含义已对齐为 `仓库管理员`。
- 八类核心系统管理对象（用户/角色/部门/菜单/岗位/字典/参数/通知）已切换到规范化数据库表作为运行态真源；历史 `system_management_snapshot` 快照桥接方案已下线。

**目标范围**：

- 长期口径按已确认 `V1` 收敛：真实部门先按 `研发部 / 采购部 / 仓库` 三个一级部门建模，不展开二级部门。
- 当前主角色按 `系统管理员 / 仓库管理员 / 研发小仓管理员 / 采购人员` 四类设计；`老板 / 财务` 作为后续可增加的查看型角色预留。
- 架构上把 `system-management` 视为平台治理主题；运行态真源继续分别落在 `rbac`、`session`、`audit-log` 等模块，不在前后端再复制第二套权限 / 会话 / 审计真源。
- 系统管理主数据必须以规范化数据库表持久化，不依赖 JSON 快照或进程内内存状态作为长期真源。

## 原 Java 来源与映射范围

- `ruoyi-system` 中 `SysUser`*、`SysRole`*、`SysMenu*`、`SysDept*`、`SysPost*`、`SysConfig*`、`SysNotice*`
- `ruoyi-admin` 中 `SysUserOnlineController`
- `ruoyi-framework` / `ruoyi-system` 中登录日志、操作日志相关实现

映射规则：

- 旧平台 `sys_*` 能力只作为产品形态与接口语义参考，不作为新系统正式真源整表迁移目标；新系统应按自身方案重建规范化平台表。
- 用户、角色、部门、菜单与数据权限真源继续下沉到 `rbac`，以规范化数据库表落库；在线用户真源下沉到 `session`；登录日志、操作日志真源下沉到 `audit-log`，已有独立 Prisma model。
- `调度任务 / AI 支持` 虽然可能从系统菜单进入，但不纳入 `system-management` 第一版长期范围；当前作为非核心平台邻接能力顺延到 Phase 3 规划，现阶段暂不执行。

## 能力归属矩阵


| 能力                | 当前主要承接模块                     | `system-management` 中的职责 | 边界说明                                |
| ----------------- | ---------------------------- | ------------------------ | ----------------------------------- |
| 用户 / 角色 / 部门 / 菜单 | `rbac`                       | 提供系统管理管理面与长期治理口径         | 权限字符串、路由树、角色菜单关系、数据权限仍统一由 `rbac` 收口 |
| 岗位 / 字典 / 参数 / 通知 | `rbac`（当前共置）                 | 作为平台兼容与辅助配置对象统一维护        | 不是业务真源，不得回流承担库存、项目、流程等核心事实          |
| 在线用户              | `session`                    | 作为平台治理能力纳入系统管理主题         | 以 Redis 会话为真源，强退与在线查询不回写权限模型        |
| 登录日志 / 操作日志       | `audit-log`                  | 作为审计能力纳入系统管理主题           | `sys_logininfor` / `sys_oper_log` 是落库真源；日志异步写入，不得反向决定授权结果 |
| 调度任务 / AI 支持      | `scheduler` / `ai-assistant` | 当前不纳入第一版，Phase 3 再规划     | 只保留邻接关系，不并入长期主题边界，现阶段暂不执行           |


## V1 组织与角色基线

### 真实组织口径

- 一级部门固定为 `研发部 / 采购部 / 仓库`
- 当前阶段不做二级部门展开
- 部门树用于组织归属与数据权限表达，不直接代表库存仓别

### V1 角色矩阵


| 角色        | 主要目标         | 能看什么                                                  | 能做什么                                        | 不能做什么                                   |
| --------- | ------------ | ----------------------------------------------------- | ------------------------------------------- | --------------------------------------- |
| `系统管理员`   | 管系统、管账号、兜底排障 | 全部系统管理页、全部业务页、全量报表、在线用户、登录日志、操作日志                     | 管用户 / 角色 / 部门 / 菜单 / 字典 / 参数 / 通知；查看并排障全部业务 | 不建议作为日常业务主操作角色使用，但保留全量兜底能力              |
| `仓库管理员`   | 管主仓与公司级实物流转  | 主仓业务页、主仓库存 / 流水、入库 / 出库 / 领退料、RD 协同必要信息、相关报表          | 主仓验收、主仓入出库、车间领退料、主仓到 RD 协同、查看 RD 协同结果       | 不做系统管理；不做 RD 小仓内部盘点 / 报废 / 研发使用记录；不维护角色权限 |
| `研发小仓管理员` | 管 RD 小仓日常作业  | RD 工作台、本仓库存 / 流水、本仓报表、研发使用记录、本仓盘点 / 调整、本仓报废、采购需求及主仓协同结果 | 提采购需求、做本仓使用 / 退料 / 报废 / 盘点调整、查看主仓交接与验收结果    | 不看主仓运营面；不做主仓验收；不做系统管理；不处理公司级对外出入库       |
| `采购人员`    | 管采购执行链路      | 采购需求、供应商相关主数据、采购跟踪结果、与采购闭环有关的主仓验收结果                   | 接单采购、推进采购状态、维护采购过程信息                        | 不做主仓库存操作；不做 RD 小仓内部操作；不做系统管理；不改角色权限     |


补充约束：

- `老板 / 财务` 当前只作为后续查看型角色预留，不进入第一版必交付范围。
- 当前阶段优先用"角色"表达系统使用者差异，不额外引入岗位层增加治理复杂度。

## Controller 接口草案

- `GET /system/user/list`
- `GET /system/role/list`
- `GET /system/dept/list`
- `GET /system/menu/list`
- `GET /system/post/list`
- `GET /system/dict/type/list`
- `GET /system/config/list`
- `GET /system/notice/list`
- `GET /sessions/online`
- `GET /audit/login-logs`
- `GET /audit/oper-logs`

说明：

- 八类核心系统页当前可继续沿用 `/system/*` 路由分组。
- 在线用户与审计日志保留各自模块接口，由系统管理前端导航与权限模型统一收口。

## Application 层编排

- `SystemManagementService` 当前作为八类核心系统页的管理面编排入口，负责查询、写入、导出与表单聚合。
- 用户、角色、部门、菜单等授权相关变更完成后，需要联动 `session` 失效已有会话或刷新权限快照。
- 在线用户查询与强退委托 `session`；登录日志与操作日志查询委托 `audit-log`。
- 后续若拆出独立 `system-management` 代码模块，也只承接管理面编排，不复制 `rbac` / `session` / `audit-log` 的底层真源逻辑。

## Domain 规则与约束

- `system-management` 管的是平台治理对象，不是库存、车间、研发项目、销售项目或采购业务真源。
- 组织权限不等于库存范围；真实库存访问仍沿 `stockScope / workshopScope` 收口。
- `consoleMode` 只负责壳层和默认入口体验，不替代真实授权。
- `角色` 负责权限包与数据权限边界；`岗位` 当前只是兼容对象，不作为本轮真实组织建模必选项。
- `系统管理员` 保留全量系统管理与跨域排障兜底能力。
- `仓库管理员` 可以查看 RD 协同必要信息，但不进入 RD 小仓日常运营面。
- `研发小仓管理员` 固定 RD 视角，不承担主仓运营职责。
- `采购人员` 可以回看与采购闭环有关的主仓验收结果，但不承担库存操作。
- 旧平台 `sys_user / sys_role / sys_menu / sys_dept / sys_post / sys_config / sys_notice / sys_dict_*` 不作为新系统正式业务真源直接迁入，应按新系统方案重建为规范化平台表。

## Infrastructure 设计

- 用户、角色、部门、菜单与数据权限必须以规范化数据库表持久化，运行态由 `rbac` 服务与仓储层承接。
- 岗位、字典、参数、通知等平台对象同样以规范化 Prisma model 落库，支持标准 CRUD；菜单树、授权关系、数据权限联查优先 raw SQL。
- 登录日志、操作日志已有规范化 Prisma model（`LoginLog` / `OperLog`），由 `audit-log` 异步记录与查询，`system-management` 只消费其管理能力。
- 用户状态、密码、角色授权等变更需要调用 `SessionService` 失效受影响会话，避免旧权限快照长期滞留。
- 在线用户列表继续基于 Redis 会话扫描构建；强退以删除会话为准。
- 系统管理初始化应作为"新系统平台初始化"单独准备，不与旧业务数据迁移混为同一条导入链路。

## 与其他模块的依赖关系

- 依赖 `rbac`：用户、角色、部门、菜单、权限字符串、数据权限、路由树真源（规范化数据库表）
- 依赖 `session`：在线用户、强退、授权变更后的会话失效
- 依赖 `audit-log`：登录日志、操作日志与系统管理变更审计（规范化数据库表）
- 间接依赖 `auth`：密码策略与登录入口语义保持一致
- 与 `scheduler`、`ai-assistant` 保持邻接但不并入第一版范围，相关规划顺延到 Phase 3

## 事务边界与一致性要求

- 用户、角色、部门、菜单等授权相关写操作必须在各自单事务内提交，不允许出现一半成功一半失败的授权状态。
- 用户状态、密码、角色、数据权限变更后，需要在提交完成后触发受影响会话失效或刷新。
- 在线用户强退以会话删除成功为准；日志写入失败不能阻断主业务。
- 系统管理对象的初始化、变更与导入要和业务数据迁移解耦，避免把旧平台组织配置误当成正式业务主数据。

## 权限点、数据权限、审计要求

- 系统管理所有管理接口都需要明确 `Permissions`，不能依赖前端菜单可见性做唯一防线。
- 用户、角色、部门等查询可复用数据权限，但数据权限不能替代真实库存或 RD 固定范围限制。
- 用户、角色、菜单、部门、岗位、字典、参数、通知的新增、修改、删除、授权、导出都应记录操作审计。
- 在线用户查询 / 强退、登录日志 / 操作日志查询与清理都应配置独立系统权限点。

## 当前缺口 / 后续切片

- `老板 / 财务` 两类查看型角色尚未细化到报表与字段边界。
- 若后续需要"次级系统管理员"或更细颗粒度授权，应新开切片，不在本轮直接膨胀角色层次。

## 暂不实现范围

- `调度任务`
- `AI 支持`
- 用部门树直接替代真实库存范围
- 为了当前小团队场景引入独立岗位治理体系
