# 销售项目主题需求

## Metadata

- ID: `domain-sales-project-management`
- Status: `draft`
- Scope: `domain-level`
- 状态说明: 当前运行时已将历史 `RD` 内部项目实现独立为 `src/modules/rd-project` 与 `RdProject*` 逻辑模型，对应物理表为 `rd_project*`；销售项目仍未实现。本文档继续作为 `sales-project` 的长期真源，后续实现需以 `sales` 出库维度和可选项目分配 / 预留能力为基础推进。

## 主题定义

- `sales-project` / `salesProject` 专指对外销售的大型项目，不包含内部 `RD`（Research and Development）主题。
- 本主题解决“某个销售项目需要哪些物料、当前可供货情况如何、已经向客户真实发出了哪些原料 / 半成品 / 成品、还差多少未供、能否先分配再分批供货”的问题。
- 销售项目不是项目管理系统，不承担排期、甘特图、审批流、任务协同或工程进度管理。
- 销售项目不是新的物理库存池，也不替代 `sales` 出库单；真实库存增加继续来自 `inbound`，真实库存减少继续来自 `sales`。
- 本主题可选支持“项目分配 / 预留”语义，但项目分配 / 预留不等同真实出库；它只是把部分库存标记为优先供某个销售项目使用。
- 本文档只保留长期约束、长期业务口径、能力清单和能力合同；单次交付直接从本 domain 创建 `task-*.md`。

## 承接的项目级能力版图

### 销售项目（`sales-project`）

- 本 domain 承接项目总纲中 `sales-project` 模块的长期职责，作为销售项目主档、项目维度库存视图、项目维度发货统计和可选项目分配 / 预留语义的真源。
- 项目页面需要作为独立入口存在，供仓库管理员或业务人员维护销售项目主档、查看项目相关物料、查看当前库存 / 可供货 / 已分配 / 已发货 / 已退货情况，并在需要时快速发起与项目相关的销售出库草稿。
- 销售项目本身不直接写库存；项目页上的“一键出库”本质上应生成 `sales` 域的标准销售出库单草稿或正式单据，而不是让项目模块旁路改库存。
- 入库页可以提供“分配到项目”的快捷入口，但后台必须沉淀为独立的项目分配 / 预留记录，而不是直接视为销售出库。

### 相邻共享边界

- `sales` 继续承接销售出库 / 销售退货，并作为销售项目真实库存变动的唯一下游业务入口；项目相关统计默认复用 `sales` 单据事实。
- `inbound` 继续承接真实入库；项目可以读取入库结果、辅助判断可供货情况，但不拥有入库主单据。
- `inventory-core` 继续作为所有库存变动的唯一写入口；项目域只组织项目语义、项目视图和项目分配 / 预留规则，不旁路改库存。
- `rd-subwarehouse` 是独立内部研发主题，不与销售项目强绑定；`RD` 物料流转与小仓协同不应再借用销售项目语义。

## 长期约束

- `C1` 不是项目管理系统：销售项目域只承接主档、供货视图、项目维度统计和可选项目分配 / 预留，不引入排期、阶段、任务协同、里程碑或审批流。状态：`生效中`
- `C2` 销售项目与 `RD` 分离：销售项目不与 `RD` 小仓、研发采购、研发领退报废建立默认强关联。状态：`生效中`
- `C3` 项目不是物理库存池：销售项目只承担业务归属、查询与统计语义，不形成新的 `stockScope`。状态：`生效中`
- `C4` 真实库存变化只走标准单据：库存增加只由 `inbound` 等真实入库单据产生，库存减少只由 `sales` 出库单产生；销售项目本身不直接过账库存。状态：`生效中`
- `C5` 项目维度真实发货口径：凡属于销售项目的真实供货，必须在销售出库单行上绑定 `salesProjectId` 或等价销售项目快照；不能只在报表层事后猜测归属。状态：`生效中`
- `C6` 分配 / 预留不等于出库：项目分配 / 预留若启用，必须形成独立记录，可释放、可转出库、可追溯，但不得直接减少 `quantityOnHand`。状态：`生效中`
- `C7` 项目可覆盖多类物料：销售项目允许统计流向客户的原料、半成品、成品，不把销售出库限制为成品。状态：`生效中`
- `C8` 当前 RD 运行时不反向定义需求：现有 `rd-project` 运行时及其 `rd_project*` 物理表不能继续作为销售项目需求口径的依据。状态：`生效中`

## 长期业务口径

### 项目主档口径

- 销售项目主档第一阶段至少需要支撑：建档、查询、修改、删除 / 作废、项目相关物料查看、项目发货统计查看。
- 项目主档字段优先围绕供货与统计需求设计，例如：`salesProjectCode`、`salesProjectName`、关联客户、项目负责人、业务日期、备注，以及可选合同 / 订单参考信息。
- 未发生任何下游引用的项目允许轻量删除；一旦已经关联销售出库、销售退货、项目分配 / 预留或其他下游事实，则只能保留历史，不做静默硬删。
- 项目主档不是库存单据，也不是销售订单主单据；它更像“供货与统计上下文”。

### 项目库存视图与供货口径

- 销售项目没有独立物理库存池；项目页看到的“项目库存”是衍生视图，不是新的库存真源。
- 项目库存视图至少需要回答：当前相关物料在系统里还有多少、其中多少已分配给本项目、多少已真实发货、多少已退货、多少仍待供货。
- 第一阶段项目库存视图应优先基于真实库存余额、项目分配 / 预留记录、销售出库 / 退货记录派生，不另建第二套平行库存账。
- 项目是否需要严格“锁货”，应由项目分配 / 预留能力显式表达，而不是把入库单直接写成项目专属所有权。

### 项目分配 / 预留口径

- 项目分配 / 预留是独立业务事实，表示“这部分库存优先留给该项目”，但货仍在仓里，尚未真正发给客户。
- 分配 / 预留记录至少应包含：项目、物料、数量、仓别、来源单据或来源范围、状态、创建人、释放人 / 转出库人等基础信息。
- 分配 / 预留创建后，不应直接减少 `quantityOnHand`；若后续实现“可用量”口径，则只能影响 `available` 或项目供货可见范围，不能伪造成真实出库。
- 已分配 / 预留的库存应支持释放、改分配、部分转出库；转成真实供货时，必须生成 `sales` 出库单。
- 入库页可以提供快捷按钮帮助用户从新到货中直接生成项目分配 / 预留记录，但后台仍需沉淀为独立记录，而不是把入库行直接写成“已出库”。

### 项目出库与统计口径

- 销售项目的真实供货统一通过 `sales` 出库单完成；项目页上的“一键全部出库”应理解为“按当前项目上下文生成销售出库单草稿并带出物料”，而不是直接扣减库存。
- 项目真实统计以销售出库 / 销售退货为准，至少应支持：累计出库数量、累计退货数量、净发货数量、出库金额、退货金额、净发货金额、出库成本、净发货成本。
- 项目统计的核心关注点是“流向客户的物料”，允许覆盖原料、半成品、成品；不强制要求先经过成品化再出库。
- 若一个项目分多批供货，项目页面必须能同时看到“已发货部分”和“待发货部分”，而不是只有项目总量。

### 查询与角色口径

- 仓库管理员需要把销售项目列表和项目详情作为日常入口，用于查看项目主档、项目相关库存视图、项目分配 / 预留和项目发货统计。
- 业务或销售协同人员若后续接入，可主要消费项目供货进度、待发货和已发货汇总，不直接改库存。
- `sales` 域用户 / 页面在录入出库时需要能选择项目，保证真实发货数据从单据入口就带上项目维度。
- `RD` 小仓管理员不应因为系统里存在销售项目而默认进入项目流程；两类主题的入口、口径和统计应保持分离。

## 能力合同（推荐）

### `F1` 销售项目主档轻量 CRUD

- In scope:
  - 维护项目编码、项目名称、关联客户、负责人、业务日期、备注等轻量主档字段。
  - 提供项目新建、查询、修改、删除 / 作废的基础能力。
  - 形成稳定的销售项目引用，供 `sales` 出库单和项目统计使用。
- Out of scope / non-goals:
  - 不引入 `立项 / 进行中 / 暂停 / 关闭` 这类项目管理状态机。
  - 不引入甘特图、任务协同、审批流和多层项目结构。
  - 不把项目主档扩展成库存单据或销售出库主单据。
- Completion criteria:
  - `[TC-1]` 文档明确销售项目主档只提供业务上下文，不直接改库存。
  - `[TC-2]` 文档明确 `salesProjectCode` 在系统内唯一，且项目名称 / 客户 / 负责人等字段用于后续出库和统计引用。
  - `[TC-3]` 文档明确已被销售出库、销售退货或项目分配 / 预留引用的项目不能静默硬删。
- Evidence expectation:
  - domain 文档真源 + 项目主档 API / 数据模型 / 权限收口的交付证据。
- Default derived slice acceptance mode: `full`

### `F2` 项目维度库存 / 可供货视图

- In scope:
  - 项目详情按物料查看当前库存、已分配 / 预留、已发货、已退货、净发货、待供货等核心视图。
  - 支持按项目上下文查看原料、半成品、成品三类物料的供货状态。
  - 视图优先基于真实库存余额、项目分配 / 预留记录、销售出库 / 销售退货事实派生。
- Out of scope / non-goals:
  - 不引入第二套“项目库存余额主表”与 `inventory-core` 并行记账。
  - 不把项目视图误做成真实库存真源。
- Completion criteria:
  - `[TC-1]` 文档明确项目库存视图是衍生查询，不是新的库存池。
  - `[TC-2]` 文档明确项目页至少能区分当前库存、已分配 / 预留、已发货、已退货和待供货。
  - `[TC-3]` 文档明确原料、半成品、成品都可进入项目供货视图，而不只限成品。
- Evidence expectation:
  - domain 文档真源 + 项目详情查询接口 / 报表原型或交付证据。
- Default derived slice acceptance mode: `light`

### `F3` 项目关联销售出库与一键生成草稿

- In scope:
  - `sales` 出库单行支持绑定项目。
  - 项目页可按当前项目上下文选择物料并生成销售出库草稿，减少重复录入。
  - 若项目已有分配 / 预留记录，可优先带出已分配物料和数量。
- Out of scope / non-goals:
  - 不允许项目页面直接扣减库存。
  - 不绕过 `sales` 家族新增一套项目出库单。
- Completion criteria:
  - `[TC-1]` 文档明确销售项目的真实库存减少只由销售出库单产生。
  - `[TC-2]` 文档明确项目页上的“一键出库”后台必须沉淀为 `sales` 出库草稿或正式单据，而不是直接改库存。
  - `[TC-3]` 文档明确出库单行上的 `salesProjectId` 与项目快照是项目维度真实统计的基础事实。
- Evidence expectation:
  - domain 文档真源 + `sales` / 项目集成 task 的接口与页面验证。
- Default derived slice acceptance mode: `full`

### `F4` 项目维度发货 / 退货 / 净发货统计

- In scope:
  - 提供按项目、按物料的发货统计视图，至少覆盖出库数量、退货数量、净发货数量、金额与成本。
  - 统计默认来自 `sales` 出库 / 退货事实，不旁路自己维护平行账。
  - 支持按原料、半成品、成品查看项目供货结果。
- Out of scope / non-goals:
  - 不把项目统计扩展为总账、月结或标准成本系统。
  - 不要求本能力单独定义全部导出模板和财务报表格式。
- Completion criteria:
  - `[TC-1]` 文档明确项目统计聚焦“流向客户的物料”，而不是 `RD` 领退报废。
  - `[TC-2]` 文档明确项目统计至少区分出库、退货和净发货三类结果。
  - `[TC-3]` 文档明确项目金额 / 成本汇总来自真实客户单据及其来源成本结果，不能靠静态估算替代。
- Evidence expectation:
  - domain 文档真源 + 项目统计页 / 月报联动验证。
- Default derived slice acceptance mode: `full`

### `F5` 项目分配 / 预留

- In scope:
  - 建立独立的项目分配 / 预留记录，表达库存优先供某项目使用。
  - 支持入库页或项目页快捷创建分配 / 预留。
  - 支持释放、改分配、部分转出库，并保留追溯关系。
- Out of scope / non-goals:
  - 不把项目分配 / 预留直接记为真实出库。
  - 不允许项目分配 / 预留直接改写 `quantityOnHand`。
  - 不要求 V1 就完成复杂 ATP、锁货优先级冲突引擎或多项目智能抢占。
- Completion criteria:
  - `[TC-1]` 文档明确项目分配 / 预留是独立记录，不等同真实出库。
  - `[TC-2]` 文档明确分配 / 预留创建后，库存在手数量不变。
  - `[TC-3]` 文档明确分配 / 预留可释放、可改分配、可部分转销售出库，并保留追溯证据。
  - `[TC-4]` 文档明确入库页“分配到项目”只是快捷入口，后台仍需生成独立的项目分配 / 预留记录。
- Evidence expectation:
  - domain 文档真源 + 后续项目分配 / 预留 task 的数据模型与交互验证。
- Default derived slice acceptance mode: `full`

## 已确认补充口径（2026-04-09）

- 销售项目与 `RD` 分开，二者不再共享同一套业务定义。
- 销售项目统计的对象是流向客户的物料，允许覆盖原料、半成品、成品。
- 销售项目的真实库存变动统一走 `sales` 出库 / 退货，不再定义项目自有库存过账动作。
- 入库页若提供“分配到项目”的快捷操作，后台也必须生成独立项目分配 / 预留记录，而不是直接视为销售出库。

## 待确认

- 项目分配 / 预留第一版是否需要直接影响“可用量”展示，还是先只做项目视图与转出库草稿辅助。
- 一个销售出库单行是否允许同时绑定项目和其他业务维度，例如销售订单号或合同号。
