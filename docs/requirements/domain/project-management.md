# 项目 / 研发主题需求

## Metadata

- ID: `domain-project-management`
- Status: `draft`
- Scope: `domain-level`
- 状态说明: 当前代码已落地项目主档、`allocation_target` 一对一映射、项目物料动作基础闭环与项目台账衍生读模型；`F2/F4/F5` 仍缺少独立 task 与验收签收，本文继续作为长期约束与能力合同真源。

## 主题定义

- `project` 是项目归集与研发成品成本主题，解决“这个项目需要哪些物料、现在缺什么、实际领退报废了什么、累计消耗成本是多少、项目成品卖出时成本怎么算”的问题。
- 本主题核心是 `项目主档 + 轻量 BOM + 项目物料台账 + 项目成本台账 + customer 销售成本引用`，不是项目管理系统，不承担排期、进度、阶段流转或项目状态管理。
- 项目不是新的物理仓库，也不替代 `RD 小仓` 的仓别协同职责；它是仓库业务中的归集、预警和成本口径真源。
- 当前最简落地版本采用统一 `allocation_target` 主表承载归集对象；第一阶段只正式落地 `RD_PROJECT`，由 `project` 主档一对一映射，不引入 `label` 自由标签。
- 本文档只保留长期约束、长期业务口径、能力清单和能力合同；单次交付直接从本 domain 创建 `task-*.md`。

## 承接的项目级能力版图

### 项目 / 研发（`project`）

- 本 domain 承接项目总纲中 `project` 模块的长期职责，作为项目主档、项目 `BOM`、项目物料归集、项目成本和项目成品成本引用的真源。
- 项目页面需要作为独立入口存在，供仓库管理员维护项目主档、维护项目 `BOM`、查看缺料预警、查看项目物料台账与成本汇总。
- 项目相关的真实库存动作聚焦于 `项目领料`、`项目退料`、`项目报废`；这些动作必须继续通过 `inventory-core` 落账，而不是在项目表里直接维护库存。
- 项目对采购的职责是提出物料需求、展示缺口和补货状态，方便及时采购补充；实际采购、验收、主仓入库和主仓到 `RD_SUB` 的交接，仍由 `inbound` / `rd-subwarehouse` 承接。
- 当 `customer` 域卖出研发项目的成品或样品时，需要能回指项目定义的物料构成与成本快照，用于核算该项目产品的销售成本。

### 相邻共享边界

- `inventory-core` 继续作为所有项目库存动作的唯一写入口；本 domain 负责项目语义、需求口径、成本口径和查询，不旁路改库存。
- `rd-subwarehouse` 负责 `RD_SUB` 的仓别协同、采购需求状态链、主仓到小仓交接和小仓日常作业；本 domain 只消费这些结果做项目归集与成本统计。
- `inbound` 继续承接实物先入主仓 `MAIN` 的规则；项目域只关心这些到货 / 验收结果是否满足项目物料需求。
- `customer` 继续承接客户出库 / 销售退货；本 domain 负责向 `customer` 提供“这个项目成品用了哪些料、成本快照是多少”的引用能力。

## 长期约束

- `C1` 不是项目管理系统：项目域不引入 `立项 / 进行中 / 暂停 / 关闭` 这类项目状态管理，只保留轻量建档、修改、删除 / 作废和历史追溯语义。状态：`生效中`
- `C2` 项目不等于物理库存池：项目是归集与核算维度，不是新的真实仓别；真实库存范围仍以 `MAIN` / `RD_SUB` 为准。状态：`生效中`
- `C3` 库存写入口统一：项目相关的任何领料、退料、报废都必须通过 `inventory-core` 产生库存事实，不能由项目模块旁路改库存。状态：`生效中`
- `C4` 项目实际作业仓别固定为 `RD_SUB`：当前阶段所有研发项目的实际领料、退料、报废与缺料计算都以 `RD_SUB` 为准；`MAIN` 只承担验收与向小仓交接，不直接承接项目实际使用。状态：`生效中`
- `C5` 轻量 BOM 原则：`BOM` 只承担需求计划、采购参考、缺口预警和成本基线作用，不承担版本控制、替代料引擎、工艺路线或强制发料语义。状态：`生效中`
- `C6` 项目成本以净耗用为准：项目成本至少要能清楚体现 `领料消耗 - 退料回补 + 报废损耗` 的净耗用数量与金额，不能只看静态 `BOM` 估算。状态：`生效中`
- `C7` customer 销售必须冻结项目成本引用：当客户出库的是研发项目成品 / 样品时，销售单据必须保留项目物料构成与成本快照，不能只记项目名称而丢失成本依据。状态：`生效中`
- `C8` 采购补货职责分离：项目域负责提出缺口和补货建议，不直接拥有外部采购入库主单据；采购实物动作仍由相邻业务域承接。状态：`生效中`

## 长期业务口径

### 项目主档与 BOM 口径

- 项目主档第一阶段至少需要支撑：建档、查询、修改、删除 / 作废、项目 `BOM` 维护、项目物料汇总、项目成本查看。
- 项目主档字段优先围绕仓库业务需求设计，例如：项目编码、项目名称、关联客户、关联研发成品 / 样品、备注；项目实际作业仓别当前阶段固定为 `RD_SUB`，不作为用户可配项目状态项扩展成完整项目管理档案。
- 未发生任何下游引用的项目允许轻量删除；一旦已经有关联 `BOM`、库存流水、采购需求或客户销售引用，则只能保留历史事实，不做静默硬删。
- 项目 `BOM` 是项目需要哪些物料的计划清单；至少应记录 `物料`、`计划用量`、`单位`、`参考单价 / 参考金额`、`备注`。
- 一个项目可以维护多个项目成品 / 样品记录；每个成品 / 样品都应能关联自己的用料清单或引用项目 `BOM` 快照。

### 项目库存、缺料与补货口径

- 项目的真实库存动作以 `领料`、`退料`、`报废` 为核心，当前阶段统一在 `RD_SUB` 落账。
- 项目缺料补货链路应体现为“项目缺口 -> RD 采购需求 / 主仓协同 -> 主仓验收入 `MAIN` -> 主仓到小仓交接 -> `RD_SUB` 项目可用 -> 小仓分发给研发人员使用”。
- `MAIN` 只负责公司级收货、验收和向 `RD_SUB` 交接，不直接作为研发项目实际使用仓别。
- 项目缺料预警是衍生口径，不是新的库存真源；预警至少基于 `BOM 计划量`、`当前可用库存`、`已领用净耗`、`采购中 / 在途数量` 进行判断。
- 项目页面应能按物料展示 `计划量`、`当前可用`、`已领`、`已退`、`已报废`、`净耗用`、`缺口量`、`补货状态`。

### 项目成本与 customer 联动口径

- 项目成本至少需要按项目查看累计计划成本、累计领料成本、累计退料回补、累计报废损耗和当前净耗用成本。
- 项目成品 / 样品记录需要支持生成或冻结一份物料构成与成本快照，回答“这个卖出的项目产品用了哪些料、每种料成本多少、总成本多少”。
- `customer` 域卖出项目成品 / 样品时，出库单行应能引用该项目成品 / 样品记录，并同步保存当时的项目成本快照。
- 第一阶段优先满足销售成本核算与追溯，不把项目域扩展为完整生产制造或标准成本系统。

### 项目单据与逆操作口径

- 项目主档与 `BOM` 属于计划与归集真源；创建、修改项目主档时可以更新项目范围、快照和 `BOM`，但不能因此直接改真实库存。
- 项目真实事务单据收敛为 `项目领料 / 项目退料 / 项目报废` 三类物料动作家族；动作语义通过 `actionType` 区分，而不是让项目主档直接兼做库存单据。
- 项目领料、项目报废属于 `RD_SUB` 的消耗动作；项目退料属于 `RD_SUB` 回补动作，并且默认必须回指同项目内有效的上游领料明细，不能伪造回冲关系。
- 项目改单 / 作废需要同时处理库存副作用、来源分配释放 / 恢复、下游依赖校验和项目台账重算，不能只改主表字段或列表展示状态。
- 项目主档允许轻量删除 / 作废的前提是没有有效项目物料动作、采购补货记录或其他下游依赖；一旦存在这些事实，就只能保留历史并通过逆操作纠偏。

### 项目查询与角色口径

- 仓库管理员、研发小仓管理员需要把项目列表和项目详情作为日常入口，用于查看项目主档、`BOM`、缺口、物料台账、成本汇总和项目物料动作历史。
- 采购角色主要消费项目缺口、补货状态和相关项目上下文，以驱动 `RD` 采购需求；采购动作本身仍在 `rd-subwarehouse` / `inbound` 完成，不回写项目库存事实。
- 系统管理员需要保留跨域查看项目、`RD` 补货、库存来源和客户销售引用的能力，用于排障和全链追溯。
- `customer` 域用户只消费冻结后的项目成品 / 样品成本快照，不直接修改项目台账、项目 `BOM` 或项目物料动作。

## 能力合同（推荐）

### `F1` 项目主档轻量 CRUD 与 RD 小仓归属

- In scope:
  - 维护项目编码、项目名称、业务日期、关联客户、关联供应商、关联负责人、关联车间、备注等轻量主档字段。
  - 项目创建时自动建立或绑定一条 `allocation_target(targetType = RD_PROJECT)`，作为后续库存归集与成本追溯的统一对象。
  - 项目实际库存范围固定映射到 `RD_SUB`，并沿角色 / 数据权限 / 查询接口持续生效。
  - 维护轻量 `BOM` 与项目快照字段，但项目主档本身不直接过账库存。
- Out of scope / non-goals:
  - 不引入 `立项 / 进行中 / 暂停 / 关闭` 这类项目生命周期状态机。
  - 不引入自由 `label`、多层项目结构、甘特图、任务分派或项目协同工作流。
  - 不把项目主档扩展成采购单、验收单或主仓交接单的写入口。
- Completion criteria:
  - `[TC-1]` 文档明确项目主档创建 / 修改只维护项目与 `BOM` 真源，不直接改真实库存。
  - `[TC-2]` 文档明确项目真实作业仓别固定为 `RD_SUB`，且该口径不是用户可自由切换的配置项。
  - `[TC-3]` 文档明确一个有效项目编码对应一个 `RD_PROJECT` 归集对象，并要求项目编码在系统内唯一。
  - `[TC-4]` 文档明确项目编码、项目作废在存在项目物料动作、采购补货或其他下游依赖时受到限制，不能静默改写历史链路。
- Evidence expectation:
  - domain 文档真源 + 项目主档 API / 数据模型 / 权限收口的交付证据。
- Default derived slice acceptance mode: `full`

### `F2` 项目 BOM、缺料预警与采购辅助

- In scope:
  - 维护轻量 `BOM` 行：`物料`、`计划用量`、`单位快照`、`参考单价`、`参考金额`、`备注`。
  - 基于项目 `BOM`、当前 `RD_SUB` 可用库存、项目已领 / 已退 / 已报废、`RD` 采购状态汇总，派生项目物料台账、剩余需求量、缺口量和补货状态。
  - 补货状态至少收敛为 `充足 / 补货中 / 待补货` 三类可读口径，服务采购辅助和项目跟进。
  - 项目详情页按物料展示计划、可用、领退报废、净耗用、缺口和补货状态。
- Out of scope / non-goals:
  - 不引入制造级多版本 `BOM`、替代料策略、工艺路线、发料齐套校验或自动发料。
  - 不在本 domain 内自动生成外部采购单、验收单或主仓交接单。
  - 不把缺料预警升级为独立工单流或审批流。
- Completion criteria:
  - `[TC-1]` 文档明确项目缺口至少由 `BOM 计划量`、`当前可用库存`、`净耗用` 与 `采购中 / 在途数量` 共同判断，而不是只看静态 `BOM`。
  - `[TC-2]` 文档明确补货状态是项目域的只读衍生结果，采购事实仍由 `rd-subwarehouse` / `inbound` 提供。
  - `[TC-3]` 文档明确轻量 `BOM` 的作用是计划、预警、采购辅助与成本基线，而不是强制库存发料引擎。
  - `[TC-4]` 文档明确项目 `BOM` 可以修改，但修改本身不会直接写库存；真实库存事实仍以后续项目物料动作落账。
- Evidence expectation:
  - domain 文档真源 + 项目详情缺口视图 / 项目物料台账查询 / 补货联动验证。
- Default derived slice acceptance mode: `light`

### `F3` 项目领料 / 退料 / 报废与库存联动

- In scope:
  - 收敛 `项目领料`、`项目退料`、`项目报废` 三类项目物料动作家族，作为项目真实库存事务入口。
  - 所有项目物料动作统一通过 `inventory-core` 在 `RD_SUB` 过账，并携带项目 `allocationTargetId` 以保留项目归集语义。
  - 领料 / 报废需要保留来源分配与成本结果；退料需要释放上游领料来源占用并按释放结果回补库存与成本。
  - 作废时原子处理下游依赖、来源释放 / 恢复和库存冲回，保持项目台账与库存事实一致。
- Out of scope / non-goals:
  - 不允许项目页面旁路 `inventory-core` 直接改库存底表。
  - 不允许把主仓 `MAIN` 直接视为项目实际消耗仓别。
  - 不把轻量审核升级为过账前阻断审批流。
- Completion criteria:
  - `[TC-1]` 文档明确项目领料 / 报废属于 `RD_SUB` 消耗，项目退料属于 `RD_SUB` 回补，三类动作都不是项目主档字段更新的附属副作用。
  - `[TC-2]` 文档明确项目退料必须回指同项目内有效的上游领料明细，且累计退料数量不得超过上游领料数量。
  - `[TC-3]` 文档明确领料 / 报废需要保留来源分配与成本快照，退料需要按释放的来源成本回补，而不是按物料静态单价回补。
  - `[TC-4]` 文档明确作废项目物料动作时，需要同步处理未作废下游依赖、来源占用释放 / 恢复和库存逆操作，不能只把单据状态改成作废。
- Evidence expectation:
  - domain 文档真源 + `project_material_action` / `inventory_log` / `inventory_source_usage` 对应验证。
- Default derived slice acceptance mode: `full`

### `F4` 项目净耗用与成本台账

- In scope:
  - 提供按项目、按物料的台账与汇总视图，至少覆盖 `计划量 / 计划金额 / 已领 / 已退 / 已报废 / 净耗用 / 缺口 / 当前可用 / 补货状态`。
  - 成本口径至少覆盖 `累计领料成本`、`累计退料回补`、`累计报废损耗` 和 `当前净耗用成本`。
  - 台账是衍生读模型，可从 `BOM`、历史项目消耗行、项目物料动作、库存余额和采购状态汇总重建，不另建一套平行库存账。
  - 后续导出、月报和跨域查询可以复用同一项目净耗用口径。
- Out of scope / non-goals:
  - 不把项目台账扩展成标准成本、在制品核算、月结关账或总账凭证系统。
  - 不引入第二套“项目库存余额主表”与 `inventory-core` 并行记账。
  - 不在本能力中强制定义完整导出模板或月报呈现格式。
- Completion criteria:
  - `[TC-1]` 文档明确项目净耗用默认按 `领料 - 退料 + 报废` 计算，成本也按同一净额口径汇总。
  - `[TC-2]` 文档明确项目台账既要显示 `BOM` 计划行，也要显示实际发生但未在 `BOM` 中的物料历史，避免真实耗用被隐藏。
  - `[TC-3]` 文档明确项目汇总页与物料明细页都能追溯回具体项目物料动作与补货状态，不是只给静态汇总数。
  - `[TC-4]` 文档明确 `reporting` / `monthly-reporting` 若复用项目净耗用口径，必须沿用本 domain 的统一公式与台账字段，不得各自定义另一套项目成本算法。
- Evidence expectation:
  - domain 文档真源 + 项目详情台账 / 汇总接口 / 后续报表联动验证。
- Default derived slice acceptance mode: `full`

### `F5` customer 销售引用项目成品成本快照

- In scope:
  - 一个项目可维护多个项目成品 / 样品记录；每条记录都能生成或冻结对应的物料构成与成本快照。
  - `customer` 域出库单行引用项目成品 / 样品时，需要同时保存项目、成品 / 样品和冻结时点的成本快照。
  - 成本快照至少能解释 `来源项目`、`成品 / 样品标识`、`冻结时间`、`物料清单`、`数量`、`单位成本`、`行成本` 和 `总成本`。
  - 快照冻结后保留历史语义，后续即使项目 `BOM` 或项目台账变化，也不回写已销售单据上的历史快照。
- Out of scope / non-goals:
  - 不把项目域扩展成完整制造配方、工艺路线、工单投料或标准成本引擎。
  - 不允许销售后通过修改项目 `BOM` 或项目台账静默改写历史销售成本证据。
  - 不在本能力中承诺一套完整售后返修 / 二次拆解的项目成本追溯模型。
- Completion criteria:
  - `[TC-1]` 文档明确 `customer` 出库引用的是冻结后的项目成品 / 样品成本快照，而不是运行时临时重新计算的动态结果。
  - `[TC-2]` 文档明确快照保存的是“这个卖出的项目产品用了哪些料、每种料成本多少、总成本多少”的可追溯证据。
  - `[TC-3]` 文档明确一个项目存在多个成品 / 样品时，销售引用必须精确到某个项目成品 / 样品记录或明确的项目级快照，不能只写项目名称。
  - `[TC-4]` 文档明确项目域向 `customer` 提供的是成本引用能力，不接管客户出库主单据与销售退货主流程。
- Evidence expectation:
  - domain 文档真源 + 后续 `customer` 集成 task 的快照冻结、引用和追溯验证。
- Default derived slice acceptance mode: `full`

## 已确认补充口径（2026-04-09）

- 项目域当前最小稳定归集模型继续以 `allocation_target` 为真源，正式落地对象仅有 `RD_PROJECT`；不额外恢复自由 `label` 口径。
- 项目真实作业仓别固定为 `RD_SUB`，项目列表、详情、主档写入和项目物料动作都应沿此固定仓别口径收口。
- 项目退料最小稳定合同是“必须回指同项目内有效的上游领料明细，并按释放来源成本回补”，当前不开放无源退料。
- 项目编码一旦已被采购补货链或项目物料动作引用，就不再允许随意改写；项目作废也必须在无项目物料动作、无采购补货、无下游依赖时才允许通过。
