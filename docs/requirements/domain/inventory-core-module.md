# 库存核心主题需求

## Metadata

- ID: `domain-inventory-core-module`
- Status: `confirmed`
- Scope: `domain-level`

## 主题定义

- `inventory-core` 是所有事务型单据统一依赖的库存中心域，解决“现在还有多少库存、这笔库存为什么变化、来自哪一层来源、如何逆操作恢复、如何传递真实成本”的问题。
- 本主题覆盖库存现值、库存流水、来源分配 / 释放、逆操作、价格层、预警和编号区间等共享能力；任何业务模块都不能绕过它直接改库存底表。
- `inventory-core` 是后端共享核心域，而不是让最终用户直接“手改库存”的独立业务模块；用户看到的库存结果会分散呈现在入库、销售业务、车间物料、销售项目视图、RD 协同、盘点 / 调整、报表等场景中。

## 承接的项目级能力版图

### 库存核心（`inventory-core`）

- 本 domain 承接项目总纲中的 `inventory-core` 模块详细职责，作为库存事实、来源追溯与成本层传递的长期真源。
- 它继续作为所有库存变动的唯一写入口，默认承接库存增加、减少、逆操作、来源追踪、预警和编号区间等核心能力。
- 第一阶段真实库存范围固定为 `MAIN` 与 `RD_SUB`；`workshop`、`rd-project` 与 `sales-project` 仅承担归属、核算和视图语义，不进入库存余额唯一键。
- 当前优先收敛库存事实、来源追溯、逆操作与成本口径，不把主题扩展为完整的多仓 / 库位 / 批次升级工程。
- `F5` 第一版先按最小闭环收口：预警只保留只读视图 / 报表提示，编号区间只服务 `sales` 出库链路，共享查询先满足当前已落地业务消费。

## 长期约束

- `C1` 唯一库存写入口：所有业务模块都必须通过 `inventory-core` 写入库存事实，不能直接更新 `inventory_balance`、`inventory_log` 或 `inventory_source_usage`。状态：`生效中`
- `C2` 可信来源顺序：若要提供完全可信的 `FIFO` 成本追溯，系统不允许负库存，不允许先出后补来源后仍宣称成本完全可信。状态：`生效中`
- `C3` 真实库存范围固定：第一阶段库存唯一维度固定为 `materialId + stockScopeId`，真实库存范围仅包含 `MAIN` / `RD_SUB`。状态：`生效中`
- `C4` 来源追溯不可省略：出库、领料、退料、报废、调价、RD handoff 等链路一旦影响来源与成本，都必须保留来源分配或桥接记录。状态：`生效中`
- `C5` 逆操作必须幂等：`reverseStock()` 必须保证同一条原始流水最多只被逆操作一次，并要求调用方提供稳定的 `idempotencyKey`。状态：`生效中`
- `C6` 成本以来源层为准：同一物料不同入库批次可以有不同单价，真实成本必须按来源层传递，不能退化为物料主档静态单价。状态：`生效中`
- `C7` 共享核心不接业务审核：`inventory-core` 与 `approval` 通过上游业务模块协作，不直接承载审核投影或业务主状态语义。状态：`生效中`
- `C8` 价格层能力继续收口：价格层可用库存、按价格层分配来源以及入库调价后的来源切换，必须继续复用同一套库存核心真源，不另建平行库存账。状态：`生效中`

## 长期业务口径

### 库存事实口径

- `inventory_balance` 表示当前库存现值，`inventory_log` 表示每一次库存变化事实，`inventory_source_usage` 表示消耗与回补时的来源分配 / 释放关系。
- 业务模块写库存时，必须同时传入业务单据类型、单据 ID、行 ID、操作人和幂等键，保证库存事实可追溯、可审计、可重放。
- 写操作原则上不直接暴露给前端页面，前端若需要库存写结果，应通过各业务单据模块间接触发。

### 来源与成本口径

- 默认按 `FIFO`（先进先出）分配来源；若用户在有业务依据时手动指定来源，则库存扣减、成本计算、逆操作和追溯都以指定结果为准。
- `sales` 出库（含销售项目维度）、车间领料属于最终消耗链；`RD handoff` 至少承担来源与成本桥接职责，保证主仓到 RD 小仓不断链。
- 价格层查询口径按 `物料 + unitCost` 聚合当前可用来源，不单独维护“价格库存余额主表”。
- 入库调价对剩余数量采用 `PRICE_CORRECTION_OUT + PRICE_CORRECTION_IN` 成对切换来源，对已消费数量只记录差异额，不回写历史消费链。

### 逆操作与共享查询口径

- 作废或纠偏时必须通过逆操作恢复库存结果，并同步释放来源占用或恢复可用量，不能直接覆盖原结果。
- `F5` 第一版共享查询至少包括库存余额、库存流水、来源使用记录、只读预警视图，以及 `sales` 出库链路的编号区间占用查询。
- `inventory_warning` 收敛为只读视图 + 报表提示；预警是衍生结果，不是新的库存真源，第一版不派生独立预警处理流。

## F5 范围说明（V1）

- `F5` 不是新增一套库存交易账，而是把 `inventory-core` 里已经存在或必须存在的读能力收口成稳定共享合同，供上游业务模块统一复用。
- “第一版先最简单实现”的意思是，这一轮只交付三件事：只读预警视图 / 报表提示、`sales` 出库链路的编号区间占用与释放查询、库存余额 / 流水 / 来源使用记录等共享查询。
- “只负责销售出库链路”的意思是，当前只有 `sales` 出库单家族会向 `inventory-core` 申请、占用、释放和查询出厂编号区间；其他业务家族暂不接入这条能力。
- 因此，`F5` 当前不包含独立的预警处理流，不包含多业务家族统一编号区间平台，也不把 `inventory-core` 扩成前端直接操作的独立模块。
