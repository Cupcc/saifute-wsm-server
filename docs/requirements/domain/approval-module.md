## 主题定义

- `approval` 主题解决“单据当前审核到哪一步、谁提交了审核、谁做了结论、改单后是否需要重置、作废前是否存在下游阻塞”的问题。
- 第一阶段 `approval` 明确不是 `BPM` 引擎，而是跨单据共享的审核投影与审核动作域。
- 本主题只负责审核投影、审核动作、重置语义和下游依赖校验，不替代业务单据自己的主状态或库存逻辑。

## 承接的项目级能力版图

### 审核（canonical：`approval`）

- 本 domain 承接项目总纲中的审核模块详细职责，作为轻量审核语义与跨单据审核边界的长期真源；代码与 API canonical 名称使用 `approval`。
- 第一阶段默认采用轻量审核模式，审核主要用于记录、查询、追溯和纠偏，不引入复杂审批流、会签或转办。
- 审核记录应与单据保持可关联、可重置、可追溯；改单或作废时，业务模块可以决定是否重置审核，但真实审核落表统一由 `approval` 执行。
- `approval` 是共享核心协作域，不要求前端单独提供一个与业务域并列的“审核业务模块”。

## 长期约束

- `C1` 轻量审核而非 BPM：第一阶段只兼容 `待审 / 通过 / 拒绝` 三态审核，不引入多级审批流、加签、转办或复杂节点编排。状态：`生效中`
- `C2` 审核投影不替代业务主状态：审核记录是横切投影，业务单据的生命周期状态仍由各业务域自己负责。状态：`生效中`
- `C3` 单据表只保留快照：单据表中的 `auditStatusSnapshot` 只用于列表读取和展示，真实审核状态以 `approval_document` 为准。状态：`生效中`
- `C4` 审核不持有库存逻辑：`approval` 不直接写库存，也不反向承担库存恢复职责；库存副作用仍由业务域和 `inventory-core` 处理。状态：`生效中`
- `C5` 重置由业务决定、执行由共享域统一：改单后是否需要重置审核由业务规则决定，但一旦需要重置，必须统一通过 `approval` 落表并累计 `resetCount`。状态：`生效中`
- `C6` 下游依赖必须可解释：作废或关键修改前的下游阻塞校验，需要返回明确失败原因，不能只给模糊的“不可操作”。状态：`生效中`
- `C7` 审核动作必须可审计：提交、通过、拒绝、重置都必须可关联操作者、时间和原因，并受明确权限控制。状态：`生效中`

## 长期业务口径

### 审核投影口径

- `approval_document` 保存单据当前有效审核状态、提交信息、决策信息和重置次数，用于列表页、详情页和排障查询。
- 业务单据创建后，应创建或刷新对应审核投影；若某类单据明确不走审核，应用层需显式写入 `NOT_REQUIRED`。
- 审核投影的目标是给业务域提供统一查询与解释口径，而不是替代业务表做“一表统管”。

### 审核动作口径

- 标准审核动作包括：提交审核、审核通过、审核拒绝、审核重置。
- 审核通过 / 拒绝只改变审核投影及相关审计，不自动代表“库存生效”或“业务完成”；是否在审核时触发附加业务动作，由具体业务域另行定义。
- 重置审核后，应能解释是由改单、作废回退还是人工纠偏触发。

### 跨单据边界口径

- `inbound`、`sales`、`workshop-material` 等单据家族默认接入 `approval`；`rd-project` 第一阶段默认不接，`sales-project` 作为未来目标模块单独评估。
- `approval` 对各业务域暴露统一审核协作接口，但不直接与各单据表建立多态外键，避免反向强耦合。
- 后续若新增需要审核的新单据家族，应先确认其“是否走审核、何时重置、作废阻塞规则”，再接入共享审核域。
