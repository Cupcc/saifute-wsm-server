## 主题定义

- `workshop-material` 是车间物料流转主题，解决“主仓物料如何按车间领用、退回、报废，以及如何按车间核对净耗用与成本”的问题。
- 本主题只覆盖 `领料`、`退料`、`报废` 三类真实事务；`生产完工入库` 不属于本主题，统一由 `inbound`（生产入库单）承接。
- `workshop` 在本主题中是归属与成本核算维度，不是独立库存池；真实库存仍以主仓 `MAIN` 为准。
- `报废` 在本主题中按独立真实事务处理，不默认视为“已领料的结果拆分”或“领料附属状态”。
- 本文档只保留长期约束、长期业务口径、能力清单和能力合同；单次交付直接从本 domain 创建 `task-*.md`。

## 承接的项目级能力版图

### 车间物料（`workshop-material`）

- 本 domain 承接项目总纲中的 `workshop-material` 模块详细职责，作为车间领退料与报废语义的长期真源。
- 第一阶段默认覆盖 `领料单`、`退料单`、`报废单` 三类真实事务，并在发生后立即更新主仓实时库存。
- 领料、退料、报废应尽量保留来源追踪、回冲关系和上下游关联，便于后续追责、纠偏和车间成本核对。
- 页面中的左侧筛选、列表分栏和业务统计应优先基于 `workshop` 维度，而不是系统管理里的 `department`。

### 相邻共享边界

- `master-data` 提供 `material`、`personnel`、`workshop` 与主仓范围等主档/快照；`workshop-material` 不复制这些主数据，也不把 `department` 替代为业务归属轴。
- `inventory-core` 继续作为库存唯一写入口；车间物料家族的 `create / revise / void` 都只能通过共享库存命令产生余额、流水、来源分配与逆操作结果。
- `approval` 继续承接轻量审核投影与重置语义；`workshop-material` 只决定何时创建、刷新或重置审核状态，不把审核升级为阻断实时过账的审批流。
- `inbound` 负责生产完工入库与入库页发起的“立即出库到车间”协同入口；确认后仍必须沉淀为独立车间领料单，不能把入库单与领料单合并成一张单据。
- `reporting` / `monthly-reporting` 承接净耗用、成本汇总和导出等只读结果；`workshop-material` 自身维持单据家族与必要查询摘要，不额外拥有报表写模型。

## 长期约束

- `C1` 车间不是独立库存池：`workshop` 只承担归属与成本核算语义，不建立车间在手库存余额。状态：`生效中`
- `C2` 主仓出入账统一：领料、报废默认对主仓 `MAIN` 扣减库存，退料默认回补主仓 `MAIN`。状态：`生效中`
- `C3` 写库存必须走共享核心：车间物料单据只能通过 `inventory-core` 产生库存副作用，不能直接更新库存底表。状态：`生效中`
- `C4` 来源追踪不可省略：领料和报废属于消耗链，必须维护来源分配；退料应尽量回指原领料行，以价格对齐和回补追溯为主，但允许无源退料。状态：`生效中`
- `C5` 修改必须按差量补偿：改单不能只改表头或覆盖旧明细，必须重新计算明细差异、库存副作用和审核快照。状态：`生效中`
- `C6` 作废必须处理上下游依赖：作废前需要校验未作废的下游退料等依赖，作废时必须原子完成逆操作与来源释放。状态：`生效中`
- `C7` 审核是轻量追溯协作：车间物料默认走轻审核，不影响录单、过账和实时库存生效；审核仅用于追溯、说明和责任留痕，能力统一由 `approval` 提供。状态：`生效中`
- `C8` 净耗用是核算指标不是库存余额：车间净耗用默认按 `领料 - 退料 + 报废` 计算，仅用于车间维度汇总、月报与成本核对，不代表车间在手库存。状态：`生效中`
- `C9` 历史字段名暂不跟随重命名：当前车间物料家族中的 `auditStatusSnapshot`，以及页面读模型中的 `auditStatus` / `auditor` / `auditTime`，继续保留历史字段名；它们在语义上统一映射到 `approval` 审批状态，不应与系统审计日志混淆。状态：`注意`

## 长期业务口径

### 单据语义口径

- `领料单` 表示主仓向车间发出原材料或耗材，用于生产或车间消耗。
- `退料单` 表示车间将未使用或可回收物料退回主仓，应尽量与原领料行形成可追溯回冲关系，以价格能对上为主；若无法可靠定位原领料行，可按无源退料处理，但不得伪造关系。
- `报废单` 表示按车间归属确认的独立物料损耗事务，属于最终消耗链，不回补库存，也不默认要求先存在对应领料单。
- 三类单据应统一收敛为同一家族建模，通过 `orderType` 区分业务语义，而不是继续散落在多个来源模块中。

### 库存与来源口径

- 领料、报废属于 `OUT` 方向，退料属于 `IN` 方向；库存事实仍统一落在主仓范围。
- 车间物料单据中的 `unitPrice / amount` 是内部领用、退料、报废的成本核算口径，不是销售价；领料和报废的实际消耗成本仍必须能通过来源分配追溯到 `inventory_log.unitCost`。
- 消耗类动作需维护 `inventory_source_usage`，以便后续成本追溯、作废恢复和月报汇总；报废若引用领料或其他上游关系，该关系仅用于追溯和成本分析，不改变其独立事务语义。
- 退料在线运行时应尽量绑定原领料行，以价格对齐和追溯可用为主；若无法可靠匹配，允许无源退料，但不能伪造关系。

### 车间查询与核算口径

- 车间侧查询重点是“这个车间本月领了多少、退了多少、报废了多少、净耗用和成本是多少”，而不是“车间现在手上有多少库存”。
- `净耗用` 是只读汇总口径，默认计算公式为 `已过账领料 - 已过账退料 + 已过账报废`；统计时默认纳入已过账且未作废的单据，轻审核状态不影响纳入时点。
- 车间物料列表、明细、导出和月报视图应能按车间、物料、经办人、单据类型等常见维度过滤。
- 车间查询页不单独建设独立“净耗用”口径页；净耗用专项汇总与只读展示统一由 `monthly-reporting` / `reporting` 承接，车间物料页只保留单据查询、明细和必要摘要。
- 入库页面发起的“立即出库到车间”属于 `inbound` 的快捷协同能力；确认后仍必须沉淀为独立的车间领料单并落入本主题家族。

## 能力合同（推荐）

### `F1` 领料 / 退料 / 报废统一单据家族

- In scope:
  - `pick`、`return`、`scrap` 在统一家族模型下共用 `workshop_material_order` / `workshop_material_order_line`，通过 `orderType`、库存方向和权限入口区分语义。
  - 对外继续暴露稳定的家族化路由与查询模型，而不是回退为三套互相漂移的后端实现。
  - 前端承载可继续使用既有 `pickOrder` / `returnOrder` / `scrapOrder` 三页，但这不改变后端统一家族这一长期真源。
- Out of scope / non-goals:
  - 不把 `生产完工入库`、销售业务、销售项目供货并入本家族。
  - 不新建“统一 workbench”作为本能力成立前提。
- Completion criteria:
  - `[TC-1]` 文档明确三类事务共用统一主从表和统一领域边界。
  - `[TC-2]` 文档明确外部语义通过 `orderType` 与路由入口区分，而不是重新拆散表和模块。
  - `[TC-3]` 文档明确查询、详情、修改、作废仍围绕同一家族合同演进。
  - `[TC-4]` 文档明确三页面承载是交付形态，不改变统一家族的长期要求。
- Evidence expectation:
  - domain 文档真源 + 对应 task 的 API / UI 验收证据。
- Default derived slice acceptance mode: `full`

### `F2` 主仓库存联动与来源追溯

- In scope:
  - 领料、报废对主仓 `MAIN` 扣减，退料对主仓 `MAIN` 回补。
  - 所有库存副作用、来源分配、来源释放与逆操作统一经 `inventory-core` 完成。
  - 领料、报废作为消耗链必须保留 `inventory_source_usage`；退料应尽量回指原领料行并保持价格与回补追溯一致。
- Out of scope / non-goals:
  - 不把 `workshop` 升级为真实库存池。
  - 不为了补齐来源追溯而伪造退料来源关系。
- Completion criteria:
  - `[TC-1]` 文档明确 `inventory-core` 是唯一库存写入口，业务模块不得旁路改库存底表。
  - `[TC-2]` 文档明确 `PICK` / `SCRAP` 属于 `OUT`，`RETURN` 属于 `IN`，且真实库存范围固定在主仓。
  - `[TC-3]` 文档明确消耗类动作必须保留来源分配；退料允许无源，但不得伪造回冲链。
  - `[TC-4]` 文档明确作废和改单时来源占用需要同步释放或重放，不能只改单据表面状态。
- Evidence expectation:
  - domain 文档真源 + `inventory_log` / `inventory_source_usage` / `document_line_relation` 对应验证。
- Default derived slice acceptance mode: `full`

### `F3` 回冲关系、作废补偿与审核快照协同

- In scope:
  - 退料与领料保持可追溯回冲关系；改单采用“逆转旧副作用 -> 重放新副作用”的确定性补偿语义。
  - 领料改单在存在未作废退料下游时必须阻断；退料、报废改单允许 restore + replay。
  - 改单、作废后同步刷新 `revisionNo`、来源关系、库存副作用状态与审核快照/重置记录。
- Out of scope / non-goals:
  - 不把轻审核升级为过账前审批流。
  - 不以“覆盖旧明细”替代补偿式改单。
- Completion criteria:
  - `[TC-1]` 文档明确改单必须重算明细差异并通过逆操作 + 重放处理库存与来源结果。
  - `[TC-2]` 文档明确领料存在有效退料下游时不得修改或作废通过。
  - `[TC-3]` 文档明确退料 / 报废改单允许补偿式重放，并保留回冲链与来源追溯可解释性。
  - `[TC-4]` 文档明确作废后 `inventoryEffectStatus`、来源释放与审核快照重置需要一致落地。
  - `[TC-5]` 文档明确历史字段名 `auditStatusSnapshot` 仍冻结，但语义指向 `approval` 状态。
- Evidence expectation:
  - domain 文档真源 + `revisionNo`、`approval_document.resetCount`、逆向库存流水与 void 结果验证。
- Default derived slice acceptance mode: `full`

### `F4` 车间维度查询、汇总与导出

- In scope:
  - 按车间、物料、经办人、单据类型等维度查询领料、退料、报废明细。
  - 提供净耗用与成本的只读汇总口径，并支持后续导出与月报协同。
  - 维持“车间是归属 / 核算维度，不是库存池”的查询解释口径。
- Out of scope / non-goals:
  - 不把 `F4` 与 `F1/F2/F3` 的已验收交付混签；当前 accepted delivery 不包含此能力。
  - 不在 `workshop-material` 内新增一套独立事务写模型来承接报表。
- Completion criteria:
  - `[TC-1]` 文档明确净耗用公式、纳入口径和过滤维度。
  - `[TC-2]` 文档明确净耗用 / 成本汇总与导出由 `reporting` / `monthly-reporting` 承接，车间物料页只保留单据查询与必要摘要。
  - `[TC-3]` 文档明确任何查询展示都不得把车间解释成真实在手库存余额。
- Evidence expectation:
  - domain 文档真源 + 后续独立 task 的查询、汇总、导出与报表验证。
