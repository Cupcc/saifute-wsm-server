# 系统管理模块主题需求

## Metadata

- ID: `domain-system-management-module`
- Status: `confirmed`
- Scope: `domain-level`

## 主题定义

- `system-management` 是平台层长期主题，解决"谁能登录、能看到什么、能操作什么、组织关系如何表达，以及哪些平台配置由系统统一维护"的问题。
- 当前真实组织口径已先收敛为 `研发部 / 采购部 / 仓库` 三个一级部门；当前系统使用者先按 `系统管理员 / 仓库管理员 / 研发小仓管理员 / 采购人员` 四类主角色设计，后续可能增加 `老板 / 财务` 两类偏查看角色。
- 当前运行态代表账号以 `admin / operator / rd-operator / procurement` 为主；其中 `operator` 保留兼容用户名，但业务语义已对齐为 `仓库管理员`，`system-manager / ai-operator` 不再作为长期主口径。
- 当前主题优先覆盖 `用户 / 角色 / 部门 / 菜单 / 字典 / 参数 / 通知` 这些核心系统管理对象；`岗位` 虽然在现有系统页中已存在，但当前真实业务口径里不是必选治理对象。
- 本文档只保留长期约束、能力清单、阶段路线图与文档关系；具体实现或修复直接从本 domain 能力合同创建 `task-*.md`。

## 承接的项目级能力版图

### 平台与横切域

- 本 domain 承接项目总纲中 `auth`、`session`、`rbac`、`system-management`、`audit-log`、`file-storage`、`scheduler` 的详细职责，作为平台与横切能力的长期真源。
- `auth`：系统默认只服务公司内部使用人员，第一阶段不开放注册、第三方登录或对外账号体系；保留验证码、用户名密码登录、登出、当前用户信息与前端路由初始化入口。
- `session`：登录态继续采用 `JWT + Redis 会话`，不切换为纯无状态认证；需要支持滑动续期、在线用户查看、强制下线和注销后立即失效。
- `rbac`：默认延续“菜单 / 按钮权限字符串 + 数据权限”模型，支撑用户、角色、菜单、授权关系维护，并向前端输出按用户过滤后的路由树。
- `system-management`：第一阶段以 `用户 / 角色 / 部门 / 菜单 / 字典 / 参数 / 通知` 为核心治理对象；`岗位` 保留兼容，但当前不作为真实组织设计的必选对象。
- `audit-log`：需要保留登录日志和操作日志，满足问题追查、责任追溯和管理审计；关键业务操作记录操作者、时间、对象、结果和异常摘要，敏感字段默认脱敏。
- `file-storage`：第一阶段延续本地磁盘文件方案，兼容历史 `/profile/**` 访问语义，支持普通上传、下载、头像管理和导入导出类文件交互。
- `scheduler`：保留数据库驱动的任务管理能力，支持任务定义、启停、立即执行和执行日志；当前仍按平台邻接能力规划，不默认引入更复杂的分布式调度体系。

## 长期约束

- `C1` 单一权限真源：`/api/auth/me`、`/api/auth/routes`、`PermissionsGuard`、前端动态菜单、按钮可见性、角色菜单关系与数据权限策略必须来自同一套系统管理真源，前后端不能各自维护一套解释。状态：`生效中`
- `C2` 会话真源统一：`JWT` 只作为会话索引，用户角色、权限、部门、`consoleMode` 等运行态真源仍在 `Redis session`；系统管理变更后必须能够解释会话刷新或失效策略。状态：`生效中`
- `C3` 平台层与业务层分离：`部门 / 角色 / 菜单 / 参数 / 通知` 属于平台治理对象，`岗位` 当前为可选兼容对象；这些平台对象都不应直接替代业务模块里的仓别、库存范围、研发项目或销售项目等业务真源。状态：`生效中`
- `C4` 组织权限不等于库存范围：部门和角色可以影响菜单、页面与数据权限，但真实库存访问仍沿 `stockScope / workshopScope` 收口，不能把组织树直接当成库存边界。状态：`生效中`
- `C5` 新系统自建口径：旧 Java 平台的 `sys_user / sys_role / sys_menu / sys_dept / sys_post / sys_config / sys_notice / sys_dict_*` 不作为当前正式业务真源整表迁移，应按新系统方案重建为规范化数据库表。状态：`生效中`
- `C6` 初始化独立于业务迁移：系统管理初始化与业务数据迁移是两套 checklist；数据库 schema 初始化不等于系统管理 seed 初始化。状态：`生效中`
- `C7` 视角裁切不能替代授权：`consoleMode` 只负责壳层与默认入口体验，不负责替代权限判定；`admin` / 系统管理员不应因为视角裁切丢失本应可见页面。状态：`生效中`
- `C8` 超级管理员语义收敛：超级管理员语义必须在数据库、后端鉴权、会话快照和前端识别上端到端一致，不能出现多套互相漂移的"admin 判定"。状态：`生效中`
- `C9` 审计与失效策略：用户、角色、菜单和授权关系的关键变更需要保留审计能力，并在合理范围内触发会话刷新或失效，避免长期 stale permissions。状态：`生效中`
- `C10` 审计与调度持久化分层：`sys_logininfor`、`sys_oper_log` 已有规范化 Prisma model 与独立 repository，是登录日志与操作日志的落库真源；`sys_job`、`sys_job_log` 属于调度任务域，Phase 3 再规划，不纳入 V1 系统管理治理范围。状态：`生效中`
- `C11` 系统管理主数据真实落库：`用户 / 角色 / 部门 / 菜单 / 岗位 / 字典 / 参数 / 通知` 等系统管理对象必须以规范化数据库表持久化，不以 JSON 快照或进程内内存状态作为长期真源；`system_management_snapshot` 桥接方案已废弃。状态：`生效中`

## 长期业务口径

### 核心对象口径

- `用户管理` 负责账号、登录身份、部门归属、角色绑定、启停与密码重置；当前阶段不要求岗位必填。
- `角色管理` 负责一组权限与路由能力的打包，也负责数据权限范围的定义。
- `部门管理` 负责组织树与角色数据范围的组织边界；当前真实组织先按 `研发部 / 采购部 / 仓库` 三个一级部门建模，不直接代表真实库存仓别，也不替代业务页面中的车间划分。生产车间相关页面若需分栏或筛选，应由业务侧按 `master-data.workshop` 承接。
- `菜单管理` 负责页面、路由与按钮权限字符串的承接，是前端动态菜单与后端 `@Permissions()` 的共同锚点。
- `岗位管理` 当前仅作为系统已有兼容能力保留；若真实业务后续仍不需要岗位维度，可在后续切片中继续降权或弱化。
- `字典 / 参数 / 通知` 属于平台辅助配置，能承接通用枚举、少量系统参数与公告，但不宜继续承接核心业务事实。

### 当前角色设计口径

- 当前主角色先按 `系统管理员 / 仓库管理员 / 研发小仓管理员 / 采购人员` 设计，满足最小可用权限模型。
- 当前运行态代表账号按 `admin / operator / rd-operator / procurement` 维护；其中 `operator` 对齐 `仓库管理员`，`procurement` 对齐 `采购人员`。
- 当前建议版本 `V1`：
  - `系统管理员`：保留全量系统管理与跨域排障兜底能力
  - `仓库管理员`：负责主仓业务与公司级实物流转，并可查看 RD 协同必要信息
  - `研发小仓管理员`：负责 RD 小仓内部日常作业，只看 RD 视角与主仓协同结果
  - `采购人员`：负责采购执行链路，可回看与采购闭环有关的主仓验收结果
- 旧的 `system-manager / ai-operator` 仅保留历史溯源意义，不再作为当前长期主角色或代表账号基线。
- `老板` 与 `财务` 当前先视为后续可能增加的查看型角色；是否在这一轮 domain 下立即细化，仍待确认。
- 当前阶段优先靠"角色"表达系统使用者差异，不额外引入岗位层来增加治理复杂度。

### 邻接能力口径

- 当前建议把 `在线用户 / 登录日志 / 操作日志` 纳入 `system-management` 第一版长期 domain，因为它们天然属于平台治理、排障与审计能力。
- 当前建议暂不把 `调度任务 / AI 支持` 纳入 `system-management` 第一版长期 domain，避免主题膨胀；作为非核心平台邻接能力顺延到 Phase 3 规划，当前状态暂不执行。
