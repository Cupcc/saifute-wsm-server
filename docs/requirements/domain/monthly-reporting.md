# 月度报表主题需求

## Metadata

- ID: `domain-monthly-reporting`
- Status: `confirmed`
- Scope: `domain-level`

## 主题定义

- 月度报表仍是 `reporting` 域的长期真源，用于定义“系统如何稳定地产出可核对、可追溯、可导出的月度统计结果”。
- 当前已确认的一阶段优先切片，不再是“大而全月报总纲”，而是：
  - `仓库侧按月核对本期发生金额`
  - 由仓库人员把系统内报表或导出结果拿去和外部财务系统核对
  - 当金额不一致时，在系统内继续下钻到业务主题与单据头做排查
- 第一阶段不以“历史余额已经可信”作为前提，也不把“正式月报冻结 / 人工重算 / 日期范围报表”作为首要交付。

## 当前已确认的一阶段切片

### 一句话定义

- 以 `bizDate` 为归月口径，按自然月统计仓库侧“本期发生金额”，支持仓库按月与外部财务系统对账，并在差异出现时从主题汇总下钻到单据头查看数量、金额、成本与异常标识。

### 使用者

- 仓库管理员：系统内查看与导出月度对账报表，负责和财务对账及差异排查。
- 研发小仓管理员：仅在自身 `RD_SUB` 范围内查看与本仓相关的月度金额主题。
- 外部财务人员：当前不作为系统内直接操作角色，而是通过仓库提供的报表 / 导出结果参与对账。

### 核心目标

- 以“本月发生了哪些金额变化”为核心对账对象，而不是先证明历史期初 / 期末余额完全可信。
- 生成一份仓库侧完整的月度发生金额报表，覆盖当前所有会影响仓库月度金额的业务主题。
- 顶层先提供可以直接拿去与财务核对的金额汇总。
- 下层提供数量、金额、成本与单据头证据，用来解释顶层金额为什么成立。

### 明确边界

- 第一阶段只生成仓库侧报表，不在系统内录入、导入或自动比对财务侧数字。
- 第一阶段不承诺修复历史余额问题，不把收发存全链闭环作为前置条件。
- 第一阶段默认追溯到单据头，不把“所有主题都必须支持单据行级下钻”写成普遍合同。
- 第一阶段不把技术性逆操作当成独立业务主题展示。
- 第二层单据头清单的统一字段合同，当前不在本主题文档里写死，留待后续任务级设计收口。

## 长期定位

- 本文档仍保留“月报域长期真源”的角色。
- 长期看，`monthly-reporting` 仍应承接：
  - 正式月报结果沉淀
  - 人工重算结果
  - 日期范围报表
  - 后续多视角月度分析
- 但当前优先顺序已经确认：
  - 先把“仓库侧本期发生金额月度对账”做扎实
  - 再扩展到正式月报冻结、重算、范围报表等更重的能力

## 统计口径

### 总体原则

- 月度报表属于只读派生结果，不是库存或财务的主数据来源。
- 第一阶段报表结果应从业务单据、库存流水和成本结果中汇总生成，保证可追溯。
- 第一阶段默认按 `bizDate` 归月，且以自然月为统计周期。
- 若未来改为按审核 / 生效日期归月，应视为正式统计口径变更，而不是实现细节微调。

### 第一阶段对账对象

- 第一阶段对账对象是：
  - `本期发生金额`
- 第一阶段不是以下对象：
  - 历史期初余额可信性证明
  - 收发存全链闭环
  - 财务系统内正式入账结果

### 目录结构

- 第一阶段报表采用两层结构：
  - 第一层：稳定总类，用于维持目录稳定
  - 第二层：具体业务主题，用于贴近仓库和财务的实际核对对象
- 第一层可按以下稳定总类组织：
  - `入库`
  - `出库`
  - `消耗`
  - `调整 / 协同`

### 当前已纳入的业务主题清单

第一阶段覆盖当前所有会影响仓库月度金额的业务主题，不只限于少数试点主题。建议按“主题家族 + 当前已知动作”组织：

- `inbound / 入库`
  - 验收入库
  - 生产入库
- `sales / 销售`
  - 销售出库
  - 销售退货
- `workshop-material / 车间`
  - 领料
  - 退料
  - 报废
- `rd-project / 研发项目`
  - 项目领用
  - 项目退回
  - 项目报废
- `rd-subwarehouse / RD 小仓协同`
  - 主仓到 RD 交接
  - RD 盘盈
  - RD 盘亏
- `调整 / 纠偏`
  - 调价转入
  - 调价转出
  - 补录影响
  - 跨月修正
  - 盘点调整

说明：

- 这份清单的目标是冻结“当前已纳入第一阶段月度金额对账的主题范围”。
- `REVERSAL_IN / REVERSAL_OUT` 这类技术性逆操作不作为独立业务主题出现在报表目录中，应归并回原业务主题理解。

### 展示与追溯合同

- 第一层默认以 `金额` 为主，不要求顶层同时展开全部数量与成本细节。
- 第二层默认展示：
  - `数量`
  - `金额`
  - `成本`
- 第二层默认最小追溯粒度是 `单据头`。
- 第一阶段至少要能回答：
  - 某个业务主题本月金额由哪些单据构成
  - 哪张单据可能是差异来源

### 异常 / 纠偏展示规则

- 调价、补录、跨月修正、盘点调整等异常 / 纠偏类金额：
  - 仍归属原业务主题
  - 但必须保留异常标识或异常栏目
- 跨月修正、前期纠偏等“非本月正常发生”的金额：
  - 仍计入本月报表
  - 但只在第二层或异常列中标识
  - 不要求在第一层主表中打断金额结构

### 技术逆操作规则

- `REVERSAL_IN / REVERSAL_OUT` 不作为独立业务主题展示。
- 若报表需要保留相关痕迹，应在第二层或异常说明中解释其归并关系。

## 长期约束

- `C1` `reporting` 继续保持只读聚合定位，不拥有事务写模型。
- `C2` 第一阶段默认按 `bizDate` 归月，以自然月为统计周期。
- `C3` 第一阶段仅产出仓库侧报表，不做财务侧数据回填、导入或系统内自动差异比对。
- `C4` 第一阶段默认追溯到单据头，不把单据行级追溯作为所有主题的通用要求。
- `C5` 第一阶段顶层以金额为主，下层以数量 + 金额 + 成本作为解释证据。
- `C6` 异常 / 纠偏金额保留在原业务主题内，但必须可被标识和追溯。
- `C7` 技术性逆操作不应污染业务主题目录。
- `C8` 系统需要支持系统内查看与 `Excel` 导出，便于仓库与外部财务离线核对。
- `C9` 长期上仍必须保留“正式月报 / 人工重算 / 日期范围报表”三类语义分离原则，后续阶段不能回退为混合语义报表。

## 查询与导出

- 第一阶段报表需要支持系统内查看，不应只依赖导出文件。
- 第一阶段报表需要支持 `Excel` 导出，供仓库与外部财务离线核对。
- 查询至少支持按月、按仓别、按车间 / 归属维度、按业务主题过滤。
- 主仓与 `RD_SUB` 小仓仍需按现有范围隔离查看，不默认把主仓运营面数据暴露给小仓管理员。
- 第二层单据头清单应尽量复用现有单据头共性字段，但其最终统一字段合同暂不在本主题文档里冻结。

## 当前阶段范围外

- 在系统内录入财务侧汇总数。
- 在系统内导入财务侧对账文件。
- 自动生成仓库 vs 财务正式差异结果。
- 以历史期初 / 期末余额可信为前提的收发存闭环。
- 所有主题统一支持单据行级追溯。
- 先行定义一份硬编码的“仓库口径 -> 财务口径”映射字典。
- 把 `REVERSAL_IN / REVERSAL_OUT` 做成独立业务主题展示。
- 将第二层单据头清单字段合同在本轮文档中写死。
- 实时数仓或复杂 `OLAP` 分析体系。

## 功能项说明

### `F1` 本期发生金额月度口径

- 这项功能要解决什么：先把“本月发生了哪些金额变化”稳定统计出来。
- 明确不包括什么：不先修复历史余额，不先做收发存全链闭环。
- 完成标准：
  - `[TC-1]` 报表按 `bizDate` 归月。
  - `[TC-2]` 报表以自然月为统计周期。
  - `[TC-3]` 用户可以理解“本月发生金额”而非“系统本月生效金额”的定义。

### `F2` 业务主题全量范围与两层目录

- 这项功能要解决什么：把第一阶段实际纳入的业务主题冻结为可执行清单。
- 明确不包括什么：不在此阶段承诺未来尚未进入仓库月度核对流程的新主题。
- 完成标准：
  - `[TC-1]` 第一层目录稳定。
  - `[TC-2]` 第二层主题覆盖当前所有会影响仓库月度金额的业务家族。
  - `[TC-3]` 技术逆操作不作为独立业务主题出现。

### `F3` 主题汇总到单据头追溯

- 这项功能要解决什么：让金额不一致时，用户能快速定位问题单据。
- 明确不包括什么：不要求所有主题默认细到单据行。
- 完成标准：
  - `[TC-1]` 用户可从主题汇总下钻到单据头清单。
  - `[TC-2]` 第二层默认能展示数量、金额、成本。
  - `[TC-3]` 用户能据此判断差异来源单据。

### `F4` 异常 / 跨月修正展示规则

- 这项功能要解决什么：在不破坏主表结构的前提下，解释调价、补录、跨月修正等影响。
- 明确不包括什么：不把异常统一拆成独立主表。
- 完成标准：
  - `[TC-1]` 异常金额仍归属原业务主题。
  - `[TC-2]` 第二层或异常列中能识别跨月修正 / 前期纠偏。
  - `[TC-3]` 技术逆操作被归并回原业务主题理解。

### `F5` 仓库侧查看与导出

- 这项功能要解决什么：让仓库能直接拿系统结果去和财务核对。
- 明确不包括什么：不在系统内录入、导入、自动比对财务数据。
- 完成标准：
  - `[TC-1]` 报表支持系统内查看。
  - `[TC-2]` 报表支持 `Excel` 导出。
  - `[TC-3]` 导出结果可被外部财务核对使用。

### `F6` 正式月报结果与人工重算

- 这项功能要解决什么：在后续阶段沉淀长期保留的正式月报结果，并区分重算结果。
- 当前状态说明：保留为后续阶段能力，不是第一阶段首要交付。

### `F7` 日期范围报表语义分离

- 这项功能要解决什么：保留临时分析能力，同时不破坏正式月报语义。
- 当前状态说明：保留为后续阶段能力，不是第一阶段首要交付。
