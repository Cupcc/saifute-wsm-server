# 月度对账需求

## Metadata

- ID: `domain-monthly-reporting`
- Status: `confirmed`
- Scope: `domain-level`
- 状态说明: 当前运行时已在 `reporting` 模块完成 `Phase 1`：系统可按 `bizDate + 自然月` 生成仓库侧月度对账汇总、业务操作下钻、异常标识、系统查看与 `Excel` 导出；`RD_SUB` 范围隔离入口与只读查询也已落地。正式月报冻结、人工重算、日期范围报表仍保留在后续阶段。

## 主题定义

- 月度报表仍是 `reporting` 域的长期真源，用于定义“系统如何稳定地产出可核对、可追溯、可导出的月度统计结果”。
- 已确认的后续扩展规则：当 `RD_SUB` 物料强制归属研发项目后，月报必须能够把同一笔 `RD handoff` 解释成主仓视角的内部转移、研发项目视角的项目入账，以及全仓视角的内部协同，而不是继续把 `RD handoff` 与研发项目完全割裂。
- 当前已确认的一阶段优先切片，不再是“大而全月报总纲”，而是：
  - `仓库侧按月核对本期发生金额`
  - 由仓库人员把系统内报表或导出结果拿去和外部财务系统核对
  - 当金额不一致时，在系统内继续下钻到业务主题与单据头做排查
- 第一阶段不以“历史余额已经可信”作为前提，也不把“正式月报冻结 / 人工重算 / 日期范围报表”作为首要交付。

## 当前已确认的一阶段切片

### 一句话定义

- 以 `bizDate` 为归月口径，按自然月统计仓库侧“本期发生金额”，先回答当前仓库范围的 `总入 / 总出 / 净发生`，再按业务领域展开操作汇总，并在差异出现时下钻到单据头查看数量、金额、成本与异常标识。

### 使用者

- 仓库管理员：系统内查看与导出月度对账报表，负责和财务对账及差异排查。
- 研发小仓管理员：仅在自身 `RD_SUB` 范围内查看与本仓相关的月度金额主题。
- 外部财务人员：当前不作为系统内直接操作角色，而是通过仓库提供的报表 / 导出结果参与对账。

### 核心目标

- 以“本月发生了哪些金额变化”为核心对账对象，而不是先证明历史期初 / 期末余额完全可信。
- 生成一份仓库侧完整的月度发生金额报表，覆盖当前所有会影响仓库月度金额的业务主题。
- 顶层先提供可以直接拿去与财务核对的 `总入 / 总出 / 净发生` 汇总。
- 第二层按业务领域展开，让仓库先按业务领域排查，而不是先按技术动作目录排查。
- 下层提供数量、金额、成本与单据头证据，用来解释领域汇总为什么成立。
- 页面与导出中的用户可见名称，应与业务实际名称保持一致，例如 `领料 / 退料 / 报废 / 销售出库 / 销售退货 / 项目交接`，不使用报表层自定义黑话。

### 明确边界

- 第一阶段只生成仓库侧报表，不在系统内录入、导入或自动比对财务侧数字。
- 第一阶段不承诺修复历史余额问题，不把收发存全链闭环作为前置条件。
- 第一阶段默认追溯到单据头，不把“所有主题都必须支持单据行级下钻”写成普遍合同。
- 第一阶段不把技术性逆操作当成独立业务主题展示。
- 第二层单据头清单的统一字段合同，当前不在本主题文档里写死，留待后续任务级设计收口。

## 长期定位

- 本文档仍保留“月报域长期真源”的角色。
- 长期看，`monthly-reporting` 仍应承接：
  - 正式月报结果沉淀
  - 人工重算结果
  - 日期范围报表
  - 后续多视角月度分析
- 但当前优先顺序已经确认：
  - 先把“仓库侧本期发生金额月度对账”做扎实
  - 再扩展到正式月报冻结、重算、范围报表等更重的能力

## 统计口径

### 总体原则

- 月度报表属于只读派生结果，不是库存或财务的主数据来源。
- 第一阶段报表结果应从业务单据、库存流水和成本结果中汇总生成，保证可追溯。
- 第一阶段默认按 `bizDate` 归月，且以自然月为统计周期。
- 若未来改为按审核 / 生效日期归月，应视为正式统计口径变更，而不是实现细节微调。

### 第一阶段对账对象

- 第一阶段对账对象是：
  - `本期发生金额`
- 第一阶段不是以下对象：
  - 历史期初余额可信性证明
  - 收发存全链闭环
  - 财务系统内正式入账结果

### 目录结构

- 第一阶段报表采用“总览优先 + 领域优先”的结构：
  - 总览层：先回答当前筛选范围内的 `总入 / 总出 / 净发生`
  - 领域层：再按业务领域展开操作
  - 追溯层：最终下钻到单据头
- 第一阶段领域层应按以下稳定领域组织：
  - `入库`
  - `车间`
  - `销售`
  - `研发项目`
  - `RD小仓`
- `销售项目` 视角属于 `销售` 域下的业务汇总，不应为了展示方便把同一批销售事实重复拆成与 `销售` 并列的一级目录。

### 当前已纳入的领域与操作清单

第一阶段覆盖当前所有会影响仓库月度金额的业务事实，但页面与导出应优先按“领域 -> 操作 -> 业务汇总 / 单据”组织，而不是把动作类型直接抬成一级目录：

- `inbound / 入库`
  - 验收入库
  - 生产入库
  - 入库调价转入 / 转出仍归属入库域，不单独升成一级域
- `workshop-material / 车间`
  - 领料
  - 退料
  - 报废
  - 车间汇总默认按 `workshop`
- `sales / 销售`
  - 销售出库
  - 销售退货
  - 销售项目汇总至少要支持 `salesProjectId / salesProject` 项目视角，回答“销售中的每个项目发生了多少”
- `rd-project / 研发项目`
  - 项目交接
  - 项目领用
  - 项目退回
  - 项目报废
  - 研发项目汇总默认按 `rd-project`
- `rd-subwarehouse / RD小仓`
  - `RD盘盈 / RD盘亏`
  - 盘点调整等 RD 内部仓务变动可以继续保留在 RD 相关读模型里，但不得替代研发项目正式入账视图
- `异常 / 纠偏`
  - 调价
  - 补录影响
  - 跨月修正
  - 盘点调整
  - 上述项默认是异常标识或领域内操作说明，不再单独作为一级领域

说明：

- 这份清单的目标是冻结“当前已纳入第一阶段月度金额对账的主题范围”。
- `REVERSAL_IN / REVERSAL_OUT` 这类技术性逆操作不作为独立业务主题出现在报表目录中，应归并回原业务主题理解。

### 已确认的后续扩展：RD 小仓项目化口径

- 当 `RD_SUB` 物料全部归属研发项目后，月报不能再把 `RD handoff` 与 `rd-project` 汇总视为完全分离的两条账。
- 同一笔 `RD handoff` 在不同视角下可以有不同呈现：
  - `MAIN` 视角：可解释为主仓总出或内部转移出
  - `RD_SUB / rd-project` 视角：可解释为项目总入或交接入项目
  - 全仓总览视角：仍可视为内部转移，不应被粗暴重复并入总入 / 总出
- 视角转换必须建立在交接事实层已有项目归属的前提上，不能靠报表层按备注、车间或单据号猜项目归属。

### 已完成的后续扩展：物料分类视角

- `monthly-reporting` 已在同一页面内新增 `领域视角 / 物料分类视角` 切换，不替换既有领域视角。
- 第一版分类视角只覆盖 `验收入库`、`生产入库`、`销售出库`、`销售退货` 四类业务事实，不扩到 `workshop-material`、`rd-project`、`rd-subwarehouse`。
- 分类金额必须基于 `stock_in_order_line` 与 `sales_stock_order_line` 的单据行事实，不允许沿用单据头月报结果做二次 regroup。
- 分类归属采用业务发生时快照：业务写侧在单据行上冻结稳定最终分类快照；历史 backfill 以迁移执行时主数据为基线补齐。
- 汇总规则固定为“单据行只归属一个稳定最终分类”；`未分类` 必须作为稳定分类参与汇总。

### 展示与追溯合同

- 总览层默认以 `金额` 为主，至少回答：
  - 当前仓库范围总共入了多少
  - 当前仓库范围总共出了多少
  - 净发生是多少
- 领域层默认展示：
  - 该领域的操作汇总
  - 该领域对应的业务汇总
- 业务汇总至少覆盖 `车间 / 销售项目 / 研发项目 / 主仓到RD交接` 等真实业务锚点
- 业务汇总至少覆盖 `车间 / 销售项目 / 研发项目` 等真实业务锚点；项目交接已并入研发项目汇总，不再保留独立历史桶
- 单据层默认展示：
  - `数量`
  - `金额`
  - `成本`
- 单据层默认最小追溯粒度是 `单据头`。
- 物料分类视角属于 follow-on 分支：
  - 总览至少回答 `验收入库金额 / 生产入库金额 / 销售出库金额 / 销售退货金额 / 净发生金额`
  - 分类汇总必须按单层分类展示，不使用分类路径、祖先链或父级树形汇总
  - 分类明细的默认最小追溯粒度是 `单据行`，并展示该行稳定最终分类
- 第一阶段至少要能回答：
  - 当前仓库本月总入 / 总出分别由哪些领域构成
  - 某个领域下某个车间 / 销售项目 / 研发项目本月金额由哪些单据构成
  - 哪张单据可能是差异来源

### 异常 / 纠偏展示规则

- 调价、补录、跨月修正、盘点调整等异常 / 纠偏类金额：
  - 仍归属原业务主题
  - 但必须保留异常标识或异常栏目
- 跨月修正、前期纠偏等“非本月正常发生”的金额：
  - 仍计入本月报表
  - 但只在第二层或异常列中标识
  - 不要求在第一层主表中打断金额结构

### 技术逆操作规则

- `REVERSAL_IN / REVERSAL_OUT` 不作为独立业务主题展示。
- 若报表需要保留相关痕迹，应在第二层或异常说明中解释其归并关系。

## 长期约束

- `C1` `reporting` 继续保持只读聚合定位，不拥有事务写模型。
- `C2` 第一阶段默认按 `bizDate` 归月，以自然月为统计周期。
- `C3` 第一阶段仅产出仓库侧报表，不做财务侧数据回填、导入或系统内自动差异比对。
- `C4` 第一阶段默认追溯到单据头，不把单据行级追溯作为所有主题的通用要求。
- `C5` 第一阶段顶层先回答 `总入 / 总出 / 净发生`，领域层再按业务领域展开，下层以数量 + 金额 + 成本作为解释证据。
- `C6` 异常 / 纠偏金额保留在原业务领域内，但必须可被标识和追溯。
- `C7` 技术性逆操作不应污染业务领域目录，也不应把异常类动作抬成主领域。
- `C8` 系统需要支持系统内查看与 `Excel` 导出，便于仓库与外部财务离线核对。
- `C9` 长期上仍必须保留“正式月报 / 人工重算 / 日期范围报表”三类语义分离原则，后续阶段不能回退为混合语义报表。
- `C10` 当 `RD_SUB` 实施项目化归属后，月报必须按视角重算 `RD handoff` 的入 / 出归类，并把项目交接直接并入研发项目入账，不能再通过单一固定“交接金额”列掩盖项目入账语义。状态：`已完成`
- `C11` 物料分类视角必须建立在单据行分类快照之上；查询时不得回读当前 `material.categoryId` 重算历史月份归属。状态：`已完成`

## 查询与导出

- 第一阶段报表需要支持系统内查看，不应只依赖导出文件。
- 第一阶段报表需要支持 `Excel` 导出，供仓库与外部财务离线核对。
- 查询至少支持按月、按仓别、按车间 / 归属维度、按业务领域 / 操作过滤。
- 主仓与 `RD_SUB` 小仓仍需按现有范围隔离查看，不默认把主仓运营面数据暴露给小仓管理员。
- 第二层单据头清单应尽量复用现有单据头共性字段，但其最终统一字段合同暂不在本主题文档里冻结。

## 当前阶段范围外

- 在系统内录入财务侧汇总数。
- 在系统内导入财务侧对账文件。
- 自动生成仓库 vs 财务正式差异结果。
- 以历史期初 / 期末余额可信为前提的收发存闭环。
- 所有主题统一支持单据行级追溯。
- 先行定义一份硬编码的“仓库口径 -> 财务口径”映射字典。
- 把 `REVERSAL_IN / REVERSAL_OUT` 做成独立业务主题展示。
- 将第二层单据头清单字段合同在本轮文档中写死。
- 实时数仓或复杂 `OLAP` 分析体系。

## 功能项说明

### `F1` 总入 / 总出 / 净发生月度口径

- 这项功能要解决什么：先把“本月发生了哪些金额变化”稳定统计出来。
- 明确不包括什么：不先修复历史余额，不先做收发存全链闭环。
- 完成标准：
  - `[TC-1]` 报表按 `bizDate` 归月。
  - `[TC-2]` 报表以自然月为统计周期。
  - `[TC-3]` 用户可以先看到当前仓库范围的 `总入 / 总出 / 净发生`，而不是只有单一总金额。

### `F2` 领域优先目录与业务汇总

- 这项功能要解决什么：把第一阶段实际纳入的业务领域冻结为可执行清单，并按业务实际名称组织汇总。
- 明确不包括什么：不在此阶段承诺未来尚未进入仓库月度核对流程的新主题。
- 完成标准：
  - `[TC-1]` 第一层目录按业务领域稳定组织，而不是按动作方向组织。
  - `[TC-2]` 领域内操作覆盖当前所有会影响仓库月度金额的业务家族。
  - `[TC-3]` 第二层至少能回答 `车间 / 销售项目 / 研发项目 / 主仓到RD交接` 等关键业务锚点。
  - `[TC-4]` 技术逆操作和异常类动作不作为独立主领域出现。

### `F3` 领域汇总到单据头追溯

- 这项功能要解决什么：让金额不一致时，用户能快速定位问题单据。
- 明确不包括什么：不要求所有主题默认细到单据行。
- 完成标准：
  - `[TC-1]` 用户可从领域汇总和车间 / 销售项目 / 研发项目 / 主仓到RD交接汇总下钻到单据头清单。
  - `[TC-2]` 单据头层默认能展示数量、金额、成本。
  - `[TC-3]` 用户能据此判断差异来源单据。

### `F4` 异常 / 跨月修正展示规则

- 这项功能要解决什么：在不破坏主表结构的前提下，解释调价、补录、跨月修正等影响。
- 明确不包括什么：不把异常统一拆成独立主表。
- 完成标准：
  - `[TC-1]` 异常金额仍归属原业务主题。
  - `[TC-2]` 第二层或异常列中能识别跨月修正 / 前期纠偏。
  - `[TC-3]` 技术逆操作被归并回原业务主题理解。

### `F9` 物料分类视角月度对账

- 这项功能要解决什么：在既有领域视角之外，为仓库提供一套按物料分类核算月度发生金额的稳定视角，回答“本月哪些分类入了多少、出了多少、净发生多少”。
- 明确不包括什么：不把分类视角扩成全主题月报，不在查询时用当前主数据重算历史分类。
- 完成标准：
  - `[TC-1]` 月报页面在既有领域视角之外，新增 `物料分类视角` 切换。
  - `[TC-2]` 第一版分类视角只覆盖 `验收入库 / 生产入库 / 销售出库 / 销售退货` 四类业务事实。
  - `[TC-3]` 分类金额按单据行事实只归属一个稳定最终分类；`未分类` 被稳定纳入，且不向祖先分类汇总。
  - `[TC-4]` 业务写侧与历史 backfill 都会落单据行分类快照，后续主数据改分类不会回改历史月份报表。
  - `[TC-5]` 分类明细默认下钻到单据行，并与领域视角在相同筛选条件下可对账。

### `F5` 仓库侧查看与导出

- 这项功能要解决什么：让仓库能直接拿系统结果去和财务核对。
- 明确不包括什么：不在系统内录入、导入、自动比对财务数据。
- 完成标准：
  - `[TC-1]` 报表支持系统内查看。
  - `[TC-2]` 报表支持 `Excel` 导出。
  - `[TC-3]` 导出结果可被外部财务核对使用。

### `F6` 正式月报结果与人工重算

- 这项功能要解决什么：在后续阶段沉淀长期保留的正式月报结果，并区分重算结果。
- 当前状态说明：保留为后续阶段能力，不是第一阶段首要交付。

### `F8` RD 小仓项目化口径与视角重算

- 这项功能要解决什么：当 `RD_SUB` 小仓改为“每笔物料都必须归属研发项目”后，让月报能够从主仓、`RD_SUB` 与研发项目三个视角一致解释同一笔交接事实。
- 明确不包括什么：不在报表层按备注 / 单据号 / 车间猜项目归属；不把研发项目提升为新的 `stockScope`。
- 完成标准：
  - `[TC-1]` 文档明确同一笔 `RD handoff` 可在不同视角下呈现为主仓总出、研发项目总入或全仓内部转移。
  - `[TC-2]` 文档明确这些视角都建立在交接事实层已有项目归属的前提上，而不是报表层事后推断。
  - `[TC-3]` 导出、下钻和异常展示在项目化口径下仍保持可追溯。

### `F7` 日期范围报表语义分离

- 这项功能要解决什么：保留临时分析能力，同时不破坏正式月报语义。
- 当前状态说明：保留为后续阶段能力，不是第一阶段首要交付。
