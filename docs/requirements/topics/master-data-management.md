# 基础数据管理主题需求

## Metadata

- ID: `topic-master-data-management`
- Version: `v1.0`
- Status: `confirmed`
- Scope: `topic-level`

## 主题定义

- 基础数据管理负责维护系统中所有业务动作共同依赖的"字典底座"：物料分类、物料、客户、供应商、人员、车间、库存范围。
- 这些数据不直接参与库存计算，而是以**快照**形式被各类业务单据引用；主数据的维护质量直接决定了单据录入的准确性与后续的追溯能力。
- 本 topic 覆盖所有主数据实体的新增、修改、停用与查询能力，以及为其他业务模块提供标准主数据下拉和快照查询的能力。

---

## 各主数据实体说明

> 本节面向系统使用者，用于理解每类基础数据的业务含义与使用场景。

### 1. 物料分类（MaterialCategory）

**这是什么**
物料分类是对物料进行归类的标签树，可以配置多级结构（如"电子元器件 → 电阻 → 贴片电阻"）。

**为什么要维护它**
分类的主要作用有两个：第一，录入物料时可以选择所属分类，方便后续按类别筛选或统计；第二，当物料数量增多时，分类结构帮助仓管人员快速定位和区分不同种类的物料。

**使用注意**

- 分类仅用于辅助归类，不直接影响库存账；一个物料只能属于一个分类。
- 停用一个分类前，需先确认该分类下没有仍在使用的物料。

---

### 2. 物料（Material）

**这是什么**
物料是系统中最核心的基础数据，代表"某一种具体的实物"，例如一颗 10Ω 贴片电阻、一卷铜箔、一套成品外壳。每条物料记录对应一种可以被入库、出库、领料、退料的实物。

**关键字段说明**


| 字段        | 含义                                                     |
| --------- | ------------------------------------------------------ |
| 物料编码      | 物料的唯一标识，系统内不允许重复，是各类业务单据引用物料的唯一凭据                      |
| 物料名称      | 物料的文字描述，便于人工识别                                         |
| 规格型号      | 同一物料名称下的具体规格，例如"100Ω"或"M6×12mm"；当同一名称存在多规格时，建议拆成独立物料条目 |
| 计量单位      | 该物料的数量计量单位（如个、米、千克），所有单据数量均以此为准                        |
| 所属分类      | 归属的物料分类，非必填，但建议填写以便统计                                  |
| 库存下限 / 上限 | 可选配置的安全库存预警阈值；系统可在库存低于下限或超过上限时给出提示（预留能力，当前阶段不强制）       |


**为什么要维护它**
物料主档是系统所有库存账、业务单据和成本追溯的根基。每一笔入库、出库、领料行为都会关联到具体的物料记录；如果物料档案不准确或不完整，后续的库存余额和成本计算都将无法追溯。

**使用注意**

- 物料编码一旦创建不建议修改，尤其是已有库存记录之后。
- 停用物料前，系统会校验该物料是否还有库存余额或未完成的业务单据；有则无法直接停用。
- 物料不会因停用而被物理删除，历史单据仍可查看停用前的物料信息。
- 当同一物品有多种规格（如电阻有不同阻值）时，推荐每种规格单独建一条物料，以减少单据录入时二次选择的负担。

### 3. 客户（Customer）

**这是什么**
客户是出库业务的收货对象，即产品卖给或交付给的公司或个人。系统的客户主档支持树形结构，可以配置"集团 → 子公司"之类的层级关系。

**为什么要维护它**
出库单、销售退货单都需要选择客户；客户信息还会被关联到项目中，便于按客户统计出货量和退货情况。

**使用注意**

- 客户编码在系统内唯一，不同客户不能共用一个编码。
- 支持父子树形结构（如一个集团下挂多个子公司），但父客户停用前需先处理下属子客户。
- 停用客户不会删除历史单据中该客户的快照信息。

---

### 4. 供应商（Supplier）

**这是什么**
供应商是入库业务的货物来源，即从外部采购物料时的卖家。每次做验收单（采购入库），都需要选择对应的供应商。

**为什么要维护它**
供应商信息出现在验收单（采购到货）和研发采购申请中，是追溯"这批货从哪里来"的关键字段，也是后续进行按供应商维度分析采购金额、到货数量的必要基础。

**使用注意**

- 供应商停用后，新建单据时将无法选择该供应商；但已完成的历史单据不受影响。
- 同一实体公司若既是客户又是供应商，目前系统中客户和供应商是两份独立主档，需分别维护。

---

### 5. 人员（Personnel）

**这是什么**
人员主档记录的是业务动作的**经办人**，例如"谁负责接收这批货"、"谁发起了这次领料"。这里的人员不等同于系统登录账号（系统账号由系统管理模块维护），而是可以在单据上选择和追溯的业务操作人员。

**为什么要维护它**
验收单、出库单、车间领料单等业务单据中都有"经办人"字段，该字段从人员主档中选取；拥有可追溯的经办人信息，是事后责任确认和操作审计的重要依据。

**使用注意**

- 人员编码在系统内唯一。
- 停用人员不会影响其出现在历史单据快照中；但新建单据时将不再出现在经办人下拉列表中。
- 若某位仓管离职，建议及时将其停用，避免新单据误选。

---

### 6. 车间（Workshop）

**这是什么**
车间主档定义了公司内部的生产/研发部门单元（如"装配车间"、"电子车间"、"研发部"等），用于在业务单据中标记物料的归属和成本核算。

**重要澄清：车间 ≠ 独立仓库**
这是最容易产生误解的地方：

- **车间不维护独立的库存账**，物料实物仍由主仓或研发小仓统一管库。
- 车间领料时，记录的是"主仓中的哪些物料被领到哪个车间使用"，对应的库存出账发生在**主仓**，车间只是标注了"这笔出库的用途归属"。
- 这样设计是为了便于按车间进行成本核算（这批物料算到哪个部门的消耗成本），而不是要实现每个车间各自管库的多仓模式。

**为什么要维护它**

- 验收单、领料单、退料单等单据中均有"归属车间"字段，用于后续按车间维度统计领用成本。
- 系统菜单中"车间"相关的视图（如按车间筛选领料记录）需要以车间主档为数据源。
- 系统管理员可配置哪些车间显示在各业务页面的筛选侧边栏，而非默认展示全公司所有车间。

**使用注意**

- 车间编码在系统内唯一。
- 停用车间后，新建单据将无法选择该车间；已完结的历史单据快照不受影响。

---

### 7. 库存范围（StockScope）

**这是什么**
库存范围是系统中**真实物理库存的隔离边界**，代表"货实际存放在哪个独立账簿空间里"。当前系统定义了两个库存范围：


| 范围编码     | 名称   | 说明                                 |
| -------- | ---- | ---------------------------------- |
| `MAIN`   | 主仓   | 公司主要仓库，绝大部分物料的入出库都发生在主仓            |
| `RD_SUB` | 研发小仓 | 研发部门独立管理的小仓，物料需经主仓验收入账后，再通过研发交接单转入 |


**与车间的关系**
库存范围和车间共同描述一条库存记录的两个不同维度：

- "**这批物料在哪**"→ 由库存范围（`StockScope`）回答，如"在主仓"
- "**这批物料算谁的**"→ 由车间（`Workshop`）回答，如"归属研发车间"

两者分开记录，避免"谁用的货放在谁那里"这种混淆。

**为什么要维护它**

- 每笔库存余额（`inventory_balance`）和每条库存流水（`inventory_log`）都挂在某个库存范围下；范围不对，库存账就不对。
- 研发小仓管理员和主仓管理员各自操作自己范围内的单据，`StockScope` 是权限和数据隔离的关键依据。

**使用注意**

- 库存范围由系统管理员维护，通常不需要频繁修改。
- 当前阶段仅支持主仓和研发小仓两个范围；若未来有新增真实仓库的需求，通过新增库存范围主档来扩展，而不是改代码。
- 停用某个库存范围前，必须确认该范围内已无库存余额。

## 长期约束

- `C1` 逻辑停用优先：所有主数据停用均使用逻辑停用（`status = INACTIVE`），不做物理删除。状态：`生效中`
- `C2` 快照隔离：业务单据录入时，主数据信息以快照形式固化在单据上；后续修改或停用主数据，不影响历史单据的显示。状态：`生效中`
- `C3` 停用前校验：物料停用前必须通过库存余额和未完成业务引用校验；其他主数据停用前需确认无在用单据引用。状态：`生效中`
- `C4` 车间与库存范围分离：`workshop` 只承担归属与成本核算维度，不承担独立库存池语义；真实库存范围由 `StockScope` 唯一控制。状态：`生效中`
- `C5` 受控自动补建：系统在特定历史兼容场景下允许自动补建主数据条目，但必须记录来源单据类型、来源单据 ID 和操作人，保证可审计、可回溯。状态：`生效中`

## 能力合同

### `F1` 物料分类 CRUD

- In scope:
  - 物料分类的新增、修改、逻辑停用
  - 树形结构查询（支持多级父子关系）
  - 停用分类前校验：该分类下是否存在 `status = ACTIVE` 的物料
- Out of scope / non-goals:
  - 分类不参与库存计算或成本核算
  - 不支持跨分类批量迁移物料
- Completion criteria:
  - `[TC-1]` 新增分类时，分类编码全局唯一；重复时返回明确错误。
  - `[TC-2]` 查询返回完整树形结构，支持按 `parentId` 递归展开。
  - `[TC-3]` 停用某分类时，若其下存在启用状态的物料，拦截并返回提示。
  - `[TC-4]` 停用某分类时，若其下存在启用状态的子分类，拦截并返回提示。
- Evidence expectation:
  - 树形查询接口返回嵌套 JSON；停用拦截有对应测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 树形查询可通过 Prisma 的 `parent` / `children` 自关联一次加载后在内存组装，数据量小无需递归 SQL。

### `F2` 物料 CRUD

- In scope:
  - 物料的新增（含手动创建与受控自动补建两种路径）
  - 物料修改（编码不可改，规格型号、单位、分类、预警阈值可改）
  - 物料逻辑停用及停用前库存余额与未完成单据引用校验
  - 物料列表与详情查询
- Out of scope / non-goals:
  - 物料版本化（不追踪物料历史变更记录）
  - 批量导入优化（归入 `F10`）
  - BOM / 物料组合拆解（归属 `project` 模块）
- Completion criteria:
  - `[TC-1]` 新增物料时，编码不为空且全局唯一；重复编码时返回明确错误提示。
  - `[TC-2]` 停用物料时，若该物料在任一库存范围下有余额 > 0，拦截并返回提示。
  - `[TC-3]` 停用物料时，若存在状态为"生效中"的引用单据，拦截并返回提示。
  - `[TC-4]` 自动补建物料时，`creationMode = AUTO_CREATED`，且 `sourceDocumentType`、`sourceDocumentId` 必须有值。
- Evidence expectation:
  - 接口联调可通过上述四个场景，并有对应的服务层单元测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 停用校验的实现优先通过 `inventory_balance` 余额聚合查询完成；单据引用校验通过各单据家族的 `lifecycleStatus = EFFECTIVE` 行数检查完成。

### `F3` 客户 CRUD

- In scope:
  - 客户的新增、修改、逻辑停用
  - 树形结构查询（支持集团 → 子公司层级）
  - 受控自动补建路径（历史兼容场景）
- Out of scope / non-goals:
  - 客户信用额度、账期等财务属性（当前阶段不涉及）
  - 客户联系人管理
- Completion criteria:
  - `[TC-1]` 新增客户时，客户编码全局唯一。
  - `[TC-2]` 树形查询返回正确的父子层级关系。
  - `[TC-3]` 停用父客户时，若其下存在启用状态的子客户，拦截并返回提示。
  - `[TC-4]` 自动补建客户时，`creationMode = AUTO_CREATED` 且来源字段完整。
- Evidence expectation:
  - 接口联调可通过，树形查询和停用拦截有测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 客户树与物料分类树实现模式一致（Prisma 自关联 + 内存组装），可复用树形查询基础设施。

### `F4` 供应商 CRUD

- In scope:
  - 供应商的新增、修改、逻辑停用
  - 列表与搜索查询
  - 受控自动补建路径（历史兼容场景）
- Out of scope / non-goals:
  - 供应商评级、资质管理
  - 供应商不支持树形结构（扁平列表即可）
- Completion criteria:
  - `[TC-1]` 新增供应商时，编码全局唯一。
  - `[TC-2]` 停用供应商后，新建验收单时下拉列表中不再出现该供应商。
  - `[TC-3]` 自动补建供应商时，`creationMode = AUTO_CREATED` 且来源字段完整。
- Evidence expectation:
  - 接口联调可通过，停用后下拉不出现有测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 供应商结构最简单（无树形、无层级），优先作为 CRUD 模板参照实现。

### `F5` 人员 CRUD

- In scope:
  - 人员的新增、修改、逻辑停用
  - 列表与搜索查询
  - 受控自动补建路径（历史兼容场景）
- Out of scope / non-goals:
  - 人员与系统账号（`sys_user`）的关联或合并（当前阶段两套独立维护）
  - 人员部门归属（当前阶段不建立人员-部门关系）
- Completion criteria:
  - `[TC-1]` 新增人员时，人员编码全局唯一。
  - `[TC-2]` 停用人员后，新建单据时经办人下拉列表中不再出现该人员。
  - `[TC-3]` 自动补建人员时，`creationMode = AUTO_CREATED` 且来源字段完整。
- Evidence expectation:
  - 接口联调可通过，停用后经办人下拉不出现有测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 人员被多个单据家族引用为 `handlerId`（经办人），停用校验可共用统一的"是否被生效单据引用"查询模式。

### `F6` 车间 CRUD

- In scope:
  - 车间的新增、修改、逻辑停用
  - 列表与搜索查询
- Out of scope / non-goals:
  - 车间不维护独立库存余额，不作为库存池
  - 车间与部门（`sys_dept`）的映射关系在本能力中不强制要求
- Completion criteria:
  - `[TC-1]` 新增车间时，车间编码全局唯一。
  - `[TC-2]` 停用车间后，新建单据时车间下拉列表中不再出现该车间。
  - `[TC-3]` 不存在以车间维度独立计算的库存余额表；车间仅作为 `inventory_balance` 中的归属维度字段存在。
- Evidence expectation:
  - 接口联调可通过，停用后下拉不出现有测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 车间不支持自动补建（无 `creationMode` 字段），只有手动管理路径。

### `F7` 库存范围 CRUD

- In scope:
  - 库存范围的新增、修改、逻辑停用
  - 停用前校验：该范围下所有 `inventory_balance.quantityOnHand` 合计必须为 0
  - 列表查询
- Out of scope / non-goals:
  - 库存范围不自动创建对应仓库物理布局或库位
  - 不在本能力中引入多层级仓库模型（如仓库 → 库区 → 库位）
- Completion criteria:
  - `[TC-1]` 新增库存范围时，范围编码全局唯一。
  - `[TC-2]` 停用库存范围时，若该范围下存在 `quantityOnHand > 0` 的余额行，拦截并返回提示。
  - `[TC-3]` 停用库存范围后，新建单据时该范围不出现在下拉列表中。
- Evidence expectation:
  - 停用前余额校验有测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 当前只有 `MAIN` 和 `RD_SUB` 两个范围，停用校验通过 `inventory_balance` 按 `stockScopeId` 聚合 `quantityOnHand` 实现。

### `F8` 主数据下拉与快照查询服务

- In scope:
  - 提供 `MasterDataQueryService`，供其他模块调用获取物料、客户、供应商、人员、车间、库存范围的标准下拉列表（仅返回 `ACTIVE` 状态条目）。
  - 提供主数据快照 DTO，供单据创建时固化主数据信息（如 `materialSnapshot`、`customerSnapshot`）。
  - 提供"确保主数据存在"能力（`EnsureMasterDataExistsUseCase`），仅对历史兼容场景开放。
- Out of scope / non-goals:
  - 其他模块不得直接 JOIN 主数据内部表；必须通过 `MasterDataQueryService` 或快照读取。
- Completion criteria:
  - `[TC-1]` 下拉接口只返回 `status = ACTIVE` 的条目，停用后不再出现。
  - `[TC-2]` 单据创建后，单据上的主数据快照字段值不随主数据后续修改而变化。
- Evidence expectation:
  - 至少一类单据（如验收单）有快照字段的集成测试覆盖。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 快照优先通过单据表自身快照字段实现（不是每次读时再查主数据），避免跨模块强耦合。

### `F9` 物料库存预警配置

- In scope:
  - 物料主档上的 `warningMinQty`（库存下限）和 `warningMaxQty`（库存上限）字段维护
  - 报表或仪表盘中根据当前库存余额与阈值对比，给出低于下限或超过上限的提示
- Out of scope / non-goals:
  - 不触发自动采购或自动补货流程
  - 不发送站外通知（如邮件、飞书消息），仅在系统内展示预警
- Completion criteria:
  - `[TC-1]` 物料编辑时可设置库存上下限，保存后持久化。
  - `[TC-2]` 报表或仪表盘页面可展示当前低于下限和超过上限的物料列表。
  - `[TC-3]` 余额为 0 且下限 > 0 的物料出现在预警列表中。
- Evidence expectation:
  - 预警查询接口有测试覆盖；前端仪表盘可展示预警数据。
- Default derived slice acceptance mode: `light`
- AI derivation note:
  - 预警查询为纯读路径，通过 `inventory_balance` 与 `material` 的 `warningMinQty` / `warningMaxQty` 做比对即可，不需要额外持久化预警状态。

### `F10` 主数据批量导入

- In scope:
  - 提供 Excel 模板下载能力
  - 支持通过上传 Excel 批量创建物料、客户、供应商、人员
  - 导入时逐行校验（编码唯一性、必填字段），返回逐行成功/失败结果
- Out of scope / non-goals:
  - 不支持通过 Excel 批量修改已有主数据
  - 不支持批量删除/停用
- Completion criteria:
  - `[TC-1]` 下载的 Excel 模板包含所有必填字段和示例行。
  - `[TC-2]` 导入时编码冲突行标记为失败，其余行正常创建，不因单行失败中断整批。
  - `[TC-3]` 导入完成后返回成功数/失败数及失败原因明细。
- Evidence expectation:
  - 导入接口有含冲突行的测试覆盖。
- Default derived slice acceptance mode: `full`
- AI derivation note:
  - 批量导入按"逐行校验 + 批量写入"模式实现，不使用数据库事务包裹全部行（避免一行失败回滚全部）。每行独立事务或使用 `createMany` + `skipDuplicates`。
