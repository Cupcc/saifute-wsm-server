# 需求文档说明

`docs/requirements/**` 保存面向用户的需求真源。

## 目录职责

| 路径 | 作用 |
| --- | --- |
| `PROJECT_REQUIREMENTS.md` | 项目级业务总纲，记录长期稳定、跨主题持续生效的业务目标、边界、规则和阶段方向 |
| `domain/*.md` | 某个长期业务领域的 PRD，按业务场景、业务流程、单据与数据、状态与规则、查询与报表、权限与数据隔离、验收标准来定义 |
| `REQUIREMENT_CENTER.md` | 需求索引看板 |

不再使用切片 `req-*.md` 文件。项目级总纲放在 `PROJECT_REQUIREMENTS.md`，领域级真源维护在 `domain/*.md`，执行计划、验证过程与验收记录写在 `docs/tasks/*.md`。

## 目录布局

```text
docs/requirements/
├── PROJECT_REQUIREMENTS.md
├── REQUIREMENT_CENTER.md
├── README.md
└── domain/
    ├── _template.md
    └── *.md
```

## 怎么理解这套分层

### 项目级文档写什么

`PROJECT_REQUIREMENTS.md` 负责回答这些问题：

- 这个 WMS 项目为什么做。
- 第一阶段先解决什么，不解决什么。
- 主仓、研发小仓、车间、项目分别是什么业务对象。
- 全局适用的业务规则是什么。
- 当前有哪些长期业务主题。

它不负责写单个主题的完整流程、状态机、字段设计和任务执行细节。

### domain 文档写什么

每个 `domain/*.md` 应尽量按业务事实组织，而不是按技术模块反推需求。推荐结构：

- **主题定义**：这个主题解决什么问题、覆盖什么边界、不覆盖什么。
- **角色与使用者**：谁会用、负责什么、看到什么。
- **业务场景**：在什么情况下触发，需要得到什么业务结果。
- **业务流程**：主流程、异常流程、逆操作与纠偏。
- **单据与数据**：有哪些业务单据、关键数据对象、数据口径。
- **状态与规则**：状态机、业务规则、追溯与成本口径。
- **查询与报表**：谁查什么、按什么维度看、是否支持导出和追溯。
- **权限与数据隔离**：哪些人能看什么、改什么、不能看什么。
- **范围外**：明确哪些内容不在本 domain 内。
- **功能项清单**：`F*` 编号，每项是一项可单独实现、单独验收的结果。
- **验收标准 / 功能项说明**：写清完成算什么、证据是什么、哪些边界不能扩。
- **阶段路线图**：各阶段目标和当前状态。

### task 文档写什么

`docs/tasks/*.md` 承接：

- 本次交付范围
- 执行计划
- 变更记录
- 验证与验收结论
- review 与阻塞记录

## 怎么开始一个新功能项

1. 先确认对应 `domain/*.md` 已经有明确的业务边界、功能项和验收标准。
2. 直接创建 `docs/tasks/task-YYYYMMDD-HHMM-*.md`，在 Metadata 中填写 `Related requirement: docs/requirements/domain/*.md (Fx)`。
3. 交付完成后，回写 domain 中对应功能项的状态和任务链接。

功能项状态由 `未开始 -> 进行中 -> conditionally-accepted / 已完成` 流转，不再为每次拆分创建中间切片文档。

## 编写约定

- 全文用中文；路径、文件名、 metadata 可保留英文。
- 项目级文档写长期目标、边界、规则和主题地图，不写任务执行过程。
- domain 文档写长期业务事实和可验收结果，不写当前回合进展。
- 验收标准使用 `[AC-*]` 或 `[TC-*]` 编号，必须是业务可读、可判定、可复核的结果。
- 如果旧文档还使用“能力清单 / 能力合同”等命名，可以逐步迁移到“功能项清单 / 功能项说明”。

## 与任务文档的关系

- `docs/tasks/*.md` 承接详细执行计划、验证过程、review 结论与 acceptance 记录。
- `docs/acceptance-tests/**` 承接 full-mode acceptance specs 与 runs。
- task 文档在 Metadata 中用 `Related requirement: docs/requirements/domain/*.md (Fx)` 指向 domain。
