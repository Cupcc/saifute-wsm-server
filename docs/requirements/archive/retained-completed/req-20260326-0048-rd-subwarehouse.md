## 用户需求

- [x] 系统需要在同一软件平台内同时支撑主仓与研发小仓协同作业，不要求拆成两套系统。
- [x] 研发小仓在业务上按真实独立运转的小仓处理，而不是简单的研发领用标签或项目视图。
- [x] 第一阶段研发小仓只与主仓对接；主仓发料/调拨后，系统应自动入库到小仓，不要求小仓管理员二次确认收货。
- [x] 货物进入公司、离开公司仍统一经过主仓；研发小仓第一阶段不直接承接外部采购入库、外部退回入库或对外退货。
- [x] 小仓管理员只关注自己小仓的信息与操作，需要提供面向小仓的独立工作台，并对主仓运营面做隔离。
- [x] 隔离采用折中模式：主操作台、数据、报表以小仓为主，但允许保留少量公共能力，如个人中心/登录态、物料只读选择、研发项目或归集项只读选择。
- [x] 展示采用“非镜像双视角”：大仓管理员以主仓运营面与必要的研发协同入口为主，小仓管理员以研发小仓独立工作台为主，不做两套完全镜像界面。
- [x] 系统管理员需要同时可见主仓侧与研发小仓侧入口及页面，用于统一配置、排障与跨域查看；不受小仓专属壳层裁切影响。
- [x] 小仓管理员第一阶段至少需要支持：查看小仓库存与流水、查看主仓调拨后自动入小仓结果、在小仓执行出库/领用、执行盘点/库存调整、执行本仓报废、查看仅属于小仓的报表。
- [x] 小仓管理员不负责退回主仓或对外退货等公司级二次流转动作，以减少额外操作负担；相关退料/退货仍由主仓统一处理。
- [x] 小仓出库原则上必须关联研发项目；若为办公用品等通用消耗，也需要落到明确的项目式归集项。
- [x] 研发小仓可以提出采购需求，由小仓管理员录入采购信息，再由采购角色执行采购。
- [x] 研发采购物料必须关联研发项目或项目式归集项，不能脱离项目单独采购。
- [x] 主仓需要可见研发采购信息，以便对照到货实物完成验收核对。
- [x] 主仓填写验收单时应能直接选择对应研发采购信息并自动带出内容，避免重复手工填写。
- [x] 研发采购到货后应先由主仓验收入主仓；考虑到主仓发料到小仓存在时间差，库存不应在主仓验收时直接记入小仓，而应在实际交接时再转入小仓。
- [x] 研发采购链路中的物料需要具备独立状态跟踪，且该状态不等同于采购单、验收单或领料单自身的审核状态。
- [x] 研发采购链路中的物料状态至少包括：`待采购`（填写采购表单保存后的默认状态）、`采购中`（采购角色确认接单后）、`取消`（被采购驳回或其他原因终止）、`到货`（实物到达主仓）、`验收`（已转入小仓）、`领取`（已被项目领用）、`报废`、`退回`。
- [x] 其中 `到货` 明确指到达主仓，`验收` 明确指完成主仓到小仓的交接；`退回` 属于公司级流转结果，仍由主仓统一执行，不要求小仓管理员直接操作退回动作。
- [x] 正式建模方向优先采用“受限子仓模型”：承认主仓与研发小仓的仓别协同，但第一阶段不扩展为完整开放式多仓体系。

## 需求矩阵（已确认）

### 角色与展示矩阵

| 角色 | 默认视角 / 壳层 | 可见入口 | 可见数据范围 | 主要动作 | 边界 / 限制 |
|------|------------------|----------|--------------|----------|-------------|
| 大仓管理员 | 主仓运营视角，默认普通后台壳层 | 主仓业务组 + 必要的研发协同入口 | 主仓全量数据 + 与研发协同有关的记录 | 主仓入出库、主仓验收、主仓到 RD 协同、查看研发采购与研发项目必要信息 | 不进入 RD 专属日常操作台，不承担小仓日常盘点 / 报废 / 领用执行 |
| 小仓管理员 | 研发小仓专属壳层，默认进入 `研发工作台` | `研发小仓` 组 + 少量公共能力 | 仅本小仓相关库存、流水、报表、项目归集信息 | 小仓库存查看、自动入库结果查看、项目领用、盘点 / 调整、本仓报废、录入研发采购需求 | 不看主仓运营面，不处理公司级外部入出，不执行退回主仓 / 对外退货 |
| 系统管理员 | 双视角全量可见，不受 `consoleMode` 裁切 | 主仓业务组 + `研发小仓` 组 + `系统管理` | 主仓侧 + 小仓侧 + 系统配置范围 | 统一配置、排障、跨域查看，必要时进入任一视角核对业务 | 不是第三套独立业务流，而是保留两侧可见与系统级管控能力 |

### 业务协同与库存责任矩阵

| 场景 | 发起 / 录入方 | 执行 / 记账方 | 库存记账结果 | 小仓是否二次确认 | 备注 |
|------|---------------|---------------|--------------|------------------|------|
| 研发采购需求录入 | 小仓管理员 | 暂不记库存 | 不变 | 否 | 先形成采购需求，供采购角色接单 |
| 外部采购到货并验收 | 采购 / 主仓链路 | 主仓 | 先入主仓 | 否 | RD 采购信息可供主仓验收选取，但库存不直接入小仓 |
| 主仓发料 / 调拨到小仓 | 主仓 | 主仓单据完成后自动过账 | 主仓减、小仓增 | 否 | 第一阶段只开放主仓到小仓的单向协同 |
| 小仓项目领用 / 出库 | 小仓管理员 | 小仓 | 小仓减 | 不适用 | 必须绑定研发项目或项目式归集项 |
| 小仓盘点 / 库存调整 | 小仓管理员 | 小仓 | 只调整本小仓 | 不适用 | 不影响主仓账 |
| 小仓报废 | 小仓管理员 | 小仓 | 小仓减 | 不适用 | 属于本仓内部动作 |
| 退回主仓 / 对外退货 | 主仓侧 | 主仓侧 | 按公司级流转处理 | 否 | 小仓管理员不直接操作 |

### 目标架构矩阵

| 架构层 | 负责内容 | 关键模块 / 实现方向 | 必须遵守的约束 |
|--------|----------|----------------------|----------------|
| 展示层 | 主仓后台壳层 + RD 专属壳层，按角色切换视角 | `web` 路由、菜单、首页 / 工作台、`consoleMode` | 大仓 / 小仓是非镜像双视角；`admin` 不能被额外裁成单一视角 |
| 权限层 | 菜单、页面、按钮、数据范围控制 | `rbac` | 前后端权限口径一致，不能出现“后端有权限、前端看不到”或反过来的断层 |
| 协同业务层 | 主仓到小仓协同、RD 采购需求、项目领用、本仓报废 / 盘点等业务用例 | `project`、`workshop-material`、后续 RD 子域切片 | 只承接“主仓 + RD 小仓”受限协同，不直接扩展成完整开放式多仓 |
| 库存核心层 | 所有库存增减、流水、仓别责任记账 | `inventory-core` | 任何单据和页面都不能绕过库存核心直接改数 |
| 报表与查询层 | 主仓报表、小仓报表、协同结果查询 | `reporting`、read models | 查询与报表必须按仓别和角色隔离，避免主仓 / 小仓数据串视图 |
