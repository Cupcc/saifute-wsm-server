## 1. 项目背景与业务现状

### 1.1 当前业务环境

- 实际仓库相关作业人员约 `3` 人，属于小团队协作场景。
- 当前组织口径先按 `研发部 / 采购部 / 仓库` 三个一级部门处理，不做二级部门。
- 业务整体仍以主仓为中心运行，现阶段已明确存在 `主仓 + 1 个研发小仓` 的协同场景；主仓与研发小仓是第一阶段仅有的真实库存范围，按“受限子仓模型”处理，不直接扩展为完整开放式多仓体系。
- 研发小仓在业务上按真实独立运转的小仓处理，但公司级对外入出仍统一经过主仓；现场仍有货物柜和柜架号，但货物柜和区域划分不完善。
- 物料没有条码体系。
- 日常管理方式仍以 `Excel + 手工记账` 为主。

### 1.2 当前业务流转

- 常见业务动作包括：入库、出库、退货、领料、退料、盘点。
- 调拨未实际使用，报废属于低频动作。
- `3` 名使用人员在仓库管理中职责相同，没有明确的岗位分工。
- 当前系统使用者先按 `系统管理员 / 仓库管理员 / 研发小仓管理员 / 采购人员` 四类主角色设计；`老板 / 财务` 暂作为后续预留的查看型角色。
- 当前阶段不单独维护岗位，先按 `部门 + 角色 + 用户` 承接组织与授权关系。
- 老板要求所有进入公司的物品都必须经过仓库管理系统记录。
- 存在“物品买来入库后很快就被领走”的场景，尤其是部分低频采购物品。
- 业务上已明确区分不同出入库语义：`出库` 主要指成品出库；`领料` 指生产部门领取原材料；项目/研发场景下还存在研发领用物料。
- 主要入库来源包括：采购入库、客户退货入库、生产退料入库。
- 尚无规范的盘盈入库口径，但由于历史库存混乱，后续需要补充库存调整类能力。
- 研发采购需求由小仓侧提出、采购角色执行；实物到达主仓后，主仓验收单只记录主仓入库事实，不再直接挂接 RD 采购需求；研发物料是否完成验收由 `RD / 研发协同` 内确认，再按真实交接时点转入研发小仓。
- 公司真实实物库存以主仓为主，并非所有物料都会下沉到研发小仓；很多物料会长期保留在主仓。
- 产品侧“生产车间”相关能力与菜单应按基础数据中的 `workshop` 概念划分；若页面需要左侧筛选或分栏，也应基于 `workshop`，而不是系统管理中的 `department`。`department` 仅承担组织与权限语义，`workshop` 仍作为单据归属与成本核算维度使用，不建立车间独立库存池。
- 仓库管理员在车间菜单中按车间录入领料、退料时，实际仍是主仓库存出入账；`workshop` 只用于标记归属车间和后续成本核算，不维护车间在手库存。
- 研发小仓管理员可在本仓独立执行领料、退料、报废、盘点/调整等操作，并需要记录本仓物料使用情况。

### 1.3 数据现实

- 物料规格型号信息不完整，部分物料缺少稳定、统一的规格字段。
- 同一种物品存在多规格的情况较常见。
- 业务上希望尽量保留来源与流转追踪信息，暂不要求有效期管理。
- 对“一物多规格”如何建模，倾向优先选择更简洁、减少仓库二次选择成本的方案。
- 同一物料和型号在不同入库批次下可能存在不同单价；后续成本核算不能只按物料静态单价处理。

### 1.4 核心痛点

- 现有记账逻辑混乱，缺少统一、清晰的记录口径。
- 统计工作麻烦，数据难以快速汇总和核对。
- 尚不清楚什么信息应该如何分类、如何记录、如何展示，需要系统设计帮助梳理。
- 核算车间生产费用和项目研发费用等成本复杂且困难，需要系统设计帮助梳理。

## 2. 项目目标与全局规则

### 2.1 建设目标

- 构建一个清晰、模块边界明确、易维护的 NestJS 仓储管理系统。
- 系统设计要贴合真实业务与 `10` 人以下仓库管理场景，避免过度复杂化。
- Java 旧系统迁移到 NestJS 时，以业务域承接真实业务事实，不做机械的表对表搬迁。
- 库存、单据、审核与来源追溯等核心业务语义必须正确、可审计、可追溯。
- 近期重点是先把主仓场景与研发小仓的受限协同做好、用顺手，而不是优先为远期开开放式多仓变化做复杂扩展。

### 2.2 单据与审核规则

- 单据录入以 `1` 人录入为主，其他人员可进行审核。
- 审核主要用于记录和追溯，不要求“审核通过后才允许落库”。
- 若审核结论不同，应通过改单或作废纠正业务结果，并同步更新系统中的库存与单据数据。
- 所有影响实物流转或库存结果的业务动作都必须以单据为依据，遵循“无单不动，凭单作业”；不允许脱离单据直接改库存，允许线下先作业后补单，但补单只用于补录业务事实，不得通过倒填单据改变已生效库存的先后顺序。
- 库存管理目标以实时库存为准；盘点用于校正库存，不替代实时台账逻辑。
- 盘点方式为人工月盘。

### 2.3 仓储与库存模型

- 第一阶段真实库存范围仅包含主仓与研发小仓；不直接扩展为完整开放式多仓体系。
- 主仓是公司级实物流转中心；研发小仓是第一阶段真实独立库存范围，但受主仓协同供货约束。
- 公司真实实物库存仍以主仓为主，并非所有物料都会下沉到研发小仓。
- `workshop` 只承担单据归属、统计与成本核算语义，不建立车间库存池或车间在手库存台账。
- 这里的 `sales-project` 专指对外销售的大型项目，不包含内部 `RD`（Research and Development）项目；`RD` 主题另按研发小仓协同处理。
- 项目页面中的“项目库存”第一阶段不建立独立物理库存池；真实库存仍以全仓实时库存为准，同时按对外销售项目展示归属库存或项目台账余额。
- 项目页若需要“分配到项目”或“预留给项目”的快捷操作，后台必须沉淀为独立项目分配 / 预留记录；该动作不等同真实出库，也不得直接减少在手库存。

### 2.4 来源与成本规则

- 默认按 `FIFO`（先进先出）确定出库、领料等消耗动作的库存来源与成本口径。
- 系统需支持用户在有明确业务依据时手动指定来源批次；手动指定后，库存扣减、成本计算、逆操作与来源追溯均以指定结果为准。
- 无论使用默认 `FIFO` 还是手动指定批次，系统都必须保留可审计、可追溯的来源分配记录。
- 同一物料 / 型号在不同入库批次下可存在不同单价；系统必须按来源批次或来源流水跟踪成本，不以物料主档静态单价替代真实成本层。
- 入库价格与来源追溯能力最终覆盖 `sales` 出库（含销售项目维度）、车间领料和 `RD handoff` 三类链路；其中 `sales` 出库与车间领料属于最终消耗链，`RD handoff` 至少承担成本桥接职责，保证主仓到 RD 小仓的来源关系不断链。
- 若系统要提供完全可信的 `FIFO` 成本追溯，则必须遵守“先有可用入库来源、后有出库消耗”的库存事实顺序；系统不允许负库存，不允许先出后补来源后仍宣称成本完全可信。
- 允许补录历史单据、补填业务日期，但补录不得改写已经生效的库存事实顺序；如需纠正历史，应通过冲回重建、重放或约定起算点后的重算处理。
- 对历史上缺少来源分配记录的数据，不承诺自动补齐完全可信 `FIFO` 成本；系统可自约定起算时间起提供完全可信的来源追溯与成本结果。

### 2.5 组织与权限原则

- 当前阶段不单独维护岗位，先按 `部门 + 角色 + 用户` 承接组织与授权关系。
- 当前系统使用者先按 `系统管理员 / 仓库管理员 / 研发小仓管理员 / 采购人员` 四类主角色设计。
- `老板 / 财务` 暂作为后续预留的查看型角色，不纳入第一阶段必交付范围。
- `department` 仅承担组织与权限语义；涉及生产归属、成本统计与业务视图时优先使用 `workshop`。
- 当存在研发小仓场景时，需要支持按仓别隔离菜单、按钮与数据视图；小仓管理员默认只看到本仓相关工作台与数据。

## 3. 领域地图

> 本节只按“能力归属”组织项目级领域地图，不再按前端可见性单独建一套目录。是否存在独立前端入口，只作为各领域的补充说明。详细职责、功能项、阶段状态和验收标准统一下沉到对应 `docs/requirements/domain/*.md`；尚未拆出独立 domain 的能力暂继续保留在项目级边界说明中。

### 3.1 业务域

#### 3.1.1 主数据（`master-data`）

- 默认承接物料、客户、供应商、人员、车间、库存范围等基础资料，并为各业务单据提供稳定下拉、快照和引用基础。
- 前端具有独立治理入口。
- 详细职责与能力清单下沉到 `docs/requirements/domain/master-data-management.md`。

#### 3.1.2 入库（`inbound`）

- **原材料、半成品、成品入库统一走** `inbound`（入库单），与车间部门退料、客户退货等其它回流路径区分。
- 入库第一阶段默认覆盖采购/验收入库和生产入库等真实入库事务，并在录入后立即影响实时库存。
- 详细职责与后续扩展边界下沉到 `docs/requirements/domain/inbound-business-module.md`。

#### 3.1.3 销售业务（`sales`）

- 销售业务第一阶段默认覆盖流向客户的原料、半成品、成品出库与销售退货两类核心事务，兼顾客户维度、销售项目维度查询与追溯。
- 出库后立即影响实时库存，销售退货用于回补库存，并需尽量保留与原出库关系的追溯能力。
- 第一阶段以完成真实销售出库 / 退货闭环和统计为先，不默认扩展复杂物流履约、波次或发运编排能力。
- 当前仍作为项目级业务边界维护；后续如拆 domain，再将详细职责与验收口径独立下沉。

#### 3.1.4 车间物料（`workshop-material`）

- 车间物料第一阶段默认覆盖领料、退料、报废三类真实事务，并在发生后立即更新主仓实时库存。
- 领料、退料、报废应尽量保留来源追踪和上下游关联，便于后续追责、纠偏和成本核对。
- 详细职责、长期约束与阶段状态下沉到 `docs/requirements/domain/workshop-material-module.md`。

#### 3.1.5 销售项目（`sales-project`）

- 这里的 `sales-project` 专指只对外销售的大型项目，不再把内部 `RD` 项目混入同一概念。
- `sales-project` 关注销售项目主档、项目维度库存 / 可供货视图、项目维度发货统计，以及可选项目分配 / 预留能力。
- `sales-project` 不是新的物理库存池；项目相关真实库存动作统一通过 `sales` 销售出库 / 退货承接，项目页面上的快捷出库也应沉淀为 `sales` 单据。
- 销售项目真源下沉到 `docs/requirements/domain/sales-project-management.md`；它与 `RD` 主题分开维护，不再共用同一套定义。

#### 3.1.6 RD / 研发协同与研发项目（`rd-subwarehouse` / `rd-project`）

- `RD`（Research and Development）是独立于 `sales-project` 的内部研发主题，不等同于对外销售的大型项目。
- `rd-subwarehouse` 负责研发采购状态链、主仓到研发小仓交接、研发小仓日常作业和小仓库存责任。
- `rd-project` 负责研发项目主档、研发项目 BOM、项目物料动作和项目台账，承接研发内部项目语义。
- `研发小仓` 不等同于项目库存池；仓别协同规则下沉到 `docs/requirements/domain/rd-subwarehouse.md`，研发项目需求真源下沉到 `docs/requirements/domain/rd-project-management.md`，当前运行时实现由 `docs/architecture/modules/rd-project.md` 约束。

#### 3.1.7 报表（`reporting`）

- 报表默认承接首页统计、库存报表、月度变动、分类汇总和跨模块查询导出等只读能力。
- 详细统计口径、月报语义和导出要求下沉到 `docs/requirements/domain/monthly-reporting.md`。

### 3.2 分析与辅助域

#### 3.2.1 AI 助手（`ai-assistant`）

- `AI` 助手第一阶段默认定位为查询、解释、导航和预填辅助工具，不直接写业务数据。
- 需要兼容前端现有流式交互方式，并可在权限范围内帮助用户查询库存、单据、预警和常见操作入口。
- 工具调用默认采用白名单和受控编排，输出结果应以可解释、可回退、不中断主业务为原则。
- 详细职责、白名单工具边界和受控编排语义下沉到 `docs/requirements/domain/ai-assistant-module.md`。

### 3.3 共享核心域

#### 3.3.1 库存核心（`inventory-core`）

- `inventory-core` 继续作为所有库存变动的唯一写入口。
- 默认覆盖库存增加、减少、逆操作、来源追踪、库存日志、预警和编号区间等核心能力。
- 它是后端库存事实、来源追溯和成本层传递的核心主体，但第一阶段不要求前端提供一个让用户直接“改库存”的独立业务模块。
- 前端用户看到的库存结果，会分散呈现在入库、销售业务、车间物料、研发项目、销售项目视图、RD / 研发协同、盘点 / 调整、报表等业务场景中；这些业务动作最终统一落到 `inventory-core`。
- 详细职责、来源追溯、逆操作与成本口径下沉到 `docs/requirements/domain/inventory-core-module.md`。

#### 3.3.2 审核（`approval`）

- 第一阶段默认采用轻量审核模式，主要用于记录、查询和追溯审核状态，不引入 `BPM`。
- 审核记录应与单据保持可关联、可重置、可追溯；改单或作废时需能同步反映审核状态变化。
- 它是跨单据共享的内部规则域，不要求前端单独提供一个与业务域并列的“审核业务模块”；详细边界、重置语义与跨单据承接关系下沉到 `docs/requirements/domain/approval-module.md`。

### 3.4 平台与横切域

#### 3.4.1 认证（`auth`）

- 系统默认只服务公司内部使用人员，第一阶段不开放注册、第三方登录或对外账号体系。
- 详细职责统一下沉到 `docs/requirements/domain/system-management-module.md` 的平台与横切能力章节。

#### 3.4.2 会话（`session`）

- 登录态继续采用 `JWT + Redis 会话`，不切换为纯无状态认证。
- 详细职责统一下沉到 `docs/requirements/domain/system-management-module.md` 的平台与横切能力章节。

#### 3.4.3 权限（`rbac`）

- 默认延续“菜单 / 按钮权限字符串 + 数据权限”模型，不在第一阶段重做组织和权限体系。
- 详细职责统一下沉到 `docs/requirements/domain/system-management-module.md` 的平台与横切能力章节。

#### 3.4.4 系统管理（`system-management`）

- 第一阶段系统管理以 `用户 / 角色 / 部门 / 菜单 / 字典 / 参数 / 通知` 为核心治理对象；`岗位` 能力可保留兼容，但当前不作为真实组织设计的必选对象。
- 在产品上通常表现为独立治理入口，但能力归属仍是平台域。
- 详细职责、角色矩阵与平台横切能力下沉到 `docs/requirements/domain/system-management-module.md`。

#### 3.4.5 审计日志（`audit-log`）

- 需要保留登录日志和操作日志，满足问题追查、责任追溯和管理审计。
- 详细职责统一下沉到 `docs/requirements/domain/system-management-module.md` 的平台与横切能力章节。

#### 3.4.6 文件（`file-storage`）

- 第一阶段默认延续本地磁盘文件方案，兼容历史 `/profile/`** 访问语义。
- 详细职责统一下沉到 `docs/requirements/domain/system-management-module.md` 的平台与横切能力章节。

#### 3.4.7 调度（`scheduler`）

- 需要保留数据库驱动的任务管理能力，支持任务定义、启停、立即执行和执行日志。
- 详细职责统一下沉到 `docs/requirements/domain/system-management-module.md` 的平台与横切能力章节；当前仍按平台邻接能力规划。

## 4. 管理输出范围

- 项目级只确认管理输出的覆盖范围，不在本文档中重复定义报表口径、汇总维度、明细粒度、导出格式和验收标准。
- 管理输出至少覆盖以下范围：
  - 主仓与研发小仓库存相关视图。
  - 仓别月度变动相关视图。
  - 公司整体月报。
  - 车间相关月报。
  - 销售域相关月报。

## 5. 文档边界与维护原则

### 5.1 本文档记录什么

- 本文档固定存在，用于存放整个项目长期、稳定、跨主题持续生效的用户需求。
- 这里记录的是“项目为什么做、长期要满足什么”，以及会持续影响系统设计判断的项目级背景信息。
- 可以记录当前业务环境信息，作为架构与产品设计的当前起点；这些信息不是永久冻结约束，后续如业务变化应同步更新。

### 5.2 本文档不记录什么

- 不记录单次任务的详细执行计划。
- 不记录临时 blocker、一次性 review 结论或实验性方案。
- 不记录实现细节与长执行日志。

### 5.3 文档关系

- `docs/requirements/domain/*.md`：承接领域级长期约束、能力清单和阶段路线图。
- `docs/tasks/*.md`：承接单次任务交付、执行过程与验收记录。
- `docs/requirements/README.md`：承接需求文档机制、命名、归档与使用规则。
- 不再使用切片 `req-*.md`；需求真源统一维护在 `docs/requirements/domain/*.md`。

### 5.4 维护原则

- 若某项需求已稳定收敛为独立领域，应迁移或补充到 `docs/requirements/domain/*.md`，而不是在本文档中持续堆积细节。
- 若项目级背景或长期约束发生变化，应直接更新正式表述，不保留“已废弃 / 已替代”的历史口径。
