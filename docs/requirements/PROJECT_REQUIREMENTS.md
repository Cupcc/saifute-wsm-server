## 1 文档定位

- 本文档固定存在，用于存放整个项目的长期、稳定、跨任务的用户需求。
- 这里记录的是"项目为什么做、长期要满足什么"，以及会持续影响系统设计判断的项目级背景信息。
- 可以记录当前业务环境信息，作为架构与产品设计的当前起点；这些信息不是永久冻结约束，后续如业务变化应同步更新。
- 主题级长期约束、能力清单和阶段路线图，继续放在 `docs/requirements/topics/*.md`。
- 单次任务、阶段性需求、当前阻塞与下一步，继续放在 `docs/requirements/req-*.md` 与 `docs/tasks/*.md` 中。
- 文档机制、命名、归档与使用规则统一写在 `docs/requirements/README.md`，不写入本文档正文。

## 2 业务背景

### 2.1 运营环境

- 实际仓库相关作业人员约 `3` 人，属于小团队协作场景。
- 当前组织口径先按 `研发部 / 采购部 / 仓库` 三个一级部门处理，不做二级部门。
- 业务整体仍以主仓为中心运行，现阶段已明确存在 `主仓 + 1 个研发小仓` 的协同场景；主仓与研发小仓是第一阶段仅有的真实库存范围，按“受限子仓模型”处理，不直接扩展为完整开放式多仓体系。
- 研发小仓在业务上按真实独立运转的小仓处理，但公司级对外入出仍统一经过主仓；现场仍有货物柜和柜架号，但货物柜和区域划分不完善。
- 物料没有条码体系。
- 日常管理方式仍以 `Excel + 手工记账` 为主。

### 2.2 业务流程

- 常见业务动作包括：入库、出库、退货、领料、退料、盘点。
- 调拨未实际使用，报废属于低频动作。
- `3` 名使用人员在仓库管理中职责相同，没有明确的岗位分工。
- 当前系统使用者先按 `系统管理员 / 仓库管理员 / 研发小仓管理员 / 采购人员` 四类主角色设计；`老板 / 财务` 暂作为后续预留的查看型角色。
- 当前阶段不单独维护岗位，先按 `部门 + 角色 + 用户` 承接组织与授权关系。
- 老板要求所有进入公司的物品都必须经过仓库管理系统记录。
- 存在"物品买来入库后很快就被领走"的场景，尤其是部分低频采购物品。
- 业务上已明确区分不同出入库语义：`出库` 主要指成品出库；`领料` 指生产部门领取原材料；项目/研发场景下还存在研发领用物料。
- 主要入库来源包括：采购入库、客户退货入库、生产退料入库。
- 尚无规范的盘盈入库口径，但由于历史库存混乱，后续需要补充库存调整类能力。
- 研发采购需求由小仓侧提出、采购角色执行；实物到达主仓且已填写验收单后，由主仓验收入主仓，再按真实交接时点转入研发小仓。
- 公司真实实物库存以主仓为主，并非所有物料都会下沉到研发小仓；很多物料会长期保留在主仓。
- 仓库管理员在车间菜单中按车间录入领料、退料时，实际仍是主仓库存出入账；`workshop` 只用于标记归属车间和后续成本核算，不维护车间在手库存。
- 研发小仓管理员可在本仓独立执行领料、退料、报废、盘点/调整等操作，并需要记录本仓物料使用情况。

### 2.3 数据特征

- 物料规格型号信息不完整，部分物料缺少稳定、统一的规格字段。
- 同一种物品存在多规格的情况较常见。
- 业务上希望尽量保留来源与流转追踪信息，暂不要求有效期管理。
- 对"一物多规格"如何建模，倾向优先选择更简洁、减少仓库二次选择成本的方案。
- 同一物料和型号在不同入库批次下可能存在不同单价；后续成本核算不能只按物料静态单价处理。

### 2.4 核心痛点

- 现有记账逻辑混乱，缺少统一、清晰的记录口径。
- 统计工作麻烦，数据难以快速汇总和核对。
- 尚不清楚什么信息应该如何分类、如何记录、如何展示，需要系统设计帮助梳理。
- 核算车间生产费用和项目研发费用等成本复杂且困难，需要系统设计帮助梳理。

## 3 需求总则

### 3.1 核心目标

- [ ] 构建一个清晰、模块边界明确、易维护的 NestJS 仓储管理系统。
- [ ] 系统设计要贴合真实业务与 `10` 人以下仓库管理场景，避免过度复杂化。
- [ ] Java 旧系统迁移到 NestJS 时，以业务域承接真实业务事实，不做机械的表对表搬迁。
- [ ] 库存、单据、审核与来源追溯等核心业务语义必须正确、可审计、可追溯。
- [ ] 近期重点是先把主仓场景与研发小仓的受限协同做好、用顺手，而不是优先为远期开开放式多仓变化做复杂扩展。

### 3.2 管理口径

- [ ] 单据录入以 `1` 人录入为主，其他人员可进行审核。
- [ ] 审核主要用于记录和追溯，不要求"审核通过后才允许落库"。
- [ ] 若审核结论不同，应通过改单或作废纠正业务结果，并同步更新系统中的库存与单据数据。
- [ ] 所有影响实物流转或库存结果的业务动作都必须以单据为依据，遵循"无单不动，凭单作业"；不允许脱离单据直接改库存，允许线下先作业后补单。
- [ ] 库存管理目标以实时库存为准；盘点用于校正库存，不替代实时台账逻辑。
- [ ] 盘点方式为人工月盘。

### 3.3 库存来源与成本口径

- [ ] 默认按 `FIFO`（先进先出）确定出库、领料等消耗动作的库存来源与成本口径。
- [ ] 系统需支持用户在有明确业务依据时手动指定来源批次；手动指定后，库存扣减、成本计算、逆操作与来源追溯均以指定结果为准。
- [ ] 无论使用默认 `FIFO` 还是手动指定批次，系统都必须保留可审计、可追溯的来源分配记录。
- [ ] 同一物料 / 型号在不同入库批次下可存在不同单价；系统必须按来源批次或来源流水跟踪成本，不以物料主档静态单价替代真实成本层。

## 4 模块需求

### 4.1 核心业务

#### 4.1.1 主数据（`master-data`）

- [ ] 默认承接物料、客户、供应商、人员、车间、库存范围等基础资料，并为各业务单据提供稳定下拉、快照和引用基础。
- [ ] 第一阶段主数据以"够用、清晰、易维护"为目标，不做过度统一建模或复杂版本化。
- [ ] 对历史兼容确有需要的场景，可支持受控自动补建主数据，但必须可审计、可追溯。

#### 4.1.2 库存核心（`inventory-core`）

- [ ] 真实库存以全仓实时库存为准，`inventory-core` 继续作为所有库存变动的唯一写入口。
- [ ] 默认需要覆盖库存增加、减少、逆操作、来源追踪、库存日志、预警和编号区间等核心能力。
- [ ] 第一阶段优先满足“主仓 + 研发小仓受限协同”、人工管理、来源可追溯的现实场景，不默认升级为完整库位/批次/开放式多仓 `WMS` 模型。
- [ ] 当存在研发小仓场景时，库存核心需支持主仓与研发小仓的受限协同、仓别隔离查询与主仓到小仓的受控转移，但不直接演进为完整开放式多仓体系。
- [ ] 第一阶段真实库存范围仅包含主仓与研发小仓；生产车间只作为领退料归属与成本核算维度，不建立车间库存余额。
- [ ] 库存成本不能只挂在物料主档或库存余额上；同物料不同入库批次可保留不同成本层，并通过来源分配记录传递到出库、领料、退料与逆操作。

#### 4.1.3 审核（`workflow`）

- [ ] 第一阶段默认采用轻量审核模式，主要用于记录、查询和追溯审核状态，不引入 `BPM`。
- [ ] 审核记录应与单据保持可关联、可重置、可追溯；改单或作废时需能同步反映审核状态变化。
- [ ] 默认继续遵循"审核不阻塞落库"的管理口径，审核更多服务于管理留痕与业务纠偏。

#### 4.1.4 入库（`inbound`）

- [ ] 入库第一阶段默认覆盖采购/验收入库和生产入库等真实入库事务，并在录入后立即影响实时库存。
- [ ] 需要支持单据新增、修改、作废、查询、导出，以及按供应商、物料、日期等常见维度追溯入库记录。
- [ ] 入库单据应与主数据、库存、审核记录保持关联，改单和作废时需同步处理库存与追溯结果。

#### 4.1.5 客户收发（`customer`）

- [ ] 客户收发第一阶段默认覆盖成品出库与销售退货两类核心事务，兼顾客户维度查询与追溯。
- [ ] 出库后立即影响实时库存，销售退货用于回补库存，并需尽量保留与原出库关系的追溯能力。
- [ ] 第一阶段以完成真实客户收发闭环和统计为先，不默认扩展复杂物流履约、波次或发运编排能力。

#### 4.1.6 车间物料（`workshop-material`）

- [ ] 车间物料第一阶段默认覆盖领料、退料、报废三类真实事务，并在发生后立即更新主仓实时库存。
- [ ] 领料、退料、报废应尽量保留来源追踪和上下游关联，便于后续追责、纠偏和成本核对。
- [ ] 第一阶段目标是支撑日常车间物料流转，不默认扩展工单驱动、工位级库存等复杂生产执行能力。
- [ ] 车间不建立独立库存池或车间在手库存台账；重点是清晰记录按车间的领料、退料、报废、净耗用和成本。

#### 4.1.7 项目/研发（`project`）

- [ ] 第一阶段以内部研发项目为主，不以对外客户项目或正式销售流程为核心。
- [ ] 需要提供单独的项目管理页面，供仓库管理员维护项目台账，并在项目上下文中录入相关业务单据。
- [ ] `研发小仓` 不等同于项目库存池；它是第一阶段受限子仓，由主仓协同供货，小仓侧出库/领用必须绑定研发项目或项目式归集项。
- [ ] 研发小仓是第一阶段真实独立库存范围；主仓交接到小仓后，小仓管理员可在本仓独立执行领料、退料、报废、盘点/调整，并记录物料使用情况。
- [ ] 第一阶段项目相关的真实库存事务至少包括：`项目采购入库`、`项目领料`、`项目退料`、`项目报废`。
- [ ] `项目采购入库` 表示为某个研发项目专门采购并入库；入库后默认归属于该项目，用于后续项目消耗、结余和成本统计。
- [ ] 研发采购需求由小仓侧提出、采购角色执行；主仓验收时应能直接选择对应采购信息自动带出内容，避免重复录入。
- [ ] `项目领料` 发生后立即影响实时库存，不做纯申请记录；`项目退料` 与 `项目报废` 应尽量关联原始入库或领料记录以保证追溯，但不强制要求现场必须找到原单据后才能登记。
- [ ] 项目页面中的"项目库存"第一阶段不建立独立物理库存池；真实库存仍以全仓实时库存为准，同时按项目展示归属库存或项目台账余额。
- [ ] 第一阶段需要支持项目成本统计，至少能按项目查看累计采购投入、研发领料消耗、退料回补、报废损耗，以及项目维度的汇总与明细。
- [ ] 第一阶段保留轻量 `BOM`/标准用料清单作为参考能力，用于辅助项目成果成本归集，但不引入复杂版本、替代料或强控领料逻辑。
- [ ] 项目成果、样品发出或未来可能发生的销售，第一阶段首要目标是成本统计而不是收入管理；相关记录可用于归集"本次成果用了哪些物料、成本是多少"，但暂不要求成为正式销售库存事务。
- [ ] 研发采购链路中的物料需要维护独立状态流，至少覆盖 `待采购`（小仓管理员填写并保存采购表单后的默认状态）、`采购中`（采购角色确认接单后）、`取消`、`验收`、`领取`、`报废`、`退回`；其中 `验收` 指实物到达主仓且已填写验收单，`领取` 指完成主仓到小仓交接。

### 4.2 分析与辅助

#### 4.2.1 报表（`reporting`）

- [ ] 报表默认承接首页统计、库存报表、月度变动、分类汇总和跨模块查询导出等只读能力。
- [ ] 统计展示应优先覆盖已确认的核心管理需求：按主仓 / 研发小仓查看当前库存与月度库存变化，按车间查看领料 / 退料 / 报废 / 净耗用与成本汇总明细，按项目 / 研发领用查看汇总明细等。
- [ ] 系统需要支持按月自动生成月度报表，至少覆盖公司整体月度汇总，以及按生产车间、销售域、研发项目划分的月度汇总与明细。
- [ ] 自动月报应服务管理核对与经营分析，保持统计口径清晰、结果可追溯，并支持后续导出与人工复核。
- [ ] 报表以口径清晰、易核对、便于人工管理为目标，不默认建设实时数仓或复杂 `OLAP` 分析。
- [ ] 当存在研发小仓场景时，报表与查询需支持按仓别隔离查看研发小仓库存、流水和相关汇总，不默认向小仓管理员暴露主仓运营面数据。

#### 4.2.2 AI 助手（`ai-assistant`）

- [ ] `AI` 助手第一阶段默认定位为查询、解释、导航和预填辅助工具，不直接写业务数据。
- [ ] 需要兼容前端现有流式交互方式，并可在权限范围内帮助用户查询库存、单据、预警和常见操作入口。
- [ ] 工具调用默认采用白名单和受控编排，输出结果应以可解释、可回退、不中断主业务为原则。

### 4.3 平台与横切

#### 4.3.1 认证（`auth`）

- [ ] 系统默认只服务公司内部使用人员，第一阶段不开放注册、第三方登录或对外账号体系。
- [ ] 需要保留验证码、用户名密码登录、登出、当前用户信息与前端路由初始化入口。
- [ ] 登录前需校验停用/删除状态、失败次数限制和基础安全策略；登录后返回可供前端直接使用的身份与权限摘要。

#### 4.3.2 会话（`session`）

- [ ] 登录态继续采用 `JWT + Redis 会话`，不切换为纯无状态认证。
- [ ] 需要支持滑动续期、在线用户查看、强制下线和注销后立即失效。
- [ ] 会话中应保留角色、权限和基础身份快照，支撑后端鉴权与前端初始化。

#### 4.3.3 权限（`rbac`）

- [ ] 默认延续"菜单/按钮权限字符串 + 数据权限"模型，不在第一阶段重做组织和权限体系。
- [ ] 需要支撑用户、角色、菜单、授权关系维护，并向前端输出按用户过滤后的路由树。
- [ ] 数据权限默认至少覆盖全部、自定义部门、本部门、本部门及子部门、仅本人等常见口径；未命中时默认从严。
- [ ] 当存在研发小仓场景时，需要支持按仓别隔离菜单、按钮与数据视图；小仓管理员默认只看到本仓相关工作台与数据。

#### 4.3.4 系统管理（`system-management`）

- [ ] 第一阶段系统管理以 `用户 / 角色 / 部门 / 菜单 / 字典 / 参数 / 通知` 为核心治理对象；`岗位` 能力可保留兼容，但当前不作为真实组织设计的必选对象。
- [ ] `系统管理员` 保留全量系统管理与跨域排障兜底能力；`仓库管理员` 负责主仓业务与公司级实物流转，并可查看 RD 协同必要信息；`研发小仓管理员` 负责 RD 小仓内部日常作业，只看 RD 视角与主仓协同结果；`采购人员` 负责采购执行链路，并可回看与采购闭环有关的主仓验收结果。
- [ ] `老板 / 财务` 当前先保留为后续可能增加的查看型角色，不纳入第一阶段必交付范围。
- [ ] 第一阶段将 `在线用户 / 登录日志 / 操作日志` 视为系统管理相邻的平台治理能力一并规划；`调度` 与 `AI 助手` 作为非核心平台能力顺延到第三阶段规划，当前状态暂不执行。

#### 4.3.5 审计日志（`audit-log`）

- [ ] 需要保留登录日志和操作日志，满足问题追查、责任追溯和管理审计。
- [ ] 关键业务操作应能记录操作者、时间、对象、结果和异常摘要，但审计失败不应阻断主业务。
- [ ] 敏感字段默认脱敏处理，不把密码、`token` 等敏感内容直接暴露到日志中。

#### 4.3.6 文件（`file-storage`）

- [ ] 第一阶段默认延续本地磁盘文件方案，兼容历史 `/profile/**` 访问语义。
- [ ] 需要支持普通上传、下载、头像管理和导入导出类文件交互。
- [ ] 文件能力以基础可用和兼容性为先，暂不默认扩展到对象存储、私有签名下载或附件中心。

#### 4.3.7 调度（`scheduler`）

- [ ] 需要保留数据库驱动的任务管理能力，支持任务定义、启停、立即执行和执行日志。
- [ ] 任务注册和执行应以可追踪、可人工管理为目标，不默认引入更复杂的分布式调度体系。
- [ ] 定时任务只能调用受控白名单能力，避免把任意脚本执行暴露为默认需求。

## 5 统计与展示

- [ ] 需要清晰展示主仓与研发小仓的当前库存。
- [ ] 需要按仓别查看某个月的库存变动情况。
- [ ] 需要由系统自动生成公司整体月度汇总报表。
- [ ] 需要按车间查看月度领料、退料、报废、净耗用与成本汇总明细，不要求维护车间在手库存。
- [ ] 需要按销售域查看月度汇总和明细。
- [ ] 需要按研发项目查看月度汇总和明细。
- [ ] 需要按项目/研发领用查看汇总和明细。
- [ ] 需要在项目详情中查看项目归属库存或项目台账余额，以及项目成本相关的汇总和明细。
